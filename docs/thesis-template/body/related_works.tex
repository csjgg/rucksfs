\section{相关技术基础}

RucksFS 的设计建立在三组基础技术之上：FUSE 负责把 Linux VFS 请求交给用户态客户端，RocksDB 提供适合元数据工作负载的持久化引擎和事务能力，POSIX 语义则规定了文件系统操作应该符合的行为。本章只挑和后续设计直接相关的部分讲，不打算对这三个主题做全面铺陈。

\subsection{FUSE 用户态文件系统框架}

ext4、XFS、Btrfs 这类传统文件系统运行在内核态，调用路径短，但开发、调试、故障定位都要直面内核环境。FUSE（Filesystem in Userspace）\cite{szeredi2005fuse}把这件事换了个思路：内核只留请求转发、进程阻塞/唤醒、权限协作这些基础机制，具体的文件系统逻辑放到用户空间去实现。从 Linux 2.6.14 起 FUSE 就进了主线内核，后来逐渐成了用户态文件系统的事实标准。JuiceFS\cite{juicefs2021}、CubeFS\cite{liu2019cubefs}、GlusterFS、SSHFS 都是走 FUSE 对外提供 POSIX 接口。

本课题用 FUSE 不是为了绕开“内核态开发”的难度，而是想要一个清晰的系统边界。Linux VFS 继续管系统调用入口和权限框架，用户态客户端只负责协议解析、请求路由，以及和元数据服务、数据服务打交道。这种分工和后面三段式分布式架构是匹配的。

    \caption{FUSE 请求的分层执行路径}
    \label{tab:fuse_path}
    \begin{tabular}{clp{8.2cm}}

    \textbf{阶段} & \textbf{位置} & \textbf{动作} \\

当应用对挂载点执行 \texttt{open()}、\texttt{stat()}、\texttt{mkdir()} 等系统调用时，请求首先进入 VFS。若目标路径位于 FUSE 挂载点下，VFS 不会继续调用 ext4 之类的内核文件系统实现，而是把请求转交给 FUSE 内核模块。该模块为请求分配请求 ID，封装操作码与参数，并将请求写入 \texttt{/dev/fuse}，同时让发起系统调用的线程阻塞等待。

用户态守护进程在挂载时已经打开 \texttt{/dev/fuse}，随后持续从该设备读取请求帧。守护进程根据 opcode 识别请求类型，再把请求派发到开发者实现的回调，例如把 \texttt{LOOKUP} 派发给 \texttt{lookup()}，把 \texttt{CREATE} 派发给 \texttt{create()}。回调完成后，守护进程把结果编码为响应帧并写回 \texttt{/dev/fuse}；内核再据请求 ID 唤醒原先阻塞的线程，将结果接回系统调用返回路径。

从系统结构上看，FUSE 把“POSIX 请求如何进入文件系统”和“文件系统内部如何实现这些请求”明确分离了。内核负责入口与调度，用户空间负责语义与存储实现，二者之间唯一共享的交互媒介就是 \texttt{/dev/fuse}。

在实现机制上，\texttt{fuse3} 的异步模型可以概括为“读请求帧、生成 Future、等完成后写回响应”三步。运行时从 \texttt{/dev/fuse} 读到请求后，并不会给每个请求独占一个阻塞线程，而是把它派发到对应的异步回调，由执行器推进这些 Future。任何回调在中途进入等待状态（比如在等元数据 RPC、数据 RPC 或本地异步 I/O），执行器就可以先挂起它，去处理其他已经就绪的请求。

这种做法的关键不在于“把同步函数改写成异步语法”，而在于把等待的开销从内核工作线程里挪出去。对分布式文件系统的客户端来说，FUSE 回调后面往往跟着一次或多次网络往返；如果仍然走“一个请求一个线程”的阻塞模型，线程数会随着并发请求和网络延迟一起涨。异步模型把并发控制交给事件循环和任务调度，在有限线程下就能维持大量未完成的请求。

对 RucksFS 这种客户端/服务端分离的形态来说，\texttt{fuse3} 的异步接口正好合适。FUSE 层负责收发请求，真正耗时的是用户态客户端到 MetadataServer、DataServer 的交互。这条路径做成异步之后，在等远端返回期间就可以继续推进其他 \texttt{lookup}、\texttt{getattr}、\texttt{readdir}，对元数据密集的并发负载比较友好。

\subsubsection{性能开销特征}

FUSE 的主要代价来自跨内核态与用户态边界的固定成本。对 ext4 这种内核态文件系统，一次 \texttt{stat()} 基本在内核里就走完了；对 FUSE 文件系统，同一次 \texttt{stat()} 至少要走完“应用线程进内核、请求写入 \texttt{/dev/fuse}、守护进程读请求、守护进程写回响应、内核唤醒原线程”这一整套。与内核态相比多出来的开销主要是两类：上下文切换，以及请求/响应两帧在 \texttt{/dev/fuse} 上的拷贝。

Vangoor 等人\cite{vangoor2017fuse}做过系统性测试：在 \texttt{lookup}、\texttt{getattr} 这类元数据操作上，FUSE 相比内核态文件系统通常会退化 20\%--60\%；顺序大文件读写场景下这部分退化一般只有 5\%--20\%。原因不难想：元数据操作本身执行时间短，这部分固定边界开销占比高；磁盘或网络 I/O 一旦主导，固定成本就被摊薄了。

对 RucksFS 这种分布式原型来说，FUSE 并不是主要瓶颈。真正压住端到端性能的是客户端到 MetadataServer 的 gRPC 网络路径：每次文件操作至少一次跨节点 RPC 往返，这个延迟远大于 FUSE 自己的内核--用户态边界开销。所以 FUSE 带来的是一个固定的常数开销，而不是会随规模变差的瓶颈。

\subsection{RocksDB 存储引擎}

RocksDB\cite{dong2021rocksdb}是 Meta 在 LevelDB\cite{google2011leveldb}基础上发展出的嵌入式键值存储引擎，针对 SSD 场景和高写入负载做了不少工程优化，已经被 TiKV、MyRocks、CockroachDB 这些上层系统广泛用作持久化后端。对“小值、高频、随机写密集”的文件系统元数据负载来说，RocksDB 的 LSM-tree、列族、前缀布隆过滤器和事务接口都是挺顺手的工具。

\subsubsection{LSM-tree 数据结构}

LSM-tree（Log-Structured Merge-tree）最早由 O'Neil 等人提出\cite{oneil1996lsmtree}。它的核心思想是把随机写换成顺序写：新写入先进内存中的 MemTable，同时追加一条 WAL；MemTable 满了就作为一整块有序 SST 文件刷盘；后台 Compaction 线程会把各层的 SST 按键范围合并重写，维持读性能和空间利用率。

对文件系统元数据来说，这套结构有两个直接的含义。一是单条目录项或 inode 属性的更新可以先在 MemTable 里做一次内存追加，不像 B 树那样马上改磁盘页。二是后台合并阶段会引入额外写放大，所以键的设计和值的大小得保持克制。RucksFS 后面也就是围绕这两点展开的：目录项和值都尽量小，文件数据完全绕开 RocksDB，免得大 value 拖垮 Compaction。

RocksDB 的读取路径反着走：先查活跃 MemTable，再查不可变 MemTable，然后自 Level 0 开始逐层搜 SST 文件。为了减小点查的代价，SST 文件内部维护索引块和 Bloom 过滤器；为了减少磁盘访问，常用数据块会进块缓存。这也解释了为什么文件系统元数据要强调键的前缀局部性和点查友好：不然读路径就会在多层 SST 里被放大。

\subsubsection{Column Family 机制}

Column Family（列族）是 RocksDB 比较实用的一个特性。它允许在同一个数据库实例里维护多组逻辑上隔离的键空间，各自有独立的 MemTable、SST 集合和压缩配置，但共享 WAL。对文件系统来说，这意味着 inode 基础属性、目录项、增量日志、系统元信息不用挤在同一种访问模式下被统一处理。

RucksFS 的元数据层就是依赖这个特性做分层配置的：\texttt{inodes} 列族承担点查为主的 inode 属性访问；\texttt{dir\_entries} 列族承担按父目录前缀扫描的目录项访问；\texttt{delta\_entries} 列族承担高频的增量追加和后台折叠；\texttt{system} 列族只放 inode 分配器这类低频元信息。拆开之后，不同列族就可以分别配置 Bloom 过滤器、前缀提取器和压缩策略，而不用在一个全局配置上折中。

这样做也有代价。列族一多，内存里的 MemTable 和后台 Compaction 状态也跟着多；跨列族的一致性更新还得靠事务或批量原子提交兜住。RucksFS 最后只保留了 4 个列族，就是在“访问模式隔离”和“系统复杂度”之间取个平衡。

\subsubsection{WriteBatch、事务与并发控制}

RocksDB 在一致性相关能力上分两层。一层是 WriteBatch，能保证一组 \texttt{put}/\texttt{delete} 原子提交；更上一层是 TransactionDB，除了原子提交，还提供 \texttt{get\_for\_update}、锁等待超时、冲突检测等事务能力。对普通 KV 业务 WriteBatch 通常就够用了，但文件系统元数据不一样——很多错误恰恰发生在“检查条件”和“执行修改”之间。

拿 \texttt{create(parent, name)} 举例。WriteBatch 可以把“写入新 inode”和“插入新目录项”做成原子提交，但如果提交前没锁住同名目录项键，就挡不住另一个事务在同一时间创建同名文件。RucksFS 因此选了 TransactionDB：先用 \texttt{get\_for\_update} 锁上相关的 inode 键和目录项键，再在同一事务里做条件检查和批量更新，最后通过事务提交保证跨列族修改的一致性。

从并发控制角度看，这其实就是悲观并发控制（Pessimistic Concurrency Control, PCC）的思路。目录是否为空、目标名字是否存在、某个 inode 是否还能删除，这些判断一旦离开事务边界就会暴露出 TOCTOU（Time-of-Check to Time-of-Use）窗口。RocksDB TransactionDB 的 \texttt{get\_for\_update} 允许调用方在读键时顺手拿到排他锁，把“检查条件”和“执行修改”收到同一事务上下文里。

锁粒度这件事同样重要——RucksFS 用 RocksDB 的行级锁，把冲突范围限定在具体的 inode 键和目录项键上，而不是锁整棵目录树。\texttt{create(parent, name)} 主要锁目标目录项键加必要的父目录状态，\texttt{rename} 则锁源目录项、目标目录项和相关 inode。只有真正共享这些键的并发操作才会互相阻塞。为了避免死锁，还得规定一个统一的加锁顺序；RucksFS 在后续实现里采用“按 inode 编号排序后依次加锁”。

这一点也是 RocksDB 相比早期 LevelDB 方案更适合文件系统元数据的原因。TableFS\cite{ren2013tablefs}验证了 KV 元数据的可行性，但没有原生事务接口，很多并发正确性问题只能往上层锁里塞。后面 RucksFS 对目录操作的设计，正是建立在“RocksDB 已经提供了可靠的悲观事务能力”这一事实之上。

\subsubsection{Bloom 过滤器与前缀迭代器}

Bloom 过滤器用于快速判断某个键是否“不存在于某个 SST 文件中”，从而避免无谓的磁盘访问；前缀迭代器则要求一组具有共同前缀的键在字节序上连续排列，以便对同一前缀做局部扫描。对文件系统目录项而言，这两个特性组合起来尤其关键。

若把目录项键设计为“父 inode + 文件名”的拼接形式，同一父目录下的全部子项就天然共享固定前缀。这样一来，\texttt{lookup(parent, name)} 可以退化为点查，\texttt{readdir(parent)} 则可以退化为前缀扫描。RucksFS 之所以采用大端序编码，也是为了让 inode 的数值顺序与字节序保持一致，从而使前缀范围查询在 RocksDB 中可以直接依赖字节比较器实现。第~\ref{sec:datamodel}~节讨论键编码时，将在这一基础上给出更具体的映射方案。

\subsection{POSIX 文件系统语义}

一个文件系统只要对外暴露 POSIX 接口，就要遵守一整套关于 inode、目录结构、权限位和错误码的约定，不能只顾“数据能否读写”。本文并不打算把 POSIX 的每个角落都实现到，而是聚焦在普通文件、目录和常用元数据操作这几块，尽量和 Linux 用户空间程序的预期保持一致。

\subsubsection{inode 模型与目录结构}

POSIX 文件系统以 inode 作为对象的持久标识。目录不是一串路径字符串的集合，而是“名字到 inode”的映射表。路径解析本质上就是从根目录开始一级一级查目录项，再顺着目录项拿到下一层 inode。普通文件、目录、符号链接共用同一种基本身份表示，只是 \texttt{mode}、\texttt{nlink} 和数据解释方式不同。

这对 RucksFS 有两个直接影响。一是路径树不能直接存成“全路径字符串到内容”的映射，不然 \texttt{rename} 就退化成对整棵子树的递归重写；二是目录项和 inode 属性得在存储上分清楚，不然 \texttt{lookup}、\texttt{readdir}、\texttt{getattr} 都不好走。第~\ref{sec:datamodel}~节的“以边为中心”键编码，就是把目录树拆解为“父目录 inode + 子名字”这类基本边关系。

\subsubsection{核心文件操作语义}

\texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rmdir} 和 \texttt{rename} 是本文最关心的 POSIX 操作。它们共同的难点不在于“写几条记录”，而在于必须同时满足若干一致性条件。表~\ref{tab:posix_ops}对这些操作的核心约束做了归纳。

\begin{table}[htbp]
    \centering
    \caption{核心 POSIX 目录操作的语义约束}
    \label{tab:posix_ops}
    \begin{tabular}{llp{7.2cm}}
    \toprule
    \textbf{操作} & \textbf{典型返回} & \textbf{关键语义约束} \\
    \midrule

    \end{tabular}

首先，目录项相关操作需要保持名字唯一。若同一父目录下已经存在名为 \texttt{foo} 的目录项，再次 \texttt{create(parent, “foo”)} 就必须返回 \texttt{EEXIST}，而不能出现两个同名条目。其次，\texttt{rmdir} 只能删除空目录；若目标目录仍含子项，就必须返回 \texttt{ENOTEMPTY}。再次，\texttt{rename} 要求原子可见：外部观察者不能看到“旧名字已经删掉，但新名字尚未建立”的中间态。

时间戳和链接计数同样属于语义的一部分。普通文件的 \texttt{create} 和 \texttt{unlink} 会更新父目录的 \texttt{mtime}/\texttt{ctime}；\texttt{mkdir} 和 \texttt{rmdir} 除了更新时间戳之外，还会改变父目录的 \texttt{nlink}。如果这些字段在高并发下仍然采用读改写模式，热点目录很快就会成为争用中心。这也是本文引入增量更新机制的直接动机之一。

最后，打开文件与删除目录项之间的关系不能被忽略。POSIX 允许一个文件在目录项被移除后继续被已打开的句柄访问，直到最后一个句柄关闭为止。这要求元数据层区分“名字是否还在目录里”和“对象是否还能被已有句柄引用”两个状态。RucksFS 在这一点上采用延迟删除策略：当 \texttt{nlink} 降到 0 但文件仍被打开时，只删除目录项而不立即清除数据，待 \texttt{release} 发生后再完成最终删除。

\subsubsection{权限模型}

POSIX 权限模型由 \texttt{uid}、\texttt{gid} 和 \texttt{mode} 三部分组成。FUSE 文件系统既可以在用户态自行实现访问控制，也可以把常规权限判断交给 Linux VFS。RucksFS 采用的是后一种方式：挂载时启用 \texttt{default\_permissions}，由内核在请求进入用户态之前完成基础权限检查。这样做的好处是，常规的 owner/group/other 语义不必在 FUSE 守护进程中重复实现；用户态更关注元数据事务和存储路径本身。

当然，这并不意味着元数据层可以忽略权限字段。文件创建和目录创建仍然必须正确记录 \texttt{uid}、\texttt{gid} 和 \texttt{mode}，否则内核后续的权限判断就失去依据。也正因如此，客户端在处理 \texttt{create} 和 \texttt{mkdir} 时会把请求发起者的身份信息一并传递给元数据服务。

这一章把 RucksFS 依赖的三件事铺了一下：FUSE 作为把 Linux VFS 请求交给用户态客户端的标准机制；RocksDB 作为一套对元数据工作负载友好的 LSM-tree + 列族 + 事务能力；POSIX 作为对目录操作、链接计数、时间戳和权限模型的行为规范。有了这些做铺垫，下一章进入 RucksFS 自己的系统设计与实现。
