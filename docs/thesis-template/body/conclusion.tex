\section{总结与展望}

\subsection{工作总结}

本文设计并实现了 RucksFS——一个以 RocksDB 为元数据引擎、通过 FUSE 客户端对外提供 POSIX 接口的分布式文件系统原型，旨在验证 LSM-tree 键值存储能否支撑高效且可并发的元数据操作。

\begin{enumerate}
    \item 设计并实现了基于 RocksDB 多列族的 FUSE 文件系统 RucksFS，支持 create、mkdir、unlink、rmdir、rename、readdir 等核心 POSIX 文件操作。采用四层列族架构（inodes、dir\_entries、delta\_entries、system）分离不同类型的元数据存储，目录项列族的键遵循"以边为中心"的编码——以父 inode 号与子文件名的大端序拼接作为键，使文件操作转化为对边的常数时间增删。
    \item 提出增量更新与懒折叠机制：文件创建或删除时不就地修改父目录记录，而是以追加写方式记录属性变化量（DeltaOp），从而将目录修改操作由"读--改--写"缩减为单次追加写入；后台压缩线程在增量日志累积超过 32 条时自动折叠，避免读路径退化。
    \item 实现了基于 PCC（Pessimistic Concurrency Control）事务模型的并发控制方案。通过 RocksDB 悲观事务完成关键键的加锁与冲突检测，并以事务提交保证 create、unlink、rename 等目录操作的跨键一致性；在临时冲突场景下，系统会执行最多 3 次自动重试。
    \item 基于 fuse3 异步接口与 tokio 运行时，实现了支持多并发客户端的 FUSE 文件系统挂载，挂载选项包括 default\_permissions 和 AllowOther，由内核 VFS 层完成 POSIX 权限检查。
    \item 在 4 节点环境中使用 mdtest 与 JuiceFS（MySQL/TiKV 后端）、NFS 等系统进行对比测试。结果表明，RucksFS 在 64 并发进程下的文件创建吞吐量显著高于 JuiceFS+MySQL 与 JuiceFS+TiKV；分层开销量化进一步表明，进程内直接访问 RocksDB 的能力明显高于经 gRPC 暴露后的端到端路径，网络往返与跨进程调用已成为当前实现的主要性能瓶颈。
\end{enumerate}

\subsection{不足与展望}

集群创建吞吐量可达 JuiceFS+MySQL 的 2.4 倍，但这一优势主要来源于"把元数据引擎和被测节点放在同一台机器上"的部署形态——一旦引入真正的网络分离，性能画面会截然不同。当前实现还有以下明显的不足：

\begin{enumerate}
    \item \textbf{gRPC 网络往返开销}。这是当前系统最大的性能短板。分布式模式下，单次 create 操作需要 4 次 RPC 调用，网络开销是本地操作的 48.6 倍——换言之，前面章节精心优化的存储路径，到了分布式场景几乎被网络延迟淹没。后续可通过合并 RPC 调用（将 4 次缩减为 1 次批量请求）以及引入客户端元数据缓存来缓解该问题。
    \item \textbf{多节点扩展能力尚未完成}。当前系统已经形成 Client、MetadataServer 与 DataServer 分离的分布式访问路径，但主实现仍以单 MetadataServer 与单默认 DataServer 为主，多节点元数据分片、数据副本与故障切换机制尚未实现。后续可继续补充元数据水平扩展与副本容错能力。
    \item \textbf{元数据预取与智能调度}。当前系统对所有元数据请求一视同仁，没有区分冷热路径。实际负载中，目录遍历后紧跟大量 stat 调用是常见模式，完全可以提前预取。后续可探索基于 eBPF（extended Berkeley Packet Filter）技术的内核态访问模式采集，结合预取策略减少用户态--内核态上下文切换的开销。不过这一方向的工程复杂度较高，短期内优先级不如前两项。
\end{enumerate}
