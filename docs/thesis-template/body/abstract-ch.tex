\abstractch

文件系统中元数据操作（创建、删除、重命名等）占 I/O 负载的 60\%--80\%，元数据的存储结构与操作效率对系统吞吐量有着决定性的影响。传统文件系统以 B+ 树管理目录项和 inode 索引，每次增删都需就地改写磁盘页并同步日志，在高并发小文件场景下会引发大量随机写入和目录级锁竞争。其根本原因是 B+ 树以页为操作粒度，哪怕只增删一条目录项，也要读写整个页。键值（KV）存储将操作粒度细化到单条记录：每条目录项独立存为一个键值对，增删只需一次追加写入，不再涉及页结构维护。基于 LSM-tree（Log-Structured Merge-tree）的 KV 引擎在此基础上将大量追加写入在内存中聚合后批量顺序刷盘，写入代价相比就地修改可低一个数量级。

本课题基于上述思路设计并实现了 RucksFS——以 RocksDB 为存储引擎、通过 FUSE 提供 POSIX 接口的用户态文件系统，实现了包括 create、mkdir、unlink、rmdir、rename、readdir、setattr 在内的完整文件操作集合。在键值存储上实现这些操作，需要回答三个设计问题：元数据如何建模为键值结构、父目录的写放大如何控制、并发操作的正确性如何保障。

存储建模方面，系统采用多列族架构，将 inode 属性与目录项分置于独立列族，分别配置各自适用的布隆过滤器与压缩策略。目录项列族的键编码遵循"以边为中心"的思路：以父目录编号与子文件名的大端序拼接作为键，每条目录项直接对应目录树中一条父子边。文件操作由此转化为对边的增删——create 插入一条边，unlink 删除一条边，rename 删旧边、插新边——代价均为常数时间。作为对比，以全路径作为键的方案在 rename 时需递归改写子树中每一条键，开销随子树规模线性增长。此外，同一父目录的全部子边在键空间中自然相邻，目录遍历与单项查找均可通过前缀扫描和点查高效完成。

写放大方面，POSIX 语义要求 create 和 unlink 同步更新父目录的时间戳与链接计数，高并发下同一父目录极易成为写热点，若仍采用读取-修改-写回的方式，吞吐量会急剧下降。本课题引入增量更新与懒折叠机制：写入时只追加一条属性增量记录，不触碰原始 inode 值，将就地改写转化为追加写入。读取时将基础值与尚未合并的增量在线叠加后返回，保证调用方看到的始终是最新状态。后台线程定期扫描累积增量，达到阈值后一次性折叠回基础值，控制增量数量以防止读路径退化。

并发安全方面，多线程同时操作同一目录时，检查与修改之间存在固有的竞态窗口——create 需先确认无同名文件再插入，rmdir 需先确认目录为空再删除，rename 还涉及跨目录的多键原子修改。本课题采用 RocksDB 悲观事务（PCC），对涉及的键逐一加锁，将检查与修改封装在同一原子操作中，消除上述竞态。

正确性方面，系统通过 pjdfstest 测试套件验证了 chmod、chown、mkdir、rename、unlink 等 8 类操作的 POSIX 合规性。性能方面，在 4 节点集群上以 mdtest 进行多并发基准测试，64 进程下 RucksFS 的文件创建吞吐量为 9,917 ops/s，是 JuiceFS+MySQL 的 2.4 倍、JuiceFS+TiKV 的 1.7 倍。删除吞吐量达 14,179 ops/s，差距更大。分层开销量化显示，进程内直接操作 RocksDB 可达 172,814 ops/s，gRPC 网络栈造成 48.6 倍损耗——当前性能瓶颈不在元数据引擎本身，而在网络通信路径上。
