\abstractch

文件系统中元数据操作（创建、删除、重命名等）占 I/O 负载的 60\%--80\%，元数据的存储结构和操作效率在很大程度上决定了系统的整体吞吐量。传统文件系统以 B+ 树管理目录项和 inode 索引，每次增删都需要就地改写磁盘页并同步日志，在高并发小文件场景下会引发大量随机写入和目录级锁竞争。根本原因在于 B+ 树以页为操作粒度，即便只增删一条目录项，也需要将整页读出再写回。键值（KV）存储把操作粒度细化到单条记录：每条目录项独立保存为一个键值对，增删只需一次追加写入，不再涉及页结构维护。基于 LSM-tree（Log-Structured Merge-tree）的 KV 引擎在此之上将大量追加写在内存中合并，最终以顺序方式批量刷盘，使写入代价相比就地修改低一个数量级。

RucksFS 以 RocksDB 为存储引擎、通过 FUSE 提供 POSIX 接口，是本课题在上述思路下设计并实现的用户态文件系统，完整覆盖了 create、mkdir、unlink、rmdir、rename、readdir、setattr 等文件操作。在键值存储上实现这些操作，需要回答三个设计问题：元数据如何建模为键值结构、父目录的写放大如何控制、并发操作的正确性如何保障。

存储建模上，系统采用多列族架构，将 inode 属性和目录项划分到两个独立列族，为两个列族分别配置合适的布隆过滤器与压缩策略。目录项列族的键编码采用“以边为中心”的思路：父目录编号和子文件名按大端序拼接作为键，每条目录项直接对应目录树中的一条父子边。文件操作由此转化为对边的增删——create 插入一条边，unlink 删除一条边，rename 删除旧边并插入新边——代价均为常数时间。作为对比，以全路径作为键的方案在 rename 时需要递归改写子树中的每一条键，开销随子树规模线性增长。此外，同一父目录的所有子边在键空间中天然相邻，目录遍历与单项查找均可通过前缀扫描和点查高效完成。

写放大方面，POSIX 语义要求 create 和 unlink 同步更新父目录的时间戳和链接计数，高并发下同一父目录极易成为写热点；若继续采用读取--修改--写回，吞吐将急剧下降。本课题的做法是引入增量更新与懒折叠机制：写入时仅追加一条属性增量记录，不改动原始 inode 值，从而把就地改写转为追加写入。读取时将基础值与尚未合并的增量在线叠加后返回，保证调用方看到的始终是最新状态。后台线程定期扫描累积的增量，达到阈值后一次性折叠回基础值，使增量数量控制在一定范围内，避免读路径退化。

并发安全方面，多线程同时修改同一目录时，检查与修改之间存在固有的竞态窗口——create 需要先确认没有同名文件再插入，rmdir 需要先确认目录为空再删除，rename 还涉及跨目录的多键原子修改。本课题采用 RocksDB 悲观事务（PCC），对涉及的键逐一加锁，将检查与修改封装在同一个原子操作中，从而消除上述竞态。

正确性方面，系统在分布式部署（客户端与 MetadataServer 分属不同节点）下运行了 pjdfstest 的 398 个用例。按本文的实现范围划分，其中 167 个用例直接或间接依赖本文明确不实现的 mknod/mkfifo 接口，1 个落在 DataServer 当前 64\,MiB 单文件上限之外，48 个是 pjdfstest 框架自身跳过的（utimensat、remount 等，与元数据设计无关）；剩下 182 个“实现范围内”用例全部通过，通过率 100\%。

性能方面，在 1 台 64 核服务器加最多 96 台 2 核客户端的集群上使用 mdtest 进行多并发压测。$N \in \{2, 8, 32\}$ 三档下将 RucksFS 与 NFS v4.2、JuiceFS+TiKV 进行对比，RucksFS 相对 JuiceFS+TiKV 在 create、remove、stat 三类操作上分别领先约 1.5--1.7 倍、1.8--2.2 倍与 1.2 倍，主要差距来自单机直连 RocksDB 省去了 TiKV 的 Raft 与两阶段提交开销，以及 DeltaOp 对共享父目录并发的优化——可以说明本文基于 RocksDB 实现的元数据方案在性能上不逊于常见的 KV 文件系统。相对 NFS v4.2，$N=2$ 时因客户端关闭属性缓存后每次 create 需要多走两次 lookup RPC，RucksFS 略低于 NFS；$N=32$ 时 NFS 被 ext4 的 \texttt{i\_rwsem} 锁限制，RucksFS 在 create 上反超 NFS 约 4.6 倍，说明 KV 方案在共享热点目录场景下比本地文件系统的 B+ 树方案更有扩展空间。$N \geq 64$ 时两套对照系统均无法稳定运行，此时改为对比 RucksFS 自身的 Delta 与 NoDelta 两个构建变体，并沿 $\mathrm{np} = N \times r$ 两个维度扫描：$\mathrm{np}$ 跨过 96 之后 NoDelta 进入事务冲突循环，稳态吞吐停在 12\,000\,ops/s 附近；Delta 继续扩展至 38\,000\,ops/s，$\mathrm{np}=384$ 时 \texttt{create} 比值达到 2.96、\texttt{remove} 比值达到 2.63。easy 模式下两者基本一致，说明这部分收益确实来自消除共享父目录的写冲突。
