\abstractch

文件系统中元数据操作（创建、删除、重命名等）占 I/O 负载的 60\%--80\%，元数据的存储结构和操作效率对系统吞吐量有着决定性的影响。传统文件系统以 B+ 树管理目录项和 inode 索引，每次增删都要就地改写磁盘页并同步日志，在高并发小文件场景下会引发大量随机写入和目录级锁竞争。根本原因在于 B+ 树以页为操作粒度，哪怕只增删一条目录项，也要把整页读出来再写回去。键值（KV）存储把操作粒度细化到单条记录：每条目录项独立存成一个键值对，增删只需一次追加写入，不再涉及页结构维护。基于 LSM-tree（Log-Structured Merge-tree）的 KV 引擎在此之上又把大量追加写在内存里聚合成批量顺序刷盘，写入代价相比就地修改可以低一个数量级。

本课题基于上述思路设计并实现了 RucksFS——以 RocksDB 为存储引擎、通过 FUSE 提供 POSIX 接口的用户态文件系统，实现了包括 create、mkdir、unlink、rmdir、rename、readdir、setattr 在内的完整文件操作集合。在键值存储上实现这些操作，要回答三个设计问题：元数据怎么建模成键值结构、父目录的写放大怎么控制、并发操作的正确性怎么保障。

存储建模上，系统采用多列族架构，把 inode 属性和目录项分到两个独立列族，各自配置适合自己的布隆过滤器和压缩策略。目录项列族的键编码走“以边为中心”的思路：父目录编号和子文件名按大端序拼起来作为键，每条目录项直接对应目录树里的一条父子边。文件操作由此变成对边的增删——create 插入一条边，unlink 删除一条边，rename 删旧边、插新边——代价都是常数时间。作为对比，以全路径作为键的方案在 rename 时需要递归改写子树里的每一条键，开销随子树规模线性增长。此外，同一父目录的所有子边在键空间里自然相邻，目录遍历和单项查找都可以通过前缀扫描和点查高效完成。

写放大上，POSIX 语义要求 create 和 unlink 同步更新父目录的时间戳和链接计数，高并发下同一父目录很容易成为写热点，如果还走读取-修改-写回，吞吐就会急剧下跌。本课题的做法是引入增量更新与懒折叠机制：写入时只追加一条属性增量记录，不去碰原始 inode 值，把就地改写转成追加写入。读取时把基础值和尚未合并的增量在线叠加后返回，保证调用方看到的始终是最新状态。后台线程定期扫描累积的增量，达到阈值后一次性折叠回基础值，把增量数量控制在一定范围内，避免读路径退化。

并发安全上，多线程同时动同一目录时，检查和修改之间存在固有的竞态窗口——create 要先确认没有同名文件再插入，rmdir 要先确认目录为空再删除，rename 还涉及跨目录的多键原子修改。本课题采用 RocksDB 悲观事务（PCC），对涉及的键逐一加锁，把检查和修改封装在同一个原子操作里，消掉上面这些竞态。

正确性方面，系统在分布式部署（客户端与 MetadataServer 分属不同节点）下跑了 pjdfstest 的 398 个用例。按本文的实现范围划分，其中 167 个用例直接或间接依赖本文明确不实现的 mknod/mkfifo 接口，1 个踩在 DataServer 开发期的 64\,MiB 单文件上限上，48 个是 pjdfstest 框架自身跳过的（utimensat、remount 等与元数据设计无关）；剩下 182 个“实现范围内”用例全部通过，通过率 100\%。

性能方面，在 1 台 64 核服务器加最多 96 台 2 核客户端的集群上用 mdtest 跑多并发压测。在 $N \in \{2, 8, 32\}$ 的可比区间里做三方横向对比，RucksFS 相对 JuiceFS+TiKV 在 create、stat、remove 三类操作上保持 2--5 倍的稳定优势，说明“针对文件系统元数据专门设计的 KV 布局 + 单机直连 RocksDB”相对“通用分布式 KV 叠 FUSE”省下的开销是实实在在的；相对 NFS v4.2 在 $N=2$ 处已有 3 倍领先，$N$ 增大后持续拉开到 38 倍，反映 LSM-tree 加边中心键编码相对 B+ 树加目录锁在元数据写入上有结构性优势。高并发区间（$N \geq 64$）两套对照系统各自因外部原因（TiKV 单节点部署上限、ext4 目录锁）失去可比性，此时改做 Delta 与 NoDelta 两种编译变体的受控消融，比较干净地验证增量更新机制本身的作用：$N=96$ 时 create 吞吐量比值达到 5.81 倍、stat 比值 136 倍，而在无目录竞争的独立目录模式下二者完全重合，确认这部分收益确实来自消除共享父目录的写冲突。
