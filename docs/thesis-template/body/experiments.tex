\section{测试与结果分析}
\label{sec:experiments}

第~\ref{sec:design}~章从设计角度给出了 RucksFS 的元数据组织方案、事务模型和增量更新机制。本章的任务是检验这些设计在真实分布式部署下是否能站住脚。实验围绕两条线索展开。

首先是 POSIX 语义合规性。使用 pjdfstest 套件在分布式部署环境下（客户端与服务器分属不同节点）运行 398 个用例，覆盖 13 类系统调用，以此判断 RucksFS 在 FUSE + RocksDB 组合之上能否正确表达 POSIX 接口的行为约定。

其次是分布式场景下的元数据性能。外部对比层面，将 RucksFS 与 NFS v4.2 和 JuiceFS+TiKV 一同在相同硬件上压测，观察横向竞争力；内部对比层面，利用编译期特性开关构造 Delta 与 NoDelta 两种变体，在严格控制变量的条件下直接量化增量更新机制的实际收益。

两类实验共享同一套硬件环境和测试工具，数据可直接交叉引用。在展开具体实验之前，表~\ref{tab:design-test-map} 先列出第~\ref{sec:design}~章中提出的几项关键设计决策，以及本章中用来验证它们的对应实验。后续各节可按此表回溯。

服务器选用 64 核大规格机型，目的是确保在所有并发档位下，被测系统的元数据后端都不会因 CPU 饱和而先于测试本身触顶。客户端则选用 2\,vCPU、2\,GiB 内存的最小规格。mdtest 的单 rank 在 FUSE 路径上的有效吞吐受内核 VFS 串行化限制，约为 1\,000\,ops/s 量级，2 核即可饱和；更大规格只会增加成本而不会改变结果，且会占用配额导致无法继续扩展客户端总数。所有客户端的硬件规格保持一致，排除横向不可比的因素。

\subsubsection{被测系统}

本章设置了三套被测系统和 RucksFS 自身的两种构建变体，每次只运行一套，切换时完全清场。

\begin{itemize}
    \item \textbf{RucksFS-Delta}：默认构建，启用第~\ref{sec:design}~章第~\ref{sec:txn_model}~节描述的 DeltaOp 增量追加机制。父目录的 \texttt{mtime}、\texttt{ctime}、\texttt{nlink} 更新走增量路径。
    \item \textbf{RucksFS-NoDelta}：编译期通过 \texttt{cargo} 特性开关关闭 DeltaOp。父目录属性更新退化为对父 inode 基值的读--改--写，其余代码与 Delta 变体完全相同。该变体仅作消融对照，不是生产构建。
    \item \textbf{NFS}：Linux 内核 NFS v4.2 服务端（\texttt{nfs-kernel-server}），导出本地 ext4 目录。客户端以 \texttt{vers=4.2,noac} 挂载，关闭属性缓存，确保每次 \texttt{stat} 都到达服务端。\texttt{nfsd} 线程数为 128。
    \item \textbf{JuiceFS+TiKV}：JuiceFS 1.3.1，元数据后端为单节点 TiKV 8.5.6（含 PD），数据存储指向同机本地目录。这一组合对应业界常见的"通用分布式 KV + 分层文件系统"路线。JuiceFS 与 RucksFS 共用 RocksDB 作为底层存储引擎（TiKV 内部即使用 RocksDB），便于隔离上层架构差异。
\end{itemize}

服务器在切换被测系统时执行严格的清场流程：停止所有守护进程，删除数据目录，确认监听端口释放，再启动下一系统。客户端侧同样卸载挂载点、清理 FUSE 残留连接并重建挂载目录。该流程由统一的 orchestrator 脚本驱动，保证每次测试从干净状态开始。

\subsubsection{评估工具与执行流程}

POSIX 合规性使用 pjdfstest（Rust 实现版本），共 398 个用例，覆盖 \texttt{chmod}、\texttt{chown}、\texttt{link}、\texttt{mkdir}、\texttt{mkfifo}、\texttt{mknod}、\texttt{open}、\texttt{rename}、\texttt{rmdir}、\texttt{symlink}、\texttt{truncate}、\texttt{unlink}、\texttt{ftruncate} 等系统调用。

性能测试使用 mdtest 4.1.0（IOR 套件子项），以 OpenMPI 4.1.2 跨客户端调度 mpi 进程。每台客户端挂载一次目标文件系统，在该挂载点上启动一个 mdtest rank，因此并发度 $N$ 等于客户端数量。按 $N=2,8,32,64,96$ 五档运行；每档分别在共享父目录（hard 模式，所有 rank 写同一目录）和独立父目录（easy 模式，每 rank 写自己的子目录）下各执行一次完整的 \texttt{create-stat-remove} 流程。为防止大 $N$ 时单次测试过长，每 rank 的文件数随 $N$ 递减，具体数值见表~\ref{tab:files-per-rank}。

mdtest 的核心参数为 \texttt{-F -C -T -r -i 1}：\texttt{-F} 仅测文件操作，\texttt{-C}/\texttt{-T}/\texttt{-r} 分别执行创建、查询、删除三个阶段，\texttt{-i 1} 不做内部重复。hard 与 easy 的区别由 \texttt{-u} 控制：默认所有 rank 在同一父目录下操作（hard），加 \texttt{-u} 后各 rank 在独立子目录下操作（easy）。本章默认讨论 hard 模式数据，easy 模式数据用于对照。

\subsection{POSIX 合规性评估}
\label{sec:pjdfstest}

POSIX 合规性测试在分布式部署下运行——客户端与 MetadataServer 分属不同节点，请求经过 gRPC 传输。因此，用例要通过的前提条件不仅包括 MetadataServer 内部的事务逻辑正确，还包括错误码能够穿过 gRPC 与 FUSE 后保持不变。这一点在后续的失败分析中会被反复提到。

以一条典型用例说明。pjdfstest 中"当 \texttt{rename} 的目标路径前缀不是目录时应返回 \texttt{ENOTDIR}"这条规则，会按 regular、dir、symlink、fifo、socket、char、block 七种文件类型各生成一条用例。后四种依赖 \texttt{mknod} 或 \texttt{mkfifo} 来准备测试对象。RucksFS 没有实现这两个接口，用例在前置阶段就因 \texttt{EOPNOTSUPP} 中断，无法进入 \texttt{rename} 本身的语义检查。测试记录为"\texttt{rename} 的某条规则失败"，但实际上 RucksFS 的 \texttt{rename} 对常规文件、目录和符号链接都返回了正确结果。168 个失败中有 161 个属于这一模式。

\subsubsection{与第~\ref{sec:design}~章设计决策的对照}
\label{sec:pjdfstest-semantic}

为避免"因为没实现所以不会失败"的误读，下面列出几条容易被忽视的边界语义，它们都属于 pjdfstest 通过列，且与第~\ref{sec:design}~章的具体设计决策直接对应。

\begin{itemize}
    \item \textbf{\texttt{unlink::open\_file\_not\_freed}}（验证目标：删除语义与句柄生命周期，第~\ref{sec:txn_model}~节）。文件被 \texttt{unlink} 后若仍有打开句柄，应可继续读写，物理回收延迟到最后一个句柄关闭。这条用例要求 MetadataServer 同时维护 \texttt{nlink} 与打开计数：当 \texttt{nlink} 归零但计数仍大于零时，对象进入 \texttt{pending\_deletes}，由 \texttt{release} 触发最终清理。如果事务未能正确处理"删除目录项"与"维持 inode 可达性"之间的分离，这条用例就会失败。
    \item \textbf{\texttt{rename::to\_multiply\_linked::regular}}（验证目标：rename 覆盖逻辑，算法~\ref{alg:rename}）。若 \texttt{rename} 目标是一个有多条硬链接的文件，覆盖动作只能将目标的 \texttt{nlink} 减一，不能直接删除目标 inode。这对应第~\ref{sec:design}~章 \texttt{rename} 中"按覆盖删除逻辑处理目标对象"的步骤。
    \item \textbf{12 条 \texttt{*::enametoolong\_component} 用例}（验证目标：错误码穿透）。文件名长度超过 \texttt{NAME\_MAX} 时必须返回 \texttt{ENAMETOOLONG}。该约束在 MetadataServer 所有接受 \texttt{name} 字段的 RPC 入口统一校验，错误码经 gRPC 与 FUSE 两层传递后保持不变。
    \item \textbf{\texttt{open::open\_trunc}}（验证目标：VfsCore 协调路径，第~\ref{sec:data_path}~节）。以 \texttt{O\_TRUNC} 打开已有文件时，\texttt{size} 必须在打开操作中被同步清零，同时刷新 \texttt{mtime} 和 \texttt{ctime}。FUSE 不一定额外发起独立的 \texttt{setattr}，因此这一行为必须在 \texttt{open} 的实现中一并完成。
    \item \textbf{\texttt{rmdir::eexist\_enotempty\_non\_empty\_dir} 系列}（验证目标：事务内空目录检查，第~\ref{sec:posix_impl}~节）。删除非空目录应返回 \texttt{ENOTEMPTY}。第~\ref{sec:design}~章说明 \texttt{rmdir} 把空目录检查放进了事务内部，通过一致性视图下的前缀扫描完成，避免检查结果在事务外失效。
\end{itemize}

上述用例覆盖的不是孤立的接口行为，而是第~\ref{sec:design}~章描述的跨列族原子提交、句柄生命周期维护、统一加锁顺序和客户端协调路径在实际运行中的综合表现。它们的通过说明，这些设计在分布式部署条件下确实达到了预期的语义效果。

\subsubsection{已知不实现项的取舍}

RucksFS 对 \texttt{mknod} 与 \texttt{mkfifo} 的处理是显式不实现，而非未完成。理由有二。其一，这两个接口管理的是 Unix 域 socket、命名管道、字符设备节点和块设备节点，其运行语义与内核设备号、内核管道缓冲区强绑定，与"通过键值对持久化命名空间"的核心研究目标正交。其二，本文聚焦的负载（容器镜像、训练数据、日志、构建产物）几乎不产生这类对象。把工程预算用于不会被使用的接口，对论证元数据机制本身没有帮助。\texttt{open::interact\_2gb} 涉及的单文件大小上限属于 DataServer 当前的开发期参数，调整分片策略即可放开。

48 个跳过用例中，27 个来自 \texttt{utimensat} 系列（该功能需要单独的 feature flag，当前未启用），其余 21 个涉及 \texttt{remount}（FUSE 配置不允许运行时重新挂载）和跨文件系统操作（测试环境未配置第二文件系统）。这些跳过项与 RucksFS 的元数据设计无关。

综上，pjdfstest 的结果可以这样理解：在 RucksFS 选择实现的接口范围内，POSIX 语义被完整地表达；未实现的接口被显式标记，且不以错误形式遮蔽真实问题。这一结论给后续的性能对比确立了一个前提——后面看到的吞吐数字，都建立在"语义正确"的基础之上。

NFS 的曲线几乎是平的：从 $N=2$ 的 605\,ops/s 到 $N=96$ 的 911\,ops/s，增幅不到 1.5 倍。瓶颈在 ext4 的目录互斥锁（\texttt{i\_rwsem}）——所有 \texttt{nfsd} 线程在同一父目录上的 \texttt{create} 都要先拿到该目录 inode 的写锁，再走 journal 提交。无论加多少客户端，实际能并发执行的 \texttt{create} 数量被这把锁限制在很低的水平。

JuiceFS+TiKV 在可测的三个档位上保持准线性增长。其元数据更新经 Percolator 两阶段提交，每次 \texttt{create} 在 TiKV 内部表现为一组对独立键的写入加 \texttt{commit\_ts} 写入。共享父目录会引发对父 inode 键的并发更新，但 MVCC 按时间戳排序，冲突被推迟到后台 GC 而非在事务路径上同步等待。不过 RucksFS-Delta 在同一区间内始终高出 2.8--4 倍。差距来源于上层架构：RucksFS 的一次 \texttt{create} 是 MetadataServer 内部的单次 \texttt{AtomicWriteBatch}，跨四类键共享同一次 WAL 同步；TiKV 的一次 \texttt{create} 仍要走完整的两阶段提交并经过 Raft 日志（即便单节点也未跳过 Raft 模块）。

RucksFS-Delta 在整个区间保持近线性增长，从 $N=2$ 的 1\,853\,ops/s 到 $N=96$ 的 68\,588\,ops/s，每客户端贡献约 714\,ops/s。$N=96$ 时对 NFS 的吞吐倍数为 75，对 JuiceFS+TiKV 在 $N=32$ 处（其最高数据点）的倍数为 4。这一差距并非偶然，与第~\ref{sec:design}~章的设计直接对应：\texttt{create} 被拆解成"一条目录项写入、若干 inode 记录写入、一次父目录属性增量追加"的组合，而 DeltaOp 又把父目录更新从"读--改--写"转变为追加写，把 LSM-tree 擅长的顺序追加发挥到了端到端层面。

\subsubsection{查询吞吐：写路径未阻塞读路径}

stat 的差距比 create 更悬殊。$N=96$ 时 RucksFS-Delta 达到 1\,792 万 ops/s，NFS 为 4.5 万 ops/s，相差约 400 倍。之所以差距如此之大，需要分别看两侧的路径。

NFS 客户端使用 \texttt{noac} 挂载，每次 \texttt{stat} 都经网络到达服务端，再经 ext4 的 inode 查找返回。本测试中所有文件都在同一父目录下，\texttt{lookup} 阶段要拿父目录读锁，会被 \texttt{create} 留下的目录元数据更新挤占。对 RucksFS 而言，MetadataServer 的 \texttt{stat} 只是 RocksDB 的一次点查；在 \texttt{create} 阶段写入的 inode 记录已经在 BlockCache 中驻留，命中后延迟在微秒级，且与写路径不争同一把锁——这是 LSM-tree 读写不互锁的天然优势。

更关键的是，DeltaOp 在这里起了间接作用：它把父目录的属性写从热路径上拿走，使得 \texttt{stat} 的 \texttt{lookup} 阶段不再被父 inode 的写锁阻塞。这一点将在第~\ref{sec:delta-vs-nodelta}~节的 NoDelta 对照中得到直接量化。

\subsubsection{删除吞吐}

remove 的整体形态与 create 相似。NFS 的 \texttt{unlink} 同样要拿父目录写锁再写 journal，扩展性曲线与 create 平行。JuiceFS+TiKV 的 \texttt{unlink} 在 TiKV 中表现为写入 tombstone，物理回收延迟到 GC，事务体量与 \texttt{create} 接近。RucksFS-Delta 的 \texttt{unlink} 同样只动几条键值记录加一次父目录增量追加，因此性能曲线也与 create 相近。$N=96$ 时 RucksFS-Delta 达到 70\,859\,ops/s，对 NFS 约 23 倍，对 JuiceFS+TiKV 在 $N=32$ 处约 4 倍。

\subsection{DeltaOp 增量机制的内部对比}
\label{sec:delta-vs-nodelta}

第~\ref{sec:design}~章第~\ref{sec:txn_model}~节的核心设计假设是：把父目录属性更新从"覆盖父 inode 基值"改为"向独立列族追加 DeltaOp"，可以避免热点目录下的事务冲突，使元数据写入在并发度提升时仍维持线性扩展。本节用 Delta 与 NoDelta 两种构建变体在同一硬件、同一负载下的直接对比来检验这一假设。这是本章最重要的消融实验：如果假设成立，应该满足三个可预测的条件——（1）低并发时两者无差距，（2）高并发共享目录下 NoDelta 出现非线性下滑而 Delta 保持线性，（3）独立目录（easy 模式）下两者始终重合。

\subsubsection{NoDelta 吞吐崩溃的机理}

NoDelta 从 $N=32$ 起出现急剧下滑，其原因可以从 MetadataServer 端的事务执行细节中解释。

NoDelta 路径下，一次 \texttt{create} 的事务序列为：对父目录 inode 键执行 \texttt{get\_for\_update} 获取写锁，读出当前基值，改写 \texttt{mtime}/\texttt{ctime} 后 \texttt{put} 回去，连同新建的目录项与 inode 一起提交。在共享父目录下，所有 \texttt{create} 事务都竞争同一条父 inode 键的写锁。第~\ref{sec:design}~章描述的悲观并发控制使拿不到锁的事务进入等待；等待超时则回滚并由服务端重试。当并发度足够高时，重试本身会形成放大效应：被拒绝的事务在释放已持有的其他锁、回滚已写入的 batch 后再次进入争抢，但此时队列中已经堆积了更多重试请求。这就是写冲突重试风暴。

启用 DeltaOp 后情况截然不同。\texttt{create} 不再触碰父 inode 基值，父目录的属性变化被编码为一条 DeltaOp 写入 \texttt{delta\_entries} 列族。由于键中包含单调递增的序列号 \texttt{seq}，每次写入的目标键都不同，根本不会与其他事务在同一键上产生锁冲突。父 inode 基值只在后台 \texttt{DeltaCompactionWorker} 折叠增量时被写入一次，且发生在测试关键路径之外。锁冲突被消除的同时，热点目录上的随机写也被转换为顺序追加。

把机理对照数字：NoDelta 在 $N=32$ 时尚能保持线性（锁等待尚未触发明显放大），$N=64$ 处的 40\% 下滑对应冲突重试开始堆积，$N=96$ 处的持续下降对应重试窗口趋近饱和。Delta 不进入这种状态，因此一直保持线性。

\textbf{remove} 与 \texttt{create} 的形态一致。\texttt{unlink} 同样要更新父目录的 \texttt{mtime}/\texttt{ctime}，机制完全相同。$N=96$ 时比值达到 6.12（略高于 create 的 5.81），可解释为 \texttt{unlink} 路径中 NoDelta 同时还要更新目标 inode 的 \texttt{nlink}，延长了锁持有时间。

\textbf{stat} 的差距更引人注目——$N=96$ 时比值达到 136。\texttt{stat} 在 RocksDB 端是只读路径，理论上不会被 DeltaOp 直接影响。但在 NoDelta 路径下，并发的 \texttt{create} 持续争抢父 inode 键的写锁，而 \texttt{stat} 在 \texttt{lookup} 阶段需要读取父目录的键状态。RocksDB 悲观事务的实现要求读路径在写锁持有期间等待，父 inode 的写锁就成了 \texttt{stat} 的隐性瓶颈。DeltaOp 把父 inode 写从关键路径上拿走之后，\texttt{stat} 不再被这把锁拖累，吞吐重新由 RocksDB 的点查能力决定。可以这样概括：DeltaOp 不仅消除了写路径的冲突，也间接把读路径从写争抢中解放出来。这一效应是第~\ref{sec:design}~章未明确预测、但与设计逻辑一致的额外收益。

\subsubsection{对第~\ref{sec:design}~章设计假设的实验验证}

把上述观察整理为对第~\ref{sec:design}~章设计陈述的逐点回应：

\begin{itemize}
    \item \textbf{"DeltaOp 在低并发下不引入额外开销"}（第~\ref{sec:txn_model}~节）。$N \leq 32$ 时 Delta 与 NoDelta 的吞吐差距在 2\% 以内。增量追加路径只在事务热路径上节省锁竞争，不增加固定开销。
    \item \textbf{"DeltaOp 在高并发下消除父目录事务冲突"}（第~\ref{sec:txn_model}~节）。$N=64$ 起 NoDelta 出现典型的冲突崩溃，Delta 维持线性。两条曲线的分叉位置恰好对应 PCC 争抢窗口被放大的临界点。
    \item \textbf{"DeltaOp 的优势仅在共享目录场景中出现"}。easy 模式下二者吞吐完全重合，hard 模式下才出现差距。这排除了 DeltaOp 收益来自其他系统层面副作用的可能性。
    \item \textbf{读路径间接受益}。\texttt{stat} 在 NoDelta 下被父 inode 写锁阻塞，在 Delta 下不再受阻。这一效应说明 DeltaOp 的收益不止于写入路径。
\end{itemize}

四个观察互相印证，构成对 DeltaOp 增量机制的端到端实验验证：第~\ref{sec:design}~章的理论设计在分布式部署中确实落到了可被实测的性能差异上，且差异方向、幅度和并发拐点都与机制本身的工作原理吻合。

\subsection{本章小结}

本章围绕两条线展开实验。

第一条线是 POSIX 合规性。pjdfstest 的 398 个用例中，182 条直接通过；168 条失败里 167 条来自 \texttt{mknod}/\texttt{mkfifo} 派生出的特殊文件类型用例，1 条来自 DataServer 的开发期文件大小上限。与核心元数据语义直接相关的失败为零。打开后 \texttt{unlink}、硬链接覆盖、长名称错误码、\texttt{O\_TRUNC} 时间戳、事务内空目录检查等边界用例全部通过，对应验证了第~\ref{sec:design}~章的跨列族原子提交、句柄生命周期维护、统一加锁顺序和客户端协调路径。\texttt{mknod}/\texttt{mkfifo} 的不实现是面向典型分布式存储负载的显式取舍，不遮蔽真实问题。

第二条线是分布式性能。在 1 台 64 核服务器加最多 96 台 2 核客户端的集群上，RucksFS-Delta 在 \texttt{create}、\texttt{stat}、\texttt{remove} 三类操作上均对 NFS 与 JuiceFS+TiKV 保持大幅领先。$N=96$ 时 \texttt{create} 达到 6.86 万 ops/s，\texttt{remove} 达到 7.09 万 ops/s，对 NFS 分别为 75 倍与 23 倍；对 JuiceFS+TiKV 在其可完成的最高档位 $N=32$ 处分别为 4 倍与 4 倍。NFS 的扩展瓶颈在 ext4 目录锁，TiKV 的瓶颈在两阶段提交叠加 Raft 日志。

DeltaOp 的消融实验是本章的核心结论。在仅切换编译特性开关的条件下，hard 模式从 $N=64$ 起 Delta 与 NoDelta 发生显著分叉：$N=96$ 时 \texttt{create} 比值 5.81，\texttt{remove} 比值 6.12，\texttt{stat} 比值 136。easy 模式下二者完全重合。这种"同硬件、同负载、只换一个机制开关"的对照直接说明：第~\ref{sec:design}~章描述的增量追加机制在分布式部署中真实承担了消除写冲突与缓解读路径阻塞两项职责——它不是停留在设计图纸上的优化思路，而是在 $N=96$ 客户端的真实压力下被实验数据所验证的有效机制。
