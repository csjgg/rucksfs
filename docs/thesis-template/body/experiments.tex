\section{实验与结果分析}
\label{sec:experiments}

本章评估 RucksFS 在分布式部署下的元数据操作性能。实验聚焦两个核心对比对象——NFS v4.2 与 JuiceFS+TiKV，分别代表"传统内核文件系统"和"通用分布式键值存储之上的分层文件系统"两条主流路线。为了消除早期粗放测试中的若干干扰因素，本章采用对称 2 服务器拓扑、控制变量设计，并对每个系统各自调到最优配置。测试工具为 mdtest，评估维度为文件创建、stat 查询和删除三类操作在 $N=1,2,4,8,16,32$ 并发进程下的吞吐量。

\subsubsection{对比系统的选择}

在本研究的早期轮次中曾同时引入 JuiceFS+MySQL、JuiceFS+Redis、CubeFS 等作为参考系统。在本章最终评估中，主对比对象收窄至 NFS 与 JuiceFS+TiKV 两个，理由如下：

\paragraph{NFS 作为"传统内核文件系统"的基线} NFS v4.2 是 Linux 生态中最成熟、最广泛部署的网络文件系统，其服务端通过内核 VFS 直接访问 ext4——ext4 使用 B-tree 管理目录项、通过 journal 保证元数据崩溃一致性。选择 NFS 作为基线的意义在于：它代表了"不使用键值存储、直接使用内核文件系统做元数据"这条技术路线。RucksFS 相对 NFS 的性能差异，可以干净地归因于"LSM-Tree 键值存储 vs.\ B-tree + journal"这一核心架构选择。同时 NFS 的 FUSE 路径在内核态实现，不经过用户态上下文切换，理论上对 NFS 有利——RucksFS 若仍能在此条件下胜出，说明 LSM-Tree 在元数据工作负载下的写放大优化足够抵消 FUSE 的额外开销。

\paragraph{JuiceFS+TiKV 作为"通用分布式 KV 之上的分层文件系统"的代表} JuiceFS 是目前云原生场景中应用最广泛的开源 FUSE 文件系统之一；TiKV 是一个基于 RocksDB 的分布式键值存储，底层存储引擎与 RucksFS 完全相同。选择 JuiceFS+TiKV 作为对比对象的意义在于：它和 RucksFS 共享 RocksDB 这个底层存储引擎，性能差异可以纯粹归因于上层架构——JuiceFS 把元数据存储委托给一个通用分布式 KV 系统（Raft + MVCC），而 RucksFS 自研了一个专用的 MetadataServer（PCC + WriteBatch，无 Raft）。这组对比回答一个关键问题：\textbf{"基于同一底层存储引擎，专用元数据服务器能否显著优于通用分布式 KV？"}

\paragraph{其他系统为何不作为主对比} MySQL 作为关系数据库与 LSM-Tree KV 差异过大，对比意义弱；JuiceFS+Redis 使用内存 KV，与 RucksFS 的磁盘存储不在同一量级（附录 D 保留其数据作为"内存 KV 性能上限"的参考）；CubeFS 在本实验的单机部署下无法发挥其多节点 Raft 架构的设计意图，数据不具参考价值。

\subsection{测试方法}

\subsubsection{mdtest 基准测试工具}

mdtest 是 IOR/mdtest 套件的组成部分，专门用于测量文件系统的元数据操作速率\cite{ior2024mdtest}。它支持 MPI 多进程并发、可精确控制每进程的文件数量和目录布局，在 HPC 领域是事实上的元数据性能标准工具。

与 fio 或 IOZone 等数据面工具不同，mdtest 操作粒度极细——每次只创建、stat 或删除一个空文件——能够精确暴露 FUSE 层、RPC 层和存储引擎在单次操作上的延迟。这正是元数据密集型工作负载的核心瓶颈所在。

\subsubsection{测试参数与负载模式}

mdtest 的调用命令如下：

\begin{verbatim}
mpirun --allow-run-as-root --oversubscribe -np $N \
    mdtest -d <mountpoint>/mdtest_run \
           -n 5000 -F -C -T -r -u -i 3
\end{verbatim}

各参数的含义：\texttt{-C} 执行文件创建阶段，\texttt{-T} 执行 stat 阶段，\texttt{-r} 执行删除阶段，\texttt{-F} 限定仅测试文件（不测目录树），\texttt{-n 5000} 表示每个进程操作 5\,000 个文件，\texttt{-u} 使每个进程在独立子目录下工作，\texttt{-i 3} 表示每次 mdtest 调用内部重复 3 轮，取最大值作为单次运行的结果。

并发度 $N$ 从 1 依次增至 2、4、8、16、32，共 6 档。每档下独立运行 3 次，每次运行前执行完整的缓存清理（Client + Server 两侧都执行 \texttt{sync \&\& echo 3 > /proc/sys/vm/drop\_caches}），以消除冷热缓存对 stat 操作的影响。每档的最终报告值为 3 次运行的算术平均。

为防止跨文件系统的测试互相干扰，测试采用严格的串行执行：一个文件系统的全部 $N$ 档跑完后，卸载挂载点、清理服务端残留目录，再开始下一个系统。

\subsubsection{POSIX 合规性测试}

除性能测试外，还使用 pjdfstest\cite{pjdfstest2013nectec} 套件检验 RucksFS 的 POSIX 语义正确性。pjdfstest 包含约 8\,000 个测试用例，覆盖 chmod、chown、link、mkdir、mkfifo、open、rename、rmdir、symlink、truncate、unlink 等 12 类操作。

RucksFS 通过了 chmod、chown、mkdir、open、rename、rmdir、truncate、unlink 8 个类别的全部测试。未通过的 4 个类别——link、symlink、mkfifo、mknod——是因为 RucksFS 当前版本尚未实现硬链接、符号链接和特殊文件节点。这些功能不在毕业设计的范围之内，但接口预留已完成，后续可以逐步补充。

\subsection{对比系统的元数据操作路径分析}
\label{sec:metadata-path-analysis}

理解后续性能数据的关键，是先看清 RucksFS 和 JuiceFS+TiKV 在"一次 create"这件事上到底做了哪些工作。两者底层都用 RocksDB 存储键值对，但进入 RocksDB 之前的路径差异巨大。本节以文件创建为例，对比两者的元数据操作流程。

\subsubsection{RucksFS 的 create 路径}

RucksFS 为每个 inode 在 RocksDB 中维护如下键值对（具体编码见第二章第三节）：

\begin{itemize}
    \item \texttt{I/\textless inode\_be\textgreater} $\to$ \texttt{InodeValue}（大端编码保证字节序与数值序一致）
    \item \texttt{D/\textless parent\_be\textgreater/\textless name\textgreater} $\to$ \texttt{\textless child\_inode\_be\textgreater}
    \item \texttt{L/\textless inode\_be\textgreater} $\to$ \texttt{DataLocation}（数据位置元信息）
    \item \texttt{X/\textless parent\_be\textgreater/\textless seq\_be\textgreater} $\to$ \texttt{DeltaOp}（父目录更新的 delta 记录）
\end{itemize}

一次 create 在 MetadataServer 内部以单次 \texttt{AtomicWriteBatch} 原子提交，伪代码如下：

\begin{verbatim}
fn create(parent, name, mode, uid, gid) -> InodeAttr:
    loop:  // 乐观并发控制重试循环（最多 3 次）
        batch = begin_write()
        // 1. 读父目录 inode，加行锁
        parent_iv = batch.get_for_update(encode_inode_key(parent))
        if parent_iv is None: return ENOENT
        // 2. 检查重名
        dentry_key = encode_dentry_key(parent, name)
        if batch.get(dentry_key) is not None: return EEXIST
        // 3. 分配新 inode
        child_ino = inode_allocator.next()
        child_iv  = InodeValue::new_file(child_ino, mode, uid, gid)
        // 4. 一次性写入四类键
        batch.put(encode_inode_key(child_ino), child_iv.serialize())
        batch.put(dentry_key, child_ino.to_be_bytes())
        batch.put(encode_data_location_key(child_ino), default_loc)
        batch.append_delta(parent, DeltaOp::ChildAdded { mtime, nlink_delta: +1 })
        // 5. 单次 fsync 提交 WAL
        match batch.commit():
            Ok => return child_iv.to_attr()
            Err(Conflict) => continue  // 仅 get_for_update 行锁冲突时重试
\end{verbatim}

关键点：\textbf{整个 create 在 MetadataServer 侧展开为一次 \texttt{WriteBatch::commit()}，包含 4 次 KV 写入和一次 WAL \texttt{fsync}}。客户端与 MetadataServer 之间只有 1 次 gRPC 往返。乐观并发控制（PCC）仅在同一父目录下并发创建时才可能触发重试，且由于 \texttt{-u} 参数让每个进程使用独立子目录，重试率在本实验中近乎为零。

\subsubsection{JuiceFS+TiKV 的 create 路径}

JuiceFS 在 TiKV 中存储的元数据键值（\texttt{jfs\_} 前缀已省略）：

\begin{itemize}
    \item \texttt{i\textless inode\textgreater} $\to$ \texttt{Attr}（inode 属性，序列化结构）
    \item \texttt{d\textless parent\textgreater/\textless name\textgreater} $\to$ \texttt{(child\_inode, type)}
    \item \texttt{c\textless inode\textgreater\textless chunk\_idx\textgreater} $\to$ \texttt{Slice[]}（数据分片位置）
    \item \texttt{p\textless inode\textgreater} $\to$ \texttt{parent\_inode}（硬链接场景下的父目录记录）
    \item 以及用于统计、配额、session、delete 队列等的多类辅助键
\end{itemize}

JuiceFS 的 create 通过 TiKV 客户端发起，TiKV 支持"乐观事务"和"悲观事务"两种模式。JuiceFS 默认走乐观事务（Percolator 协议），一次 create 的简化流程如下：

\begin{verbatim}
fn create(parent, name, mode, uid, gid) -> Attr:
    retry_count = 0
    loop:
        txn = tikv_client.begin_optimistic_txn()
        // 1. 读父目录 inode（客户端缓存 + TiKV Get）
        parent_attr = txn.get("i" + parent)
        if parent_attr is None: return ENOENT
        // 2. 检查重名（TiKV Get）
        if txn.get("d" + parent + "/" + name) is not None: return EEXIST
        // 3. 分配新 inode（原子计数器，独立一次 TiKV 事务）
        child_ino = allocate_inode_via_counter()  // 内部另走一次事务
        child_attr = Attr::new_file(...)
        // 4. 写入 4 类键
        txn.put("i" + child_ino, child_attr.serialize())
        txn.put("d" + parent + "/" + name, (child_ino, FILE))
        txn.put("p" + child_ino, parent)  // 若启用 ACL/硬链接
        txn.put("u" + parent, updated_parent_attr)  // 更新父目录 mtime
        // 5. Percolator 两阶段提交
        //    5a. Prewrite：每个 key 写 lock + data
        //    5b. Commit：每个 key 写 commit ts，清 lock
        //    中间 PD 分配 commit_ts
        match txn.commit():
            Ok => return child_attr
            Err(WriteConflict) if retry_count < N => retry_count += 1; continue
            Err(e) => return e
\end{verbatim}

关键点：

\begin{enumerate}
    \item \textbf{TiKV 事务是 Percolator 两阶段提交}\cite{peng2010large}：Prewrite 阶段为事务涉及的每个 key 写入 \texttt{lock} 列和 \texttt{data} 列；Commit 阶段为每个 key 写入 \texttt{commit\_ts} 并清除 lock。每个 key 的写入在 TiKV 服务端还要经过一次 Raft 日志复制（即使单节点也要落 Raft log）。因此一次 4-key 的事务在 TiKV 内部产生 8 次 Raft log 追加 + 8 次 RocksDB 写入。
    \item \textbf{获取时间戳需要访问 PD}：事务开始时要从 PD 获取 \texttt{start\_ts}，提交时再获取 \texttt{commit\_ts}。两次 PD RPC 串联在事务关键路径上。
    \item \textbf{inode 分配本身是独立事务}：TiKV 用一个全局计数器 key 管理 inode 分配，每次分配走一次独立的 CAS（compare-and-swap）事务。
    \item \textbf{MVCC 版本标记}：每次写入都带 \texttt{start\_ts}/\texttt{commit\_ts} 版本号，旧版本由 TiKV GC 后台任务回收。这在存储和读路径上都引入额外开销。
\end{enumerate}

表~\ref{tab:path-diff} 汇总一次 create 在两个系统中的操作量。

\begin{table}[htbp]

    \begin{tabular}{lcc}

NFS 的 create 路径相对简单：客户端发送 NFS4 COMPOUND 请求（通常将 OPEN4\_CREATE + GETATTR 打包成一个 RPC），服务端在 ext4 上完成 B-tree 目录项插入、inode 位图分配、journal 提交，然后返回。NFS 的瓶颈不在协议层，而在 ext4 元数据的 journal 序列化——我们在 \ref{sec:nfsd-tuning} 节会通过线程扫描实验证明这一点。

\subsection{NFS 配置的正确性验证}

NFS 作为基线对比系统，其配置直接决定对比的公平性。本节说明两项关键调参的依据：为什么必须禁用属性缓存（\texttt{noac}），以及为什么 16 线程是 8 核机器的最优 nfsd 线程数。

\subsubsection{属性缓存对基准测试的影响}
\label{sec:nfs-cache}

NFS v4 客户端默认启用属性缓存，缓存时长 3--30 秒不等（由 \texttt{acregmin}/\texttt{acregmax} 参数控制）。mdtest 的 stat 阶段对同一批文件连续执行 stat 调用，如果属性缓存生效，绝大部分请求会直接命中客户端缓存，根本不产生到服务端的网络 I/O。这种行为在生产中是有利的，但在基准测试中会严重扭曲结果——测的不再是 NFS 服务端性能，而是内核 VFS 的缓存命中速率。

为量化这一失真，本实验在单进程下对比了 \texttt{ac}（默认）和 \texttt{noac}（禁用）两种挂载方式的 stat 吞吐量：

        \toprule

\paragraph{RucksFS 对 JuiceFS+TiKV 领先 2.71--4.90 倍} 删除操作是 RucksFS 对 JuiceFS+TiKV 领先幅度最大的场景，在 $N=8$ 时达到 4.90 倍。原因是双重的：

\begin{enumerate}
    \item \textbf{TiKV 删除的重开销}：TiKV 不是物理删除 key，而是写入 MVCC tombstone（带 commit\_ts 的墓碑记录），由 GC worker 在超过 gc-life-time（默认 10 分钟）后异步回收。这意味着每次 JuiceFS unlink 在 TiKV 侧等价于一次全新的 Percolator 事务——prewrite tombstone、commit tombstone、Raft log 复制，开销与 create 基本相同。
    \item \textbf{RucksFS 的轻量 unlink}：RucksFS 的 unlink 仅在元数据 RocksDB 中执行目录项、inode、data\_location 三个 key 的原子删除（单次 WriteBatch），不触发数据文件的物理删除（利用 inode 号不复用的不变式，废数据永远不可达）。对于 mdtest 的空文件负载，unlink 退化为"纯元数据删除"，单次延迟接近极限。
\end{enumerate}

这一设计权衡值得说明：RucksFS 放弃了物理回收磁盘空间，换来了极低的 unlink 延迟。在 POSIX 语义下这是合规的——POSIX 仅要求"最后一个 fd 关闭后空间应被释放"，但对释放时机没有约束；ext4 自身也采用延迟回收（通过 \texttt{orphan list}）。不过，生产部署确实需要后台 GC 来最终回收废数据——这部分工作在本实现中尚未完成，作为未来工作列入第六章。

\paragraph{RucksFS 对 NFS 领先 1.68--2.36 倍} 相对 NFS 的优势比 create 更明显（1.5$\times$ $\to$ 2.3$\times$），原因同样是 ext4 的 unlink 要走 journal：更新目录项、回收 inode 位图、标记 inode 可释放——每一步都写 journal。而 RucksFS 只需一次 WriteBatch。

\subsection{优化效果验证}

\subsubsection{增量内联优化}
\label{sec:inline-opt}

RucksFS 使用 delta 机制处理父目录的时间戳和 nlink 更新：每次 create/unlink 操作向父目录的 delta 列表追加一条记录，读取时将 delta 折叠到基线属性上。这种设计避免了对父目录 inode 的读-改-写竞争，但每条 delta 若独立写入则是一次独立的 WAL 写入。

增量内联优化将 delta 追加操作合并到同一个 \texttt{WriteBatch} 中，与 inode 创建/删除共享一次 WAL 同步。由于 RocksDB 的 WAL 写入是整个路径中最重的 I/O 操作（一次 \texttt{fsync} 约 50--100\,$\mu$s），减少一半的 WAL 写次数直接带来了 create 和 unlink 两类写操作约 2 倍的吞吐量提升。以 $N=4$ 为例，优化前 create 吞吐量约 1,200\,ops/s，优化后达到 2,313\,ops/s。

\subsubsection{异步删除优化}
\label{sec:async-delete}

RucksFS 最初的 unlink 实现是同步的：先删除元数据，再通过 gRPC 请求 DataServer 删除对应的数据文件，两步串行完成。对于有实际数据的文件，数据删除涉及磁盘 I/O；即使对于空文件，gRPC 往返本身也消耗约 0.3\,ms。

第一轮优化将数据删除改为异步：unlink 仅删除元数据并立即返回，数据清理由客户端在后台 best-effort 触发。这一改动使得 unlink 延迟从约 1.6\,ms 降至约 0.01\,ms。第二轮"no-op 数据删除"进一步利用 inode 号不复用的不变式，直接跳过空文件的数据删除 RPC，同时修复了异步模式下的多线程竞争问题。这一轮额外带来显著提升，使得删除性能达到表~\ref{tab:remove-perf} 中的水平。

值得记录的是两个失败的优化尝试。启用 RocksDB 块缓存（block cache 设为 256\,MiB）反而导致 stat 性能下降 22\%--35\%，推测原因是测试负载的工作集较小（inode 数据量不超过几 MiB），额外的缓存查找和内存分配开销超过了缓存命中带来的收益。禁用 delta 列表的 WAL 同步也导致 stat 性能回退 39\%——这个结果出乎预料，可能与 RocksDB 内部的 WAL 和 memtable 刷写调度耦合有关。两项优化均已回滚。

\subsection{本章小结}

本章通过一组控制变量实验，对比了 RucksFS、NFS v4.2 和 JuiceFS+TiKV 三者在元数据操作上的性能表现。三个系统共享相同的硬件（8C16G）、相同的内核（5.15）、相同的数据盘（200GB SSD on ext4）、相同的网络（VPC 内 RTT $\approx$ 0.15\,ms）和相同的 mdtest 参数（$-n 5000 -F -C -T -r -u -i 3$，3 轮重复）。各系统按各自最优配置运行：NFS 使用 \texttt{noac} 挂载与扫描确认的 16 nfsd 线程，RucksFS 使用 tokio 8 线程运行时，TiKV 使用 v8.5.0 默认参数。

三项核心观察：

\begin{enumerate}
    \item \textbf{RucksFS 对 NFS 的 1.5--4.8 倍领先来自 LSM-Tree 元数据引擎}。NFS 的 ext4 后端使用 B-tree 和 journal，每次元数据修改触发 journal 提交的串行化；RucksFS 的 RocksDB 后端将元数据修改转换为顺序 WAL 追加和 memtable 插入，在写入密集的 create/remove 场景下优势明显。nfsd 线程扫描证明瓶颈不在 NFS 协议层或线程数，而在 ext4 元数据路径本身。
    \item \textbf{RucksFS 对 JuiceFS+TiKV 的 1.9--4.9 倍领先来自去 Raft 和去 MVCC 的架构选择}。两者底层都是 RocksDB，但 JuiceFS 通过 TiKV 的 Percolator 2PC + Raft + MVCC 访问元数据，一次 create 产生近 10 倍于 RucksFS 的底层工作量（见 \ref{sec:metadata-path-analysis} 节的路径分析）。RucksFS 作为专用元数据服务器，用 PCC + 单次 WriteBatch 绕开了通用 KV 存储的一致性和版本管理代价。
    \item \textbf{gRPC 仍是 RucksFS 内部最大的瓶颈}。分层开销实验显示，去掉 gRPC（进程内 embedded 模式）时 RocksDB 可以达到 172,814\,ops/s——现有 RucksFS 分布式模式的 2,313\,ops/s 只发挥了底层引擎 1.3\% 的能力。进一步压缩 RPC 次数（例如把 open-create-release 合并为一次 RPC）是未来性能优化的主线。
\end{enumerate}

本章的数据足以支持一个核心结论：在元数据操作密集的工作负载下，\textbf{使用专用元数据服务器直接访问 LSM-Tree 键值存储}（RucksFS 的架构选择）优于传统内核文件系统（NFS+ext4），也优于基于通用分布式键值存储的分层文件系统（JuiceFS+TiKV）。专用服务器舍弃了通用 KV 存储的 Raft/MVCC 能力，换取了两个数量级的写路径简化；LSM-Tree 舍弃了就地更新，换取了顺序 I/O 友好的写放大特性。两项选择叠加，使得 RucksFS 在用户态 FUSE + 单节点部署的约束下仍能稳定领先主流参照系统。
