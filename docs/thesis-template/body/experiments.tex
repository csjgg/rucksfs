\section{测试与结果分析}
\label{sec:experiments}

第~\ref{sec:design}~章已经说明了 RucksFS 在设计上的元数据组织方案、事务模型和增量更新机制。本章则需要把这些设计放到真实分布式部署下，检验其是否达到预期。实验分为两条线。

一条线是 POSIX 语义合规性。使用 pjdfstest 套件在分布式部署环境下（客户端和服务端分别位于不同节点）运行 398 个用例，覆盖 13 类系统调用，检验 RucksFS 在 FUSE + RocksDB 这套组合之上是否能够正确表达 POSIX 接口的行为。

服务器选用 64 核大规格机型，目的是使所有并发档位下被测系统的元数据后端都不会在测试本身之前先触顶 CPU。客户端选用 2\,vCPU、2\,GiB 内存的最小规格——mdtest 单 rank 在 FUSE 路径上的有效吞吐受内核 VFS 串行化限制，大致在 1\,000\,ops/s 量级，2 核即可用满该吞吐上限；提高规格并不改变结果，只会增加成本，还会占用配额、阻碍继续扩展客户端数量。所有客户端硬件规格保持一致，以排除横向不可比因素。

\subsubsection{被测系统}

本章设置了三套被测系统，加上 RucksFS 自身的两种构建变体；每次只运行一套，切换时完全清场。

\begin{itemize}
    \item \textbf{RucksFS-Delta}：默认构建，启用第~\ref{sec:design}~章第~\ref{sec:txn_model}~节描述的 DeltaOp 增量追加机制。父目录的 \texttt{mtime}、\texttt{ctime}、\texttt{nlink} 更新走增量路径。
    \item \textbf{RucksFS-NoDelta}：编译期通过 \texttt{cargo} 特性开关关闭 DeltaOp。父目录属性更新退化为对父 inode 基值的读--改--写，其余代码与 Delta 变体完全相同。该变体仅用于对照，并非生产构建。
    \item \textbf{NFS}：Linux 内核 NFS v4.2 服务端（\texttt{nfs-kernel-server}），导出本地 ext4 目录。客户端以 \texttt{vers=4.2,noac} 挂载，关闭属性缓存，保证每次 \texttt{stat} 都到达服务端。\texttt{nfsd} 线程数为 128。
    \item \textbf{JuiceFS+TiKV}：JuiceFS 1.3.1，元数据后端是单节点 TiKV 8.5.6（含 PD），数据存储指向同机本地目录。这一组合对应业界常见的“通用分布式 KV + 分层文件系统”路线。JuiceFS 与 RucksFS 共用 RocksDB 作为底层存储引擎（TiKV 内部同样使用 RocksDB），便于隔离上层架构差异。
\end{itemize}

服务器在切换被测系统时执行严格的清场流程：停止所有守护进程，删除数据目录，确认监听端口释放后再启动下一套。客户端侧同样卸载挂载点、清理 FUSE 残留连接并重建挂载目录。整个流程由一个统一的 orchestrator 脚本驱动，确保每次测试都从干净状态开始。

\subsubsection{评估工具与执行流程}

POSIX 合规性这一线使用 pjdfstest（Rust 实现版），共 398 个用例，覆盖 \texttt{chmod}、\texttt{chown}、\texttt{link}、\texttt{mkdir}、\texttt{mkfifo}、\texttt{mknod}、\texttt{open}、\texttt{rename}、\texttt{rmdir}、\texttt{symlink}、\texttt{truncate}、\texttt{unlink}、\texttt{ftruncate} 等系统调用。

性能测试这一线使用 mdtest 4.1.0（IOR 套件的子项），以 OpenMPI 4.1.2 跨客户端调度 MPI 进程。每台客户端挂载一次目标文件系统，并在挂载点上启动一个 mdtest rank，因此并发度 $N$ 等于客户端数量。实验按 $N=2,8,32,64,96$ 五档运行；每档分别在共享父目录（hard 模式，所有 rank 写入同一目录）和独立父目录（easy 模式，每个 rank 写入各自的子目录）下各运行一次完整的 \texttt{create-stat-remove} 流程。为了避免大 $N$ 情形下单次测试耗时过长，每 rank 的文件数随 $N$ 递减，具体取值见表~\ref{tab:files-per-rank}。

mdtest 的核心参数是 \texttt{-F -C -T -r -i 1}：\texttt{-F} 仅测试文件操作，\texttt{-C}/\texttt{-T}/\texttt{-r} 分别对应创建、查询、删除三个阶段，\texttt{-i 1} 表示不进行内部重复。hard 和 easy 的区别由 \texttt{-u} 参数控制：默认所有 rank 在同一父目录下操作（hard），加上 \texttt{-u} 后各 rank 在独立子目录下操作（easy）。本章默认讨论的都是 hard 模式的数据，easy 模式数据用于对照。

每次 mdtest 调用前，服务端和所有客户端都会同步执行 \texttt{echo 3 > /proc/sys/vm/drop\_caches}，以清除上一轮遗留的页缓存。

\subsubsection{测试范围的几点说明}

在查看具体数据之前，有几点需要先予以说明。本章的测试主要关注元数据路径的性能，并非对完整文件系统能力的全面评估，因此下列条件需要先明确，以免读者在解读数据时产生误解。

\textbf{mdtest 使用空文件负载。} 启动参数为 \texttt{-w 0}，每个文件在 \texttt{create} 之后直接 \texttt{close}，不写入任何数据。这样做是为了将元数据路径与数据路径分离，只考察元数据的表现。代价是 DataServer 基本不参与测试——它仅在 \texttt{create} 时分配一个空的 \texttt{DataLocation}，在 \texttt{unlink} 时接到一次“可以回收”的通知。真实混合读写负载下 \texttt{read}/\texttt{write} 的表现并不在本章关注范围之内。

\textbf{DataServer 目前是最小实现。} 如第~\ref{sec:dataserver}~节所述，\texttt{RawDiskDataStore} 以固定偏移公式直接映射 inode，没有块映射，也没有空间回收，\texttt{delete\_data} 目前仍是空操作。因此 \texttt{unlink} 的吞吐中有一部分来自“当前未执行真正的数据回收”这一事实，存在一定虚高，具体到 §\ref{sec:perf-comparison}~节讨论 \texttt{remove} 时会再展开。

\textbf{未提供副本容错。} RucksFS 目前的部署是单 MetadataServer 加单默认 DataServer，没有副本，也没有 Raft。JuiceFS+TiKV 即使部署为单节点，TiKV 内部仍然运行 Raft 和 Percolator 两阶段提交，这些开销原本是为多副本和跨节点事务准备的。因此本章看到的 RucksFS 对 JuiceFS+TiKV 的性能优势，一部分源于省去了这些提交层开销，不能据此推出 RucksFS 作为一个完整的分布式系统在各方面均已超过 JuiceFS+TiKV。

在明确上述前提之后，下文的数据主要用于回答两个问题：其一，RucksFS 在元数据路径上相对 NFS 与 JuiceFS+TiKV 的横向位置；其二，DeltaOp 这一具体机制在高并发下究竟发挥了多大作用。

\subsection{POSIX 合规性评估}
\label{sec:pjdfstest}

POSIX 合规性这一线的测试在分布式部署下进行——客户端和 MetadataServer 位于不同节点，请求需要经过 gRPC 传输。一个用例能够通过，前提不仅是 MetadataServer 内部的事务逻辑正确，还要求错误码在穿过 gRPC 和 FUSE 两层之后依然保持不变。

\subsubsection{pjdfstest 简介}

pjdfstest 是业界常用的 POSIX 文件系统合规性测试套件，覆盖 \texttt{chmod}、\texttt{chown}、\texttt{link}、\texttt{mkdir}、\texttt{mkfifo}、\texttt{mknod}、\texttt{open}、\texttt{rename}、\texttt{rmdir}、\texttt{symlink}、\texttt{truncate}、\texttt{unlink}、\texttt{ftruncate} 等 13 类系统调用。套件中每条 POSIX 规则都会按文件类型（regular、dir、symlink、fifo、socket、char、block）展开成多条用例，目的是系统地验证同一条规则在各种对象上的行为一致性。本文采用 pjdfstest 的 Rust 实现版本，总共 398 条用例。

这种展开方式对被测系统较为严苛：同一条 POSIX 规则在不同文件类型上共享同一代码分支，但只要其中任意一种文件类型不被支持，整组用例都会被记为“失败”——即便被测系统对该规则的核心语义本身是正确的。因此直接使用总通过率衡量语义完整度并不合适，需要先划定“本文实现覆盖范围内”的用例集，再讨论结果。

\subsubsection{实现范围与理论排除项}

在给出测试数据之前，先根据第~\ref{sec:design}~章的设计边界把 398 条用例分为“在实现范围内”和“在实现范围外”两类。

\textbf{排除项 A：\texttt{mknod} 与 \texttt{mkfifo} 相关用例。} 这两个接口管理的是 Unix 域 socket、命名管道、字符设备节点和块设备节点。它们的运行语义与内核设备号、内核管道缓冲区紧密绑定，与本文“通过键值对持久化命名空间”的研究主题正交；加之本文聚焦的负载（容器镜像、训练数据、日志、构建产物）基本不会涉及这类对象，实现这两个接口对论证元数据机制本身帮助不大。因此本文明确选择不实现这两个接口，RucksFS 对二者统一返回 \texttt{EOPNOTSUPP}（POSIX 规定的“操作不支持”错误码，用户程序收到该错误后会明确得到“文件系统不支持该操作”的结果，而不是被伪装为成功）。凡是直接调用 \texttt{mknod}/\texttt{mkfifo} 的用例，以及\emph{需要先用它们准备测试对象}的派生用例，都属于这一类。举一个具体例子：pjdfstest 中“当 \texttt{rename} 的目标路径前缀不是目录时应返回 \texttt{ENOTDIR}”这条规则会按 7 种文件类型各生成一条用例，其中后 4 种需要先用 \texttt{mknod}/\texttt{mkfifo} 准备 fifo、socket 或设备节点；用例还未进入 \texttt{rename} 本身的语义检查即因 \texttt{EOPNOTSUPP} 而中断——而 RucksFS 在常规文件、目录、符号链接上对同一条规则都能给出符合规则的响应。

\textbf{排除项 B：DataServer 单文件大小上限。} 作为最小实现，当前 \texttt{RawDiskDataStore} 使用固定偏移公式映射 inode，单文件逻辑上限为 64\,MiB。\texttt{open::interact\_2gb} 这条用例直接位于该上限之上，属于 DataServer 的开发期参数，与元数据设计所要回答的问题无关。

\textbf{排除项 C：与元数据设计无关的跳过项。} pjdfstest 框架本身会跳过 48 条用例：27 条来自 \texttt{utimensat} 系列（需要单独的 feature flag，当前未启用），21 条涉及 \texttt{remount}（FUSE 配置不允许运行时重新挂载）和跨文件系统操作（测试环境未配置第二个文件系统）。这些项与 RucksFS 的元数据设计无关。

\subsubsection{与第~\ref{sec:design}~章设计决策的对照}
\label{sec:pjdfstest-semantic}

仅凭“实现范围内 182 条全部通过”这一汇总数据，尚不足以说明 RucksFS 确实将第~\ref{sec:design}~章的若干设计决策落到了实处。为避免“因为未实现所以不会失败”这样的误读，下面挑选几条较易被忽视的边界语义进行说明。这些用例都位于实现范围内、都在通过列表之中，且每一条都可以直接对应到第~\ref{sec:design}~章的某个具体设计决策。

\begin{itemize}
    \item \textbf{\texttt{unlink::open\_file\_not\_freed}}（对应：删除语义与句柄生命周期，第~\ref{sec:txn_model}~节）。文件被 \texttt{unlink} 之后，若仍有打开句柄，应当能够继续读写，物理回收需推迟到最后一个句柄关闭。这要求 MetadataServer 同时维护 \texttt{nlink} 和打开计数：当 \texttt{nlink} 归零但计数仍大于零时，对象进入 \texttt{pending\_deletes}，由 \texttt{release} 触发最终清理。若事务未将“删除目录项”和“维持 inode 可达性”分开处理，该用例将不通过。
    \item \textbf{\texttt{rename::to\_multiply\_linked::regular}}（对应：rename 覆盖逻辑，算法~\ref{alg:rename}）。\texttt{rename} 的目标若是一个带有多条硬链接的文件，覆盖动作只能将目标的 \texttt{nlink} 减一，而不能直接删除目标 inode。这一点对应第~\ref{sec:design}~章 \texttt{rename} 流程中“按覆盖删除逻辑处理目标对象”一步。
    \item \textbf{12 条 \texttt{*::enametoolong\_component} 用例}（对应：错误码穿透）。文件名长度超过 \texttt{NAME\_MAX} 时应返回 \texttt{ENAMETOOLONG}。该约束在 MetadataServer 所有接受 \texttt{name} 字段的 RPC 入口统一校验，错误码需经 gRPC 和 FUSE 两层传递后保持不变。
    \item \textbf{\texttt{open::open\_trunc}}（对应：VfsCore 协调路径，第~\ref{sec:vfscore}~节）。以 \texttt{O\_TRUNC} 打开已有文件时，\texttt{size} 需在打开操作中同步清零，同时刷新 \texttt{mtime} 和 \texttt{ctime}。FUSE 并不一定会再发一次独立的 \texttt{setattr}，因此这些工作必须在 \texttt{open} 实现中一并完成。
    \item \textbf{\texttt{rmdir::eexist\_enotempty\_non\_empty\_dir} 系列}（对应：事务内空目录检查，第~\ref{sec:posix_impl}~节）。删除非空目录应返回 \texttt{ENOTEMPTY}。第~\ref{sec:design}~章已经说明，\texttt{rmdir} 将空目录检查也纳入事务中，在一致性视图下执行前缀扫描，避免检查结果在进入事务前失效。
\end{itemize}

这些用例覆盖的并非孤立的接口行为，而是第~\ref{sec:design}~章的跨列族原子提交、句柄生命周期、统一加锁顺序以及客户端协调路径在实际运行时的综合表现。它们能够全部通过，说明这套设计在分布式部署条件下达到了预期的语义效果。

\subsubsection{小结}

综上所述：在明确排除 \texttt{mknod}/\texttt{mkfifo} 相关用例、DataServer 单文件大小用例以及 pjdfstest 框架自身的跳过项之后，剩下 182 条“实现范围内”用例全部通过，通过率 100\%。排除项中的 167 条失败由 pjdfstest 按文件类型展开的机制所放大，并不构成对元数据正确性的实际反例；\texttt{mknod}/\texttt{mkfifo} 的不实现是面向典型分布式存储负载所做的显式取舍，在返回值上亦被标记为 \texttt{EOPNOTSUPP}，而非伪装为“通过”。结合前述几条边界用例的对照，可以认为 RucksFS 的元数据实现在本课题要求的范围内达到了预期的语义正确性目标。这一结果同时也是后续性能数据的前提——后面的吞吐数据都建立在“实现范围内语义正确”的基础之上。

\textbf{与 JuiceFS+TiKV 的对比。} RucksFS 始终高于 JuiceFS+TiKV，三档下的比值分别为 2.8、3.5、4.0 倍，较为稳定。差距主要来自两方面。其一，RucksFS 采用单机直连 RocksDB，一次 \texttt{create} 在 MetadataServer 内部即一次 \texttt{AtomicWriteBatch} 提交，多类键共享一次 WAL 同步即可写入；而 TiKV 需要走完整的 Percolator 两阶段提交，再经过一遍 Raft 日志，即便单节点部署、没有跨节点复制，Raft 这一层也照样执行。这部分提交层开销正是 RucksFS 所省去的。其二，DeltaOp 机制将父目录属性更新从“读--改--写”改为对独立列族的追加写，使共享父目录下的 \texttt{create} 事务不再需要竞争父 inode 键的写锁，并发度提升后吞吐仍能保持，不会急剧下降。JuiceFS 与 TiKV 没有类似机制，共享父目录更新走的是常规的 MVCC 读写路径。

这一对比说明两件事。其一，RucksFS 基于 RocksDB 实现元数据能够达到业界 KV 文件系统的水平——JuiceFS+TiKV 是标准的“通用分布式 KV + FUSE 文件系统”方案，与其在同一硬件下比较吞吐是有意义的。其二，单机直连加针对共享热点目录的并发优化确实能够带来明显的性能提升，这正是本文的主要设计点。

\textbf{与 NFS 的对比。} RucksFS 对 NFS 的比值从 $N=2$ 的 3.1 倍升至 $N=32$ 的 38 倍，差距随 $N$ 增大而不断拉开。NFS 扩展能力受限的原因主要在本地的 ext4——所有 \texttt{nfsd} 线程在共享父目录上的 \texttt{create} 都需要先获取目录 inode 的 \texttt{i\_rwsem} 写锁，再走 journal 提交，这把锁将并发限制住了。$N=2$ 处的 3 倍差距尚未被锁竞争放大，更能反映两种方案在元数据写入路径上的差别：在 ext4 上一次文件创建需要动 6 个磁盘区域（见图~\ref{fig:ext4-create-path}），而在 KV 方案中只是对若干条键值对的追加。$N=32$ 的 38 倍差距则主要是共享目录锁被放大的结果。整体来看，使用 KV 来保存文件系统元数据相较于传统本地文件系统的 B+ 树方案，在高并发共享目录场景下更具优势。

\subsubsection{查询吞吐}

表~\ref{tab:perf-stat-hard} 给出 \texttt{stat} 的吞吐数据。$N \in \{2, 8, 32\}$ 三档下，RucksFS-Delta 分别达到 7\,467、37\,041、357\,020\,ops/s，\texttt{stat} 读路径在本文实现中运行速度较高。同样三档下 JuiceFS+TiKV 为 3\,749、14\,485、52\,661\,ops/s，RucksFS 约为其 2.0、2.6、6.8 倍；NFS 为 2\,100、5\,934、20\,021\,ops/s，RucksFS 约为其 3.6、6.2、17.8 倍。

这里需要就 NFS 的挂载参数作一点说明。NFS 客户端默认启用属性缓存，命中缓存的 \texttt{stat} 无需走网络，此时所比较的就不再是服务端的真实处理能力。而 JuiceFS 的 FUSE 客户端本身没有这一层属性缓存，每次 \texttt{stat} 都要回到服务端。为了使三套系统在同一条件下比较元数据本身的处理能力，本文对 NFS 客户端也以 \texttt{noac} 挂载，关闭属性缓存，强制每次 \texttt{stat} 都走网络。

RucksFS 的 \texttt{stat} 吞吐较高，原因有两方面。其一，MetadataServer 端的 \texttt{stat} 就是一次 RocksDB 点查，上一轮 \texttt{create} 刚写入的 inode 记录仍在 BlockCache 中，命中后延迟可压至微秒级。其二，LSM-tree 的读与写本身并不共用一把锁，\texttt{stat} 不会被并发的写路径挤占；再加上 DeltaOp 将父目录属性写从关键路径上移除，\texttt{stat} 的 \texttt{lookup} 阶段读取父 inode 时也不会被父 inode 的写锁阻塞。这一点在下一节 Delta 与 NoDelta 的对照中可以得到明显印证。

\subsubsection{删除吞吐}

\texttt{remove} 的数据见表~\ref{tab:perf-remove-hard}。$N \in \{2, 8, 32\}$ 三档下 RucksFS-Delta 的吞吐分别为 1\,674、6\,879、22\,958\,ops/s，相对 JuiceFS+TiKV 约为 3.0、5.1、3.9 倍，相对 NFS 约为 2.7、4.3、9.1 倍，形态与 create 相近。

这里有一点需要先说明：当前 RucksFS 的 DataServer 仍是最小实现，\texttt{delete\_data} 目前是空操作，并未进行真正的物理空间回收。相比之下，NFS 的 \texttt{unlink} 会触发 ext4 的 inode 位图翻转与 journal 提交，TiKV 的 \texttt{unlink} 需要写一条 tombstone 并最终参与后台 GC。mdtest 使用的又是空文件负载，两侧本就没有实际数据需要回收，这部分工作量差异在本文的测试中会被放大。因此 \texttt{remove} 的倍数中有一部分是“尚未进行数据回收”所带来的虚高，并非全部源于设计上的收益。真实的数据回收补齐后，这一倍数会相应回落。

另一方面，元数据侧的 \texttt{remove} 路径在 RucksFS 中已完整实现。一次 \texttt{unlink} 在 MetadataServer 中同样走 \texttt{AtomicWriteBatch} 原子提交，跨键更新按 inode 编号排序加锁，与 \texttt{create} 对称；父目录属性的更新走 DeltaOp 追加，共享父目录下的并发 \texttt{unlink} 也不会在父 inode 键上互相抢锁。因此即便扣除前述虚高部分，元数据侧的 \texttt{remove} 路径仍能保持较高的并发度——下一节 Delta 与 NoDelta 对照中 \texttt{remove} 的分叉形态也可以印证这一点。

\subsubsection{本节小结}

本节的数据主要说明两件事。其一，本文基于 RocksDB 实现元数据，在性能上能够达到业界常见 KV 文件系统的水平——与 JuiceFS+TiKV 相比，\texttt{create}/\texttt{stat}/\texttt{remove} 三类操作均能稳定领先 2--5 倍；分项来看，一部分差距来自单机直连省去了 Raft 一层提交开销，另一部分来自 DeltaOp 对共享父目录并发的优化。其二，相对传统本地文件系统，KV 这条路线在元数据密集场景下更为合适——$N=2$ 时已对 NFS 领先约 3 倍，$N$ 继续增大后，NFS 被 ext4 的目录锁限制住，而 RucksFS 仍在接近线性增长。

$N \geq 64$ 时两套对照系统都难以稳定出数，而这恰好是 DeltaOp 机制发挥作用的场景。下一节通过 Delta 与 NoDelta 两种仅有一个编译开关差异的构建变体，在相同硬件、相同负载下进行对照实验，以说明“高并发共享目录下 DeltaOp 究竟发挥了多大作用”。

\subsection{DeltaOp 增量机制的内部对比}
\label{sec:delta-vs-nodelta}

上一节主要讨论 $N \leq 32$ 的数据，原因是 $N \geq 64$ 时 JuiceFS+TiKV 和 NFS 均无法稳定运行。但高并发共享父目录恰好是当初设计 DeltaOp 机制所要针对的场景，因此本节专门考察 DeltaOp 在高并发下的实际作用。

做法是：保留主实现不动，仅通过一个 \texttt{cargo} 特性开关关闭 DeltaOp，得到一个名为 NoDelta 的构建。它与 Delta 的区别仅在父目录属性更新这一处——NoDelta 退回传统的“读父 inode、修改、写回”方式，其余代码完全一致。二者在同一台服务器、同一批客户端、同一份负载下分别运行，直接比较。第~\ref{sec:design}~章第~\ref{sec:txn_model}~节当时的设想是：将父目录属性更新从“读--改--写”改为向一个独立列族追加 DeltaOp，就可以避开共享父目录下的事务冲突，使 \texttt{create} 在并发度上升时仍能继续线性增长。若该设想成立，应当可以观察到三点：（1）低并发时两者差异不大；（2）高并发共享父目录下 NoDelta 出现下降，Delta 仍保持线性；（3）若换用独立父目录（easy 模式）、不存在写热点，两者应当重合。下面依次检验这三点。

\subsubsection{NoDelta 为何会下降}

NoDelta 从 $N=32$ 起吞吐急剧下降，这一现象的原因可以从 MetadataServer 端的事务执行过程中看出。

在 NoDelta 下，一次 \texttt{create} 事务内部需要完成的工作是：对父目录 inode 键调用一次 \texttt{get\_for\_update} 获取写锁，读取当前父 inode 的值，修改 \texttt{mtime}/\texttt{ctime}，再 \texttt{put} 回去，连同新建的目录项和 inode 一并提交。共享父目录下，所有客户端的 \texttt{create} 事务都在争抢同一条父 inode 键的写锁。如第~\ref{sec:design}~章所述，本文采用的是悲观并发控制：未取得锁的事务会等待，等待超时后回滚，由服务端再次重试。并发度提升后，这个“等待—回滚—重试”的循环会自我放大——被拒绝的事务需要先释放已经取得的其他锁、回滚已经写入的 batch，之后才能重新进入锁竞争；而此时队列中又堆积了更多重试请求。此时吞吐量就会急剧下降。

启用 DeltaOp 后，路径发生了明显变化。\texttt{create} 不再改动父 inode 的值，父目录的属性变化被编码为一条 DeltaOp，写入 \texttt{delta\_entries} 列族。DeltaOp 的键中带有一个单调递增的序列号 \texttt{seq}，因此每次写入的目标键互不相同，根本不会与其他事务在同一个键上争锁。父 inode 的值只在后台的 \texttt{DeltaCompactionWorker} 折叠增量时才被写一次，该动作运行在测试关键路径之外，不会影响 create 本身。锁竞争消除的同时，热点目录上的随机写也转为了顺序追加。

将上述原因对应到数据上：NoDelta 在 $N=32$ 时尚未出现吞吐下降，说明此时锁等待还未达到引发重试堆积的程度；$N=64$ 时吞吐下降 40\%，对应重试请求开始累积；$N=96$ 继续下降，对应重试队列接近饱和。Delta 这边不会进入这一循环，因此吞吐始终保持线性。

\subsubsection{对 remove 和 stat 的影响}

DeltaOp 的设计目标虽然是父目录属性更新，但其影响并不仅限于 \texttt{create}。表~\ref{tab:delta-impact-other} 给出 \texttt{remove} 和 \texttt{stat} 在 $N=64$ 与 $N=96$ 下两种构建的对比。

\textbf{remove} 的形态与 \texttt{create} 相似。\texttt{unlink} 同样需要更新父目录的 \texttt{mtime}/\texttt{ctime}，走相同的事务流程。$N=96$ 时比值达到 6.12，略高于 create 的 5.81，原因可能是 \texttt{unlink} 路径上 NoDelta 还需多改一次目标 inode 的 \texttt{nlink}，持锁时间更长。

\textbf{stat} 一侧的差距看起来最大——$N=96$ 时达到 136 倍。\texttt{stat} 本身是只读操作，理论上不应被 DeltaOp 这类写路径优化直接影响。但在 NoDelta 下，并发的 \texttt{create} 持续抢占父 inode 键的写锁，而 \texttt{stat} 在 \texttt{lookup} 阶段需要读取父目录的状态。RocksDB 的悲观事务规定读路径在写锁持有期间必须等待，因此父 inode 的写锁也会使 \texttt{stat} 排队。换用 Delta 之后，父 inode 的写从热路径上移除，\texttt{stat} 不再排队，吞吐重新由 RocksDB 自身的点查能力决定。由此可见，DeltaOp 不仅减少了写路径上的冲突，也间接降低了读路径上的等待。这一点在第~\ref{sec:design}~章中未特别预测，但与整体设计思路一致，可视为一项附加收益。

\subsubsection{回到第~\ref{sec:design}~章的设计}

将上述几项结果与第~\ref{sec:design}~章关于 DeltaOp 的几条说法对照，整体上是一致的：

\begin{itemize}
    \item \textbf{“DeltaOp 在低并发下不引入额外开销”}（第~\ref{sec:txn_model}~节）。$N \leq 32$ 时 Delta 与 NoDelta 的吞吐差距均在 2\% 以内。增量追加路径只在存在写冲突时才能省下开销，没有冲突时也不会额外增加负担。
    \item \textbf{“DeltaOp 在高并发下消除父目录事务冲突”}（第~\ref{sec:txn_model}~节）。$N=64$ 起 NoDelta 出现明显的吞吐下降，而 Delta 继续线性增长。两条曲线的分叉点正好对应 PCC 锁等待开始累积的位置。
    \item \textbf{“DeltaOp 的优势只在共享父目录场景下出现”}。easy 模式下两者吞吐完全重合，仅 hard 模式下存在差距。这也说明 DeltaOp 的收益并非来自其他无关因素。
    \item \textbf{读路径也间接受益}。\texttt{stat} 在 NoDelta 下被父 inode 的写锁阻塞，在 Delta 下不存在此问题。说明 DeltaOp 的作用不仅体现在写入路径上。

\end{itemize}

上述四点合起来基本可以说明，第~\ref{sec:design}~章所述的 DeltaOp 机制在分布式部署下确实发挥了作用，差异出现的方向、幅度以及与并发度的关系都与当初的设想吻合。

\subsection{本章小结}

本章的实验沿两条线展开。

第一条线是 POSIX 合规性。将 pjdfstest 的 398 条用例按本文实现范围分解：扣除 \texttt{mknod}/\texttt{mkfifo} 相关用例（167 条）、DataServer 单文件大小用例（1 条）和 pjdfstest 框架自身跳过的用例（48 条）之后，剩下 182 条“实现范围内”用例全部通过，通过率 100\%。“打开后 \texttt{unlink}”、硬链接覆盖、长名称错误码、\texttt{O\_TRUNC} 时间戳、事务内空目录检查等边界用例同样位于通过列表中，正好对应第~\ref{sec:design}~章的跨列族原子提交、句柄生命周期维护、统一加锁顺序以及客户端协调路径。\texttt{mknod}/\texttt{mkfifo} 的不实现是面向典型分布式存储负载所作的显式取舍，接口统一返回 \texttt{EOPNOTSUPP}，不以错误来遮盖真实问题。

第二条线是分布式性能。主体数据来自 $N \in \{2, 8, 32\}$ 三档下对 NFS 和 JuiceFS+TiKV 的横向对比。与 JuiceFS+TiKV 相比，RucksFS 在 \texttt{create}/\texttt{stat}/\texttt{remove} 上稳定领先 2--5 倍，一部分原因是本文采用单机直连 RocksDB，省去了 TiKV 的 Raft 和两阶段提交开销；另一部分原因是 DeltaOp 对共享父目录并发的优化——这说明本文基于 RocksDB 实现的元数据方案达到了业界常见 KV 文件系统的水平。与 NFS 相比，$N=2$ 时已领先 3 倍，$N=32$ 时升至 38 倍；NFS 在共享父目录下被 ext4 的 \texttt{i\_rwsem} 锁限制住，而 RucksFS 仍在接近线性增长——这说明使用 KV 来保存元数据相对传统本地文件系统的 B+ 树方案更具优势。$N \geq 64$ 时 NFS 和 JuiceFS+TiKV 均无法稳定出数，横向对比意义有限，数据仅在表中列出。

在高并发区间，RucksFS 自身的 Delta 与 NoDelta 对比提供了进一步的证据。两种构建仅差一个编译开关，其余完全一致。结果是 hard 模式从 $N=64$ 起二者明显分开：$N=96$ 时 \texttt{create} 比值 5.81、\texttt{remove} 比值 6.12、\texttt{stat} 比值 136；而 easy 模式下两者完全重合。这组对比直接说明 DeltaOp 在高并发共享父目录场景下确实消除了当初希望消除的父 inode 写冲突，并间接将 \texttt{stat} 从写锁的等待中解放出来——这是本章最直接地将实验结果对应到具体设计决策的一组结果。
