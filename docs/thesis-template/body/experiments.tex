\section{测试与结果分析}
\label{sec:experiments}

第~\ref{sec:design}~章讲了 RucksFS 在设计上的元数据组织方案、事务模型和增量更新机制。这一章要做的就是把这些设计放到真实分布式部署下看看能不能站得住。实验分两条线。

一条线是 POSIX 语义合规性。用 pjdfstest 套件在分布式部署环境下（客户端和服务器分别在不同节点上）跑 398 个用例，覆盖 13 类系统调用，看看 RucksFS 在 FUSE + RocksDB 这套组合之上能不能把 POSIX 接口的行为表达对。

另一条线是分布式场景下的元数据性能。对外这一层，把 RucksFS 和 NFS v4.2、JuiceFS+TiKV 放到同一套硬件上一起压，看横向位置；对内这一层，利用编译期特性开关做出 Delta 和 NoDelta 两种变体，在严格的控制变量条件下直接量化增量更新带来的差别。

\subsubsection{被测系统}

本章设置了三套被测系统，加上 RucksFS 自己的两种构建变体，每次只跑一套，切换时完全清场。

\begin{itemize}
    \item \textbf{RucksFS-Delta}：默认构建，启用第~\ref{sec:design}~章第~\ref{sec:txn_model}~节描述的 DeltaOp 增量追加机制。父目录的 \texttt{mtime}、\texttt{ctime}、\texttt{nlink} 更新走增量路径。
    \item \textbf{RucksFS-NoDelta}：编译期通过 \texttt{cargo} 特性开关关闭 DeltaOp。父目录属性更新退化回对父 inode 基值的读--改--写，其余代码与 Delta 变体完全相同。这个变体只做消融对照用，不是生产构建。
    \item \textbf{NFS}：Linux 内核 NFS v4.2 服务端（\texttt{nfs-kernel-server}），导出本地 ext4 目录。客户端用 \texttt{vers=4.2,noac} 挂载，关闭属性缓存，保证每次 \texttt{stat} 都到达服务端。\texttt{nfsd} 线程数为 128。
    \item \textbf{JuiceFS+TiKV}：JuiceFS 1.3.1，元数据后端是单节点 TiKV 8.5.6（含 PD），数据存储指向同机本地目录。这一组合对应业界常见的"通用分布式 KV + 分层文件系统"路线。JuiceFS 和 RucksFS 共用 RocksDB 作为底层存储引擎（TiKV 内部就是用 RocksDB），方便隔离上层架构差异。
\end{itemize}

服务器在切换被测系统时执行严格的清场流程：停止所有守护进程，删除数据目录，确认监听端口释放，再启动下一套。客户端侧同样要卸载挂载点、清理 FUSE 残留连接并重建挂载目录。整个流程由一个统一的 orchestrator 脚本驱动，确保每次测试都从干净状态开始。

\subsubsection{评估工具与执行流程}

POSIX 合规性这一线用 pjdfstest（Rust 实现版），共 398 个用例，覆盖 \texttt{chmod}、\texttt{chown}、\texttt{link}、\texttt{mkdir}、\texttt{mkfifo}、\texttt{mknod}、\texttt{open}、\texttt{rename}、\texttt{rmdir}、\texttt{symlink}、\texttt{truncate}、\texttt{unlink}、\texttt{ftruncate} 等系统调用。

性能测试这一线用 mdtest 4.1.0（IOR 套件的子项），以 OpenMPI 4.1.2 跨客户端调度 mpi 进程。每台客户端挂载一次目标文件系统，在挂载点上启动一个 mdtest rank，因此并发度 $N$ 等于客户端数量。按 $N=2,8,32,64,96$ 五档运行；每档分别在共享父目录（hard 模式，所有 rank 写同一目录）和独立父目录（easy 模式，每 rank 写自己的子目录）下各跑一次完整的 \texttt{create-stat-remove} 流程。为了防止大 $N$ 时单次测试拖太久，每 rank 的文件数随 $N$ 递减，具体数值见表~\ref{tab:files-per-rank}。

mdtest 的核心参数是 \texttt{-F -C -T -r -i 1}：\texttt{-F} 只测文件操作，\texttt{-C}/\texttt{-T}/\texttt{-r} 分别跑创建、查询、删除三个阶段，\texttt{-i 1} 不做内部重复。hard 和 easy 的区别由 \texttt{-u} 控制：默认所有 rank 在同一父目录下操作（hard），加 \texttt{-u} 后各 rank 在独立子目录下操作（easy）。本章默认讨论的都是 hard 模式数据，easy 模式数据用来做对照。

每次 mdtest 调用前，服务端和所有客户端都会同步执行 \texttt{echo 3 > /proc/sys/vm/drop\_caches}，清除上一轮遗留的页缓存。

\subsubsection{实验的有效范围与已知偏置}

在给出具体数字之前，需要先说明本章测试能说明什么、不能说明什么。本文测试的是 \emph{元数据路径} 的相对差异，而不是完整文件系统的全面能力。下列几项条件必须在解读后续数字时保持清醒：

\textbf{负载是元数据密集型空文件测试。} mdtest 用 \texttt{-w 0} 启动，每个文件只 \texttt{create} 之后立即 \texttt{close}，不写入任何数据字节。这样做的动机是把元数据路径和数据路径隔离开来，但也意味着 DataServer 几乎没有被测试到——它唯一参与的是分配空 \texttt{DataLocation} 以及在 \texttt{unlink} 时被通知一次"可回收"。实际混合读写工作负载下，\texttt{read}/\texttt{write} 的吞吐和延迟并不在本章的观察范围内。

\textbf{RucksFS 的 DataServer 是最小实现。} 如第~\ref{sec:data_path}~节所述，\texttt{RawDiskDataStore} 用固定公式 \texttt{inode $\times$ max\_file\_size + file\_offset} 计算物理偏移，不维护块映射，也不做空间回收，\texttt{delete\_data} 当前是空操作。因此 \texttt{unlink} 吞吐数据里有一部分收益来自"根本没做真实删除工作"：NFS 的 \texttt{unlink} 会触发 ext4 的 inode 位图翻转和 journal 提交，TiKV 的 \texttt{unlink} 会写入 tombstone 并最终参与 GC，而 RucksFS 在空文件场景下只动元数据侧。读者在看 \texttt{remove} 的倍数时应把这一点纳入考虑，本章后续表述也会相应收敛用词。

\textbf{没有副本容错，对比对象要打折理解。} RucksFS 目前是单 MetadataServer、单默认 DataServer，没有副本也没有 Raft。JuiceFS+TiKV 的 TiKV 即便部署成单节点，也仍然跑着 Raft 模块和 Percolator 两阶段提交，这些是为多副本和跨节点事务准备的开销。因此本章观察到的相对差距，主要反映的是"轻量单机 RocksDB 直连"与"通用分布式 KV 叠 FUSE"两种架构的固有区别，而不应解读为 RucksFS 作为一个完整分布式系统已经领先若干倍。第~\ref{sec:perf-comparison}~节在讨论具体倍数时会明确标注这一点。

\textbf{NFS 用 \texttt{noac} 挂载，\texttt{stat} 结果会放大差距。} 为了避免客户端属性缓存掩盖服务端性能，NFS 客户端挂载参数强制每次 \texttt{stat} 都回到服务端。这对反映 NFS 的原始元数据吞吐上限是公平的，但会让 \texttt{stat} 的倍数看起来异常悬殊；实际部署中 NFS 通常开启属性缓存。后续讨论 \texttt{stat} 吞吐时会把 \texttt{noac} 的影响单独说明。

明确这四点后，下面给出的数据就主要用来回答两个有限问题：一是 RucksFS 的设计在\emph{元数据关键路径}上是否取得了预期的相对优势；二是 DeltaOp 这一具体机制在受控消融实验下是否真实起作用。这也是本章的全部目标。

举一个具体例子。pjdfstest 里"当 \texttt{rename} 的目标路径前缀不是目录时应返回 \texttt{ENOTDIR}"这条规则，会按 regular、dir、symlink、fifo、socket、char、block 七种文件类型各生成一条用例。后四种依赖 \texttt{mknod} 或 \texttt{mkfifo} 来准备测试对象。RucksFS 这两个接口没实现，用例还没跑到 \texttt{rename} 本身的语义检查就因为 \texttt{EOPNOTSUPP} 中断了。测试结果记成"\texttt{rename} 的某条规则失败"，但实际上 RucksFS 对常规文件、目录、符号链接的 \texttt{rename} 都返回了正确结果。168 个失败里有 161 个都是这种情况。

\subsubsection{与第~\ref{sec:design}~章设计决策的对照}
\label{sec:pjdfstest-semantic}

为了避免"因为没实现所以不会失败"的误读，下面挑几条比较容易被忽视的边界语义出来看。这些用例都在 pjdfstest 的通过列里，而且每一条都能对上第~\ref{sec:design}~章里的某个具体设计决策。

\begin{itemize}
    \item \textbf{\texttt{unlink::open\_file\_not\_freed}}（对应：删除语义与句柄生命周期，第~\ref{sec:txn_model}~节）。文件被 \texttt{unlink} 之后如果还有打开句柄，应该能继续读写，物理回收要拖到最后一个句柄关闭。这要求 MetadataServer 同时维护 \texttt{nlink} 和打开计数：当 \texttt{nlink} 归零但计数还大于零时，对象进 \texttt{pending\_deletes}，由 \texttt{release} 触发最终清理。事务要是没把"删目录项"和"维持 inode 可达性"分开处理，这条用例就会挂。
    \item \textbf{\texttt{rename::to\_multiply\_linked::regular}}（对应：rename 覆盖逻辑，算法~\ref{alg:rename}）。\texttt{rename} 目标如果是一个有多条硬链接的文件，覆盖动作只能把目标的 \texttt{nlink} 减一，不能直接删目标 inode。这对应第~\ref{sec:design}~章 \texttt{rename} 流程里"按覆盖删除逻辑处理目标对象"那一步。
    \item \textbf{12 条 \texttt{*::enametoolong\_component} 用例}（对应：错误码穿透）。文件名长度超过 \texttt{NAME\_MAX} 时要返回 \texttt{ENAMETOOLONG}。这个约束在 MetadataServer 所有接受 \texttt{name} 字段的 RPC 入口统一校验，错误码经 gRPC 和 FUSE 两层传递后要保持不变。
    \item \textbf{\texttt{open::open\_trunc}}（对应：VfsCore 协调路径，第~\ref{sec:data_path}~节）。用 \texttt{O\_TRUNC} 打开已有文件时，\texttt{size} 要在打开操作里同步清零，同时刷新 \texttt{mtime} 和 \texttt{ctime}。FUSE 并不一定会再发一个独立的 \texttt{setattr}，所以这件事必须在 \texttt{open} 实现里一并搞定。
    \item \textbf{\texttt{rmdir::eexist\_enotempty\_non\_empty\_dir} 系列}（对应：事务内空目录检查，第~\ref{sec:posix_impl}~节）。删除非空目录应该返回 \texttt{ENOTEMPTY}。第~\ref{sec:design}~章讲过 \texttt{rmdir} 把空目录检查也放进了事务里，在一致性视图下做前缀扫描，免得检查结果在进事务前失效。
\end{itemize}

这些用例覆盖的不是孤立的接口行为，而是第~\ref{sec:design}~章里跨列族原子提交、句柄生命周期、统一加锁顺序和客户端协调路径在实际跑起来时的综合表现。它们能全部通过，就说明这套设计在分布式部署条件下达到了预期的语义效果。

\subsubsection{已知不实现项的取舍}

\texttt{mknod} 和 \texttt{mkfifo} 是本文有意不实现的，不是没来得及做。理由有两个。一是这两个接口管的是 Unix 域 socket、命名管道、字符设备节点、块设备节点，运行语义和内核设备号、内核管道缓冲区强绑在一起，跟"通过键值对持久化命名空间"这个研究主题是正交的。二是本文聚焦的负载（容器镜像、训练数据、日志、构建产物）基本不会用到这类对象，把工程预算花在用不到的接口上对论证元数据机制本身帮助不大。\texttt{open::interact\_2gb} 涉及的单文件大小上限，属于 DataServer 当前的开发期参数，调下分片策略就能放开。

48 个跳过用例里，27 个来自 \texttt{utimensat} 系列（该功能需要单独的 feature flag，当前没启用），剩下 21 个涉及 \texttt{remount}（FUSE 配置不允许运行时重新挂载）和跨文件系统操作（测试环境没配置第二个文件系统）。这些跳过项和 RucksFS 的元数据设计无关。

综上，pjdfstest 的结果可以这样理解：在 RucksFS 选择实现的接口范围内，POSIX 语义被完整地表达出来了；没实现的接口被显式标记，也不会用错误的形式去遮蔽真实问题。这个结论也给后面的性能对比立了一个前提——后面看到的那些吞吐数字，都建立在"语义正确"的基础之上。

NFS 的曲线几乎是平的：从 $N=2$ 的 605\,ops/s 到 $N=96$ 的 911\,ops/s，增幅不到 1.5 倍。瓶颈在 ext4 的目录互斥锁（\texttt{i\_rwsem}）——所有 \texttt{nfsd} 线程在同一父目录上的 \texttt{create} 都要先拿到该目录 inode 的写锁，再走 journal 提交。不管客户端加到多少，能真正并发执行的 \texttt{create} 都被这把锁压在了很低的水位。

JuiceFS+TiKV 在可测的三档里保持了准线性增长。它的元数据更新走 Percolator 两阶段提交，每次 \texttt{create} 在 TiKV 内部表现为对几个独立键的写入加上一次 \texttt{commit\_ts} 写入。共享父目录确实会引发对父 inode 键的并发更新，但 MVCC 按时间戳排序，冲突被推到后台 GC 里处理，不用在事务路径上同步等。不过同区间内 RucksFS-Delta 始终高出 2.8--4 倍。差距主要来自上层架构的不同：RucksFS 的一次 \texttt{create} 是 MetadataServer 内部的一次 \texttt{AtomicWriteBatch}，四类键共享一次 WAL 同步就完事了；TiKV 的一次 \texttt{create} 还要走完整的两阶段提交、过一遍 Raft 日志（单节点部署下 Raft 模块照样跑）。

RucksFS-Delta 在整个区间基本是近线性增长的，从 $N=2$ 的 1\,853\,ops/s 一直到 $N=96$ 的 68\,588\,ops/s，每客户端贡献大约 714\,ops/s。$N=96$ 时相对 NFS 有 75 倍左右，相对 JuiceFS+TiKV 在 $N=32$ 处有 4 倍。这两个倍数得分开看。对 NFS 的大幅领先主要是架构层面的差异，NFS 被 ext4 目录互斥锁串行化在了那里，这是本地文件系统面对共享热点目录的通用短板，和 RucksFS "设计得多好"关系不大。对 JuiceFS+TiKV 的 4 倍差距也不能简单地说成"设计更好"，更靠谱的说法是：RucksFS 作为单机直连 RocksDB 的原型，天然省掉了 TiKV 的 Percolator 两阶段提交和 Raft 日志开销——代价是当前实现没有副本容错，也没有跨节点事务。DeltaOp 把父目录更新从"读--改--写"改成追加写这件事，在这个背景下起到的作用是让共享父目录不会成为事务冲突热点，而不是把一条本来就很慢的路径"变快若干倍"。

\subsubsection{查询吞吐：写路径未阻塞读路径}

stat 的差距数字看起来挺大（$N=96$ 时 RucksFS-Delta 达到 1\,792 万 ops/s，NFS 约 4.5 万 ops/s），但这组数不能直接拿去做"某系统快几百倍"的宣传。两侧路径根本不一样。

NFS 客户端挂载时加了 \texttt{noac}，是为了不让客户端属性缓存把服务端能力盖住，代价就是每次 \texttt{stat} 都被逼着走一遍网络。实际部署中 NFS 一般是允许客户端缓存属性的，真实吞吐会高得多。再看 RucksFS 这边，MetadataServer 的 \texttt{stat} 就是 RocksDB 的一次点查，上一轮 \texttt{create} 刚写进去的 inode 记录还在 BlockCache 里，命中后延迟在微秒级。另外 LSM-tree 的读写不共用同一把锁，\texttt{stat} 不会被写路径挤占——这是 KV 引擎本身自带的特性，不是本文另外做的设计。

更关键的是，DeltaOp 在这里起了个间接作用：把父目录的属性写从热路径上拿走之后，\texttt{stat} 的 \texttt{lookup} 阶段就不会再被父 inode 的写锁挡住。这一点会在下一节 NoDelta 对照里被直接量化。

\subsubsection{删除吞吐}

remove 的整体形态和 create 很像，但对比口径要单独说一下：本文的 DataServer 当前没实现真正的空间回收，\texttt{delete\_data} 是空操作；而 NFS 的 \texttt{unlink} 要触发 ext4 的 inode 位图翻转和 journal 提交，TiKV 的 \texttt{unlink} 要写 tombstone 并最终参与 GC。在 mdtest 这种空文件负载下，这部分工作量差异会被放大。NFS 的扩展曲线形状和 create 差不多，瓶颈还是父目录写锁加 journal 提交。$N=96$ 时 RucksFS-Delta 达到 70\,859\,ops/s，对 NFS 约 23 倍，对 JuiceFS+TiKV 在 $N=32$ 处约 4 倍。这里给出数字，主要不是为了强调倍数本身，而是想说明"元数据路径在 hard 模式下不会成为瓶颈"这个结论在删除侧也站得住；要把它当作一个完整文件系统的删除吞吐来解读，还得把真实的数据回收路径补上。

\subsection{DeltaOp 增量机制的内部对比}
\label{sec:delta-vs-nodelta}

第~\ref{sec:design}~章第~\ref{sec:txn_model}~节的核心设计假设是这样的：把父目录属性更新从"覆盖父 inode 基值"改成"向独立列族追加一条 DeltaOp"，就可以避开热点目录下的事务冲突，让元数据写入在并发度提升时还能维持线性扩展。本节要做的，是用 Delta 和 NoDelta 两种构建变体在同一硬件、同一负载下的直接对比来验证这个假设。这是本章最重要的一组消融实验：假设要是成立，应该能看到三个可预测的现象——（1）低并发时两者没差距，（2）高并发共享目录下 NoDelta 出现非线性下滑而 Delta 保持线性，（3）独立目录（easy 模式）下两者始终重合。

\subsubsection{NoDelta 吞吐崩溃的机理}

NoDelta 从 $N=32$ 起出现急剧下滑，原因得从 MetadataServer 端的事务执行细节里找。

NoDelta 路径下，一次 \texttt{create} 的事务序列是这样走的：对父目录 inode 键执行 \texttt{get\_for\_update} 拿写锁，读出当前基值，改写 \texttt{mtime}/\texttt{ctime} 后 \texttt{put} 回去，连同新建的目录项和 inode 一起提交。共享父目录下，所有 \texttt{create} 事务都在抢同一条父 inode 键的写锁。第~\ref{sec:design}~章里讲过的悲观并发控制会让拿不到锁的事务先等着；等超时就回滚，再由服务端重试。当并发度足够高的时候，重试自己会形成放大效应：被拒掉的事务先释放已持有的其他锁、回滚已写进去的 batch，然后再冲回去抢，可这时队列里又积了更多重试请求。这就是写冲突重试风暴。

启用 DeltaOp 以后情况就完全不一样了。\texttt{create} 不再去动父 inode 基值，父目录的属性变化被编码为一条 DeltaOp 写到 \texttt{delta\_entries} 列族里。因为键里包含单调递增的序列号 \texttt{seq}，每次写入的目标键都不同，根本不会和其他事务在同一个键上产生锁冲突。父 inode 基值只在后台 \texttt{DeltaCompactionWorker} 折叠增量时被写一次，这件事发生在测试的关键路径之外。锁冲突被消掉的同时，热点目录上的随机写也被转成了顺序追加。

把机理对上数字：NoDelta 在 $N=32$ 时还能保持线性（锁等待还没触发明显放大），$N=64$ 处的 40\% 下滑对应冲突重试开始堆积，$N=96$ 处的持续下降对应重试窗口快饱和了。Delta 不会进入这种状态，所以一路都是线性。

\subsubsection{对第~\ref{sec:design}~章设计假设的实验验证}

把上面几项观察整理一下，对第~\ref{sec:design}~章的几条设计陈述做一个逐点回应。

\begin{itemize}
    \item \textbf{"DeltaOp 在低并发下不引入额外开销"}（第~\ref{sec:txn_model}~节）。$N \leq 32$ 时 Delta 和 NoDelta 的吞吐差距都在 2\% 以内。增量追加路径只在事务热路径上省锁竞争的开销，不额外加固定开销。
    \item \textbf{"DeltaOp 在高并发下消除父目录事务冲突"}（第~\ref{sec:txn_model}~节）。$N=64$ 起 NoDelta 出现典型的冲突崩溃，Delta 维持线性。两条曲线分叉的位置正好对着 PCC 争抢窗口被放大的临界点。
    \item \textbf{"DeltaOp 的优势仅在共享目录场景中出现"}。easy 模式下二者吞吐完全重合，只有 hard 模式才出现差距。这就排除了 DeltaOp 收益来自别的系统层面副作用的可能性。
    \item \textbf{读路径间接受益}。\texttt{stat} 在 NoDelta 下被父 inode 写锁挡住，在 Delta 下不再受阻。这一条说明 DeltaOp 的收益不仅停留在写入路径。
\end{itemize}

四条观察互相印证，构成对 DeltaOp 增量机制的端到端实验验证：第~\ref{sec:design}~章的理论设计在分布式部署下确实落到了可被实测的性能差异上，而且差异的方向、幅度和并发拐点都和机制本身的工作原理对得上。

\subsection{本章小结}

本章的实验沿着两条线展开。

第一条线是 POSIX 合规性。pjdfstest 的 398 个用例里，182 条直接通过；168 条失败中有 167 条来自 \texttt{mknod}/\texttt{mkfifo} 派生出的特殊文件类型用例，1 条来自 DataServer 的开发期文件大小上限。和核心元数据语义直接相关的失败为零。"打开后 \texttt{unlink}"、硬链接覆盖、长名称错误码、\texttt{O\_TRUNC} 时间戳、事务内空目录检查这些边界用例全部通过，正好对上第~\ref{sec:design}~章的跨列族原子提交、句柄生命周期维护、统一加锁顺序和客户端协调路径。\texttt{mknod}/\texttt{mkfifo} 的不实现是针对典型分布式存储负载做出的显式取舍，并不是用失败来遮盖真实问题。

第二条线是分布式性能。在 1 台 64 核服务器加最多 96 台 2 核客户端的集群上，RucksFS-Delta 在 \texttt{create}、\texttt{stat}、\texttt{remove} 三类操作上相对 NFS 和 JuiceFS+TiKV 都保持了稳定的扩展优势。$N=96$ 时 \texttt{create} 达到 6.86 万 ops/s，\texttt{remove} 达到 7.09 万 ops/s。具体倍数不在这里复述，但有几句话需要强调：对 NFS 的大幅差距主要来自 ext4 目录锁对本地文件系统的天然制约；对 JuiceFS+TiKV 的 4 倍差距更多反映的是"单机直连 RocksDB"相对"通用分布式 KV 叠 FUSE"在 2PC+Raft 开销上的取舍，而不是 RucksFS 作为一个完整分布式系统已经实现了质的超越。此外，空文件负载、DataServer 的最小实现以及 \texttt{noac} 挂载这些实验设置都会影响对比口径，这些限制已经在第~\ref{sec:expr-env}~节单独说明过。

DeltaOp 的消融实验是本章最有说明力的一组。在只切换编译特性开关的条件下，hard 模式从 $N=64$ 起 Delta 和 NoDelta 出现明显分叉：$N=96$ 时 \texttt{create} 比值 5.81，\texttt{remove} 比值 6.12，\texttt{stat} 比值 136；easy 模式下两者则完全重合。因为同硬件、同负载、只换一个机制开关，这组对比直接说明了增量更新机制在共享父目录下确实消掉了它本该消掉的那种写冲突，读路径也因此不再被写锁拖慢。这是本章里能在诚实的对比口径下、比较干净地归因到具体设计决策的一组结果。
