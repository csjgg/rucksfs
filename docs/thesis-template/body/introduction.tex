\section{绪\hspace{1em}论}

文件系统的元数据管理——包括 inode 属性的维护、目录项的组织以及文件操作的事务性保证——直接决定了存储系统在高并发场景下的吞吐上限。当传统 B+ 树引擎面对每秒数万次的小文件创建时，就地修改带来的随机 I/O 和目录锁串行化问题会迅速成为瓶颈。基于 LSM-tree 的键值存储以顺序写入取代随机写入，为这一问题提供了有竞争力的替代方案。

\subsection{课题背景、目的与意义}

\subsubsection{研究背景与趋势}

大规模模型训练、容器化微服务和云端数据湖等场景有一个共同特征：它们产生并消费海量小文件。一次分布式训练作业可能在 checkpoint 阶段写出数十亿个参数分片；Kubernetes 集群中的 Pod 在启动过程中需要创建和读取大量配置文件与临时卷。百度沧海存储团队在 Mantle\cite{li2025mantle}中给出了一组直观的数据：其云对象存储管理的文件对象数已达万亿量级，且元数据请求规模仍在逐年数倍增长。当文件数量攀升至这一量级时，系统的性能瓶颈通常不在数据传输带宽，而在元数据路径上。Ren 和 Gibson\cite{ren2013tablefs}的测量表明，\texttt{stat}、\texttt{open}、\texttt{readdir} 等元数据操作占文件系统全部 I/O 操作的 60\%--80\%。单个热点目录下的文件数一旦达到百万级，元数据查询与更新的延迟就会成为系统吞吐的天花板。

现有的本地文件系统（ext4、XFS）采用 B+ 树或其变体（如 ext4 的 H-tree）组织目录项和 inode 索引。B+ 树的写入方式是就地修改（in-place update），每次变更直接改写磁盘上对应数据页的内容。这种方式在顺序扫描和读密集负载下表现稳健，但面对高频写入存在两个固有瓶颈。其一，每次写入都是随机 I/O——即便实际只修改几十字节的目录项，也要完整写出一个数据页外加一条 journal 日志。其二，目录级锁把同一目录下的并发写入串行化了：XFS 在单目录下连续创建数万文件时，日志区域的锁竞争足以将吞吐量压到明显低于理论峰值的水平。

LSM-tree\cite{oneil1996lsmtree}（Log-Structured Merge-tree）的写入模型与此截然不同。写入操作先顺序追加到内存中的 MemTable；MemTable 积累到一定容量后，由后台线程排序并批量刷出为磁盘上的 SSTable 文件；多层 SSTable 之间再通过 compaction 消除冗余。随机写入被转化为顺序写入，这一特性对于大量小粒度写操作——正是文件创建和删除所产生的负载类型——有天然的优势。工业界的实践印证了这一判断：Google 的 LevelDB\cite{google2011leveldb}在 2011 年将 LSM-tree 工程化；Facebook 在此基础上开发的 RocksDB\cite{dong2021rocksdb}增加了列族、事务和前缀布隆过滤器，现已成为 CockroachDB、TiKV 等数十个系统的底层存储引擎。在文件系统领域，TableFS\cite{ren2013tablefs}在 2013 年率先将全部元数据存入 LevelDB 并通过 FUSE 挂载，小文件创建速度达到了同期 ext4 的 2--3 倍。此后 Apache Ozone\cite{apache2021ozone}和 Facebook Tectonic\cite{pan2021tectonic}先后在 EB 级生产环境中部署了以 RocksDB 为核心的元数据引擎。从 2013 年的学术原型到 2021 年的 EB 级工业系统，KV 元数据方案的可行性和竞争力已无需证明，但其在键编码设计、写放大控制和事务保障等方面仍有开放的设计问题。

本课题聚焦的正是这些问题。具体而言，核心研究内容是元数据的 KV 存储设计——如何将 inode 属性、目录项和属性变更日志高效地映射到键值引擎中，并据此实现 create、unlink、rename、readdir 等 POSIX 文件操作。存储方案决定了操作的性能上限，文件操作则是检验方案是否可行的载体。系统通过 FUSE\cite{szeredi2005fuse}框架对外暴露标准 POSIX 接口。选择 FUSE 是出于工程上的务实考量——用户态开发迭代更快，与 JuiceFS\cite{juicefs2021}和 CubeFS\cite{liu2019cubefs}等主流系统的接入方式一致，且 FUSE 层的文件操作逻辑在本质上等价于内核 VFS 接口，必要时可平滑迁移到内核态。

KV 引擎的数据模型是从字节串到字节串的扁平映射，而文件系统是一棵以根目录为起点的层次化树。键编码方案必须同时满足两类需求：一是 \texttt{readdir}——同一父目录下的所有子项在键空间中要连续排列，以便用前缀迭代器一次性扫出；二是 \texttt{lookup}——给定父 inode 号和文件名，能在 $O(1)$ 时间内定位到对应的目录项。如果直接用全路径（如 \texttt{/a/b/c}）做键，\texttt{rename} 就不得不递归修改整棵子树的所有键——代价不可接受。TableFS\cite{ren2013tablefs}的方案以"父 inode 号 + 文件名"拼接作为键，方向正确，但 inode 号使用原生字节序存储，导致同一目录下的条目在 LSM-tree 中未必相邻，无法利用前缀布隆过滤器。键编码还必须保持字节序与数值序的一致性（byte-order preserving），否则范围查询的语义会出错。

\textbf{父目录更新引发的写放大。}
POSIX 语义\cite{ieee2017posix}要求：每创建或删除一个文件，父目录的 \texttt{mtime}（修改时间）和 \texttt{nlink}（链接计数）必须随之更新。在 KV 存储中，这意味着每次 \texttt{create} 或 \texttt{unlink} 都要对父目录 inode 记录执行一次读取-修改-写回。对于低频访问的目录这不构成问题；但一旦某个目录成为热点——\texttt{/tmp}、CI 构建输出目录、模型训练的 checkpoint 目录都属于此类——开销会急剧膨胀。设想一个目录在 1 秒内发生 1,000 次文件创建，父目录 inode 就要被读写 1,000 次，绝大部分读取到的值在写回前就已过期。更深层的问题在于，RocksDB 中对同一 key 的密集 \texttt{Get+Put} 序列会加速 MemTable 填满和 L0 文件堆积，触发不必要的 compaction，形成写放大的连锁反应。

\textbf{并发目录操作的正确性。}
多个线程（或未来的多个客户端）同时操作同一目录时，TOCTOU（Time-of-Check-to-Time-of-Use）竞态几乎不可避免。一个典型场景：线程 A 在执行 \texttt{rename} 时确认目标路径尚不存在，但写入新目录项之前，线程 B 已经在该路径上创建了同名文件——结果要么丢失 B 的文件，要么目录项出现重复。传统内核文件系统用 VFS 层的 inode 互斥锁串行化目录操作；KV 引擎上没有这层保护，需要依赖存储层自身的事务机制。RocksDB 提供悲观和乐观两种事务模式，但事务粒度怎么选——锁整个目录还是只锁涉及的 key——直接影响并发度和实现复杂度之间的权衡。

\subsubsection{课题目的与意义}

本课题的目标是：设计并实现一个以 RocksDB 为元数据引擎的用户态文件系统 RucksFS，围绕上述三个问题——键编码、写放大、并发一致性——给出一套完整的解决方案，并通过 FUSE 接口将其落地为可运行的 POSIX 文件系统加以验证。

键编码方面，系统采用多列族架构，将 inode 属性与目录项分置于独立列族，各自配置适用的布隆过滤器与压缩策略。目录项列族的键遵循"以边为中心"的思路：以父 inode 号与子文件名的大端序拼接作为键，每条目录项直接对应目录树中一条父子边。文件操作由此转化为对边的增删——\texttt{create} 插入一条边，\texttt{unlink} 删除一条边，\texttt{rename} 删旧边、插新边——代价均为常数时间。全路径方案在 \texttt{rename} 时需递归改写子树中每条键，开销随子树规模线性增长。大端序保证字节序与数值序一致，使得同一父目录的全部子边在键空间中自然相邻，\texttt{readdir} 和 \texttt{lookup} 均可高效完成。

写放大方面，本课题提出增量更新（delta-based update）机制。文件创建或删除时不直接改写父目录 inode，而是将变化量（nlink $+1$、mtime 更新等）作为独立的增量记录（DeltaOp）追加到专用列族。读取 inode 时再将基值与所有增量在线合并。后台压缩线程在增量条目超过 32 条时执行折叠，将累积变更写入一条新的基值。热目录的写路径因此从一次随机 \texttt{Get+Put} 变为一次顺序 \texttt{Put}。

一致性方面，系统采用 RocksDB 悲观事务（PCC）为 \texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rmdir}、\texttt{rename} 提供原子性保证——事务在读取阶段即获取行级锁，封堵 TOCTOU 窗口。锁冲突时以 \texttt{EAGAIN} 语义自动重试，上限 3 次。

预期交付物是一个以 Rust\cite{klabnik2023rust}编写的单二进制程序，在同等硬件条件下与 JuiceFS、CubeFS、NFS 做元数据吞吐量和延迟的对比。学术层面上，本课题填补了 TableFS\cite{ren2013tablefs}之后在"单机 KV 元数据引擎 + 完整事务保障"方向上的空缺——TableFS 基于 LevelDB 且不具备事务能力，IndexFS\cite{ren2014indexfs}侧重分布式扩展而非单机深度优化。工程层面上，Rust 的所有权和生命周期机制在编译期排除内存安全问题，对于文件系统这类可靠性要求极高的基础设施有切实价值。


\subsection{国内外研究现状}

用 KV 引擎管理文件系统元数据的研究始于 2013 年，至今已积累了从单机原型到 EB 级生产系统的丰富实践。

\subsubsection{基于KV存储的元数据管理研究}

TableFS\cite{ren2013tablefs}是这一方向最早的系统性工作。2013 年，Carnegie Mellon University 的 Ren 和 Gibson 将 ext4 的目录项和 inode 信息整体存入 LevelDB，并通过 FUSE 挂载为标准文件系统。TableFS 的设计思路直截了当：以（父 inode 号, 文件名）的拼接作为 LevelDB 的键，以序列化后的 \texttt{struct stat} 加上可选的内联数据作为值。在小文件（$<$ 4~KB）场景下，TableFS 的 \texttt{create} 吞吐量是同期 ext4 的 2--3 倍，这一结果首次验证了 KV 存储用于元数据管理的可行性。但 TableFS 也暴露了若干缺陷：（1）\texttt{readdir} 操作需要扫描 LevelDB 的全部键空间并过滤出指定父目录的条目——当文件系统中的文件总数增长到百万级别时，\texttt{readdir} 的延迟急剧上升；（2）LevelDB 本身不提供任何事务语义，多线程写入时的正确性完全依赖外部加锁；（3）键编码方案中父目录 inode 号使用原生字节序存储，导致同一目录下的条目在 LSM-tree 中不一定连续排列，无法利用前缀迭代器或前缀布隆过滤器加速查找。此外，TableFS 没有考虑父目录 inode 更新带来的写放大问题。

同一研究团队在次年推出了 IndexFS\cite{ren2014indexfs}，将 TableFS 的思路扩展到分布式环境。IndexFS 在每个元数据节点上运行一个 LevelDB 实例，目录项通过 GIGA+ 动态哈希分区算法分布到不同节点。其关键贡献是引入了列式目录布局（columnar directory layout）：将 \texttt{readdir} 所需的信息（文件名列表）与 \texttt{getattr} 所需的信息（inode 属性）分开存储在不同的 SSTable 列中，使得目录遍历只需读取文件名列，跳过体积更大的 inode 属性数据。IndexFS 在 128 节点的集群上实现了每秒超过 100 万次 \texttt{create} 操作，获得了 SC'14 的最佳论文奖。不过 IndexFS 的部署要求相当苛刻：每个节点需要独立维护一个 LevelDB 实例和一套目录缓存状态机，还需要一个独立的 GIGA+ 组件来处理跨节点的目录分裂。对于中小规模的使用场景，这样的架构存在过度设计之嫌。

LocoFS\cite{li2017locofs}在 2017 年从一个不同的角度切入。Siyang Li 等人观察到文件访问具有时间局部性和空间局部性——近期被创建的文件很可能很快被访问，同一目录下的文件倾向于被一起操作。LocoFS 据此设计了一种松耦合的元数据分布策略，将热点目录的元数据根据父目录的哈希值分配到固定的服务器上，客户端可以通过哈希直接计算出目标服务器地址，无需查询中心化的映射表。这种优化对 \texttt{stat}-after-\texttt{create} 这类访问模式效果不错，但预取策略的命中率在访问模式频繁变化的场景下会打折扣。

Xu 等人\cite{xu2014drop}在 2014 年提出了一种面向 EB 级规模的可扩展元数据管理方案，将目录树按子树粒度划分到不同的元数据服务器上，并通过动态迁移机制来平衡负载。在大规模静态工作负载下，该方案表现良好，但频繁的 \texttt{rename} 操作（跨子树移动）会触发元数据迁移，引入较高的延迟。

Yang 等人\cite{yang2020batchfile}在 2020 年从一个实用角度分析了大规模文件访问中的批量操作优化问题。该工作发现，在包含百万量级文件的目录中，逐个执行 \texttt{stat} 和 \texttt{open} 的效率远低于批量接口。通过设计面向批量文件操作的内核扩展，将 \texttt{readdir}+\texttt{stat} 的复合操作在内核内部合并执行，减少了用户态/内核态切换次数。这一思路对本课题的 \texttt{readdirplus} 实现有参考价值——RucksFS 通过 FUSE 的 \texttt{readdirplus} 接口在一次调用中同时返回目录项和 inode 属性，实现了类似的批量优化效果。

SingularFS\cite{li2022singularfs}在 ATC'23 上展示了一个激进的设计选择：仅用单个元数据服务器支撑十亿级文件规模的分布式文件系统。其核心技巧是通过无日志元数据操作（log-free metadata operations）消除崩溃一致性开销，并结合层次化并发控制和 NUMA 感知的 inode 分区来提升并行度。SingularFS 利用 RDMA 和持久内存技术，在单台服务器上实现了远超传统多元数据节点方案的吞吐量。这一结果表明，单机元数据引擎的性能天花板远高于通常的认知。范立衡等人\cite{fan2014kvmetadata}也较早在国内探索了键值存储用于元数据集群副本一致性的管理策略，通过优化副本放置和读写机制来保证最终一致性。

\subsubsection{云原生分布式文件系统}

在 KV 元数据研究逐步成熟的同时，工业界的分布式文件系统也在快速演进。回顾这一演进路径有助于理解本课题的设计选择所处的技术背景。

早期的分布式文件系统以集中式元数据管理为主。Google File System（GFS）\cite{ghemawat2003gfs}于 2003 年发表于 SOSP，采用单 Master 节点维护所有文件的命名空间和块映射信息，全部保存在内存中。GFS 的设计假设是文件数量有限（百万量级）但单个文件体积很大（GB 级），因此将元数据全部驻留内存是可行的。这一假设在后续的实践中被打破：随着 MapReduce 作业产生越来越多的中间文件，GFS 的 Master 节点逐渐成为整个系统的容量瓶颈和单点故障源。Hadoop Distributed File System（HDFS）\cite{shvachko2010hdfs}沿袭了 GFS 的单 NameNode 设计，将所有元数据保存在 Java 堆内存中。HDFS 社区在此后多年间持续应对这一架构限制：Federation 通过多个 NameNode 分担命名空间，但每个 NameNode 仍受限于 JVM 堆大小；HA 方案通过 ZooKeeper 实现了故障切换，但元数据本身并未持久化到外部存储。Yahoo! 在 2015 年报告称，其最大的 HDFS 集群运行着约 6 亿个文件对象，NameNode 的 JVM 堆占用超过 100~GB，启动时间长达 30 分钟以上。这些案例促使研究者寻找元数据外置化（off-heap metadata）的方案。

Ceph\cite{weil2006ceph}在 2006 年的 OSDI 上提出了一种不同的思路：用动态子树划分（dynamic subtree partitioning）将目录树的不同子树分配到不同的元数据服务器（MDS）上。当某棵子树的访问负载超过阈值时，Ceph 将其拆分并迁移到负载较低的 MDS。这种方案在理论上具备自适应的负载均衡能力，但实际部署中发现子树迁移的开销不容忽视——迁移期间需要冻结子树下所有文件的写入操作，对延迟敏感的应用造成干扰。Ceph 后来在单 MDS 模式下应用最为广泛，多 MDS 的生产部署仍然相对少见。Ceph 的元数据存储在本地文件系统（最初是 Btrfs，后迁移到 BlueStore/RocksDB），但元数据的内部组织并未利用 KV 引擎的前缀扫描等特性来优化目录遍历。

2015 年 Kubernetes 生态成型后，面向容器和微服务架构的云原生文件系统密集涌现。

JuiceFS\cite{juicefs2021}是国内使用较广的开源 POSIX 文件系统。它的架构将元数据和数据完全分离：元数据存储在 Redis、TiKV 或 MySQL 等外部引擎中，数据块则写入 S3 兼容的对象存储。选择 Redis 作为元数据引擎时，JuiceFS 利用 Redis 的 Hash 结构存储 inode 属性，用 Sorted Set 存储目录项。这种设计的优势在于部署灵活——用户可以根据自身条件选择最合适的后端组合。劣势则在于性能和可靠性完全取决于外部组件：Redis 的单线程模型在元数据操作超过每秒数万次时会成为瓶颈；TiKV 虽然吞吐量更高，却引入了 Raft 共识协议带来的额外延迟。JuiceFS 官方基准测试显示，Redis 后端在单客户端下的文件创建吞吐量约为 1,410 ops/s，而 MySQL 后端仅约 350 ops/s。JuiceFS 的 FUSE 客户端使用 Go 语言实现，Go 运行时的垃圾回收（GC）暂停在极端负载下曾被部分用户报告为延迟抖动的来源之一。

CubeFS\cite{liu2019cubefs}（最初名为 CFS）于 2019 年在 SIGMOD 上发表，后捐赠给 CNCF 并于 2024 年成为毕业项目。与 JuiceFS 依赖外部数据库不同，CubeFS 自带元数据引擎：每个元数据分区（MetaPartition）使用内存中的 B-tree 维护目录结构，多副本之间通过 Multi-Raft 协议保持一致性。每个分区独立运行一个 Raft 组，分区的粒度是 inode 号范围。CubeFS 在阿里云和京东云的生产环境中管理着 PB 级数据。它的元数据在读路径上表现出色（内存 B-tree 的查找延迟在微秒级），但写路径受到 Raft 日志同步的制约——每次写操作至少需要等待多数副本确认，典型延迟在毫秒级。内存中的 B-tree 也限制了单节点可管理的元数据量：按每个 inode 约 300 字节内存估算，十亿文件需要约 280~GB 内存，CubeFS 文档建议单个分区不超过 1,000 万个 inode。此外，CubeFS 的 MetaPartition 一旦创建，其 inode 范围就被固定，后期调整分区边界需要人工干预。

Apache Ozone\cite{apache2021ozone}作为 HDFS 的下一代替代方案，将元数据引擎从 NameNode 的堆内内存替换为 RocksDB + Raft 的组合。Ozone Manager（OM）节点将所有元数据操作先写入 Raft 日志，然后应用到本地的 RocksDB 实例中。这种设计解决了 HDFS NameNode 因 JVM 堆大小限制而无法管理超过 5 亿文件的问题。Ozone 1.4 版本的基准测试显示，单个 OM 节点能够支持每秒约 2 万次 \texttt{create} 操作。但 Ozone 当前的 POSIX 兼容性较弱，其原生接口是 S3 和 Hadoop FileSystem API 而非 FUSE。在社区中曾有通过 s3fs-fuse 挂载 Ozone 的尝试，但多出一层 S3 协议转换带来了明显的性能折损。

Facebook 的 Tectonic\cite{pan2021tectonic}在 FAST'21 上公开了其架构细节，管理着数 EB 的数据和数千亿个文件对象。Tectonic 将元数据存储在一个深度定制的分布式 KV 系统 ZippyDB 中（底层同样是 RocksDB）。Tectonic 的一个关键设计是将元数据操作的语义约束从存储层上移到客户端库——由客户端负责检测冲突并重试，从而减少了存储层的协调开销。这种"胖客户端"的理念对本课题的事务设计有一定启发：RucksFS 的 EmbeddedClient 模式下，元数据服务和客户端运行在同一进程中，事务的开始、提交和重试都在进程内完成，不涉及网络往返。但 Tectonic 高度特化于 Facebook 的内部基础设施，并未开源，其具体的 KV 键编码方案和事务协议也未完整公开。

上述系统的元数据管理方案按技术路线可归为三类：内存驻留型（GFS、HDFS、CubeFS）、外部数据库型（JuiceFS）、嵌入式 KV 型（Ozone、Tectonic）。三类方案各有侧重，其核心指标对比如表~\ref{tab:cloud_fs_compare}所示。

\subsubsection{近年前沿研究}

2023 年以来，文件系统元数据领域出现了若干方向性的新探索。

FUSEE\cite{shen2023fusee}发表于 FAST'23，针对 FUSE 框架本身的性能瓶颈进行优化。FUSEE 的核心思想是推测性执行（speculative execution）：在用户态守护进程收到请求之前，根据历史访问模式预测即将到来的操作，并提前将可能需要的数据加载到内核缓存中。例如，在观察到一个 \texttt{lookup} 请求后，FUSEE 推测后续大概率会出现针对同一 inode 的 \texttt{getattr} 或 \texttt{open} 调用，于是提前将对应的 inode 数据加载到缓存中。在元数据密集型工作负载上，FUSEE 的推测命中率达到 85\% 以上，将端到端延迟降低了约 30\%。不过推测失败时的回滚开销和额外的内存占用是该方案需要关注的成本。FUSEE 的预测模型基于频率统计的马尔可夫链，对非规律性的访问模式适应能力有限。

Dong 等人在 ICCD'23 上提出了一种低延迟的全路径索引方案\cite{dong2023duplex}。该方案维护了两种索引结构：对于深层路径（超过 3 级），系统维护从根到叶的全路径哈希索引以避免逐级 \texttt{lookup}；对于浅层路径，则沿用传统的父目录加文件名索引。两种模式的切换由路径深度自动决定。初步测量数据显示，在深度超过 10 层的目录结构中，该方案比逐级解析快 3--5 倍。

Wang 等人\cite{wang2023cfs}在 EuroSys'23 上发表的 CFS（注意区别于 CubeFS 的早期同名项目）对分布式元数据服务中临界区（critical section）的范围做了精细化分析。传统方案在处理 \texttt{rename} 等涉及多个目录的操作时，往往采用粗粒度的目录级锁，导致不相关的操作被串行化。CFS 通过形式化方法识别出各个 POSIX 操作中真正需要互斥的 key 集合，将临界区缩减到最小。实验表明，这种优化将 \texttt{rename} 密集型工作负载的吞吐量提升了 3.2$\times$。

MetaHive\cite{heidari2024metahive}在 2024 年针对 KV 存储引擎在文件系统元数据场景下的缓存效率问题展开研究。Heidari 等人观察到，通用 KV 存储的缓存替换策略（如 LRU）未考虑文件系统元数据的访问局部性——同一目录下的 inode 被连续访问的概率远高于随机 key。MetaHive 据此设计了一种元数据感知的缓存分区方案，将热目录的 inode 条目和目录项缓存在独立的分区中，避免被其他类型的 KV 操作驱逐。

Mantle\cite{li2025mantle}发表于 SOSP'25，由百度沧海存储团队完成。Mantle 面向云对象存储的层次化元数据管理，其核心优化是引入 IndexNode 组件将多次递归路径解析 RPC 合并为单次 RPC，减少深层路径查找的网络往返次数。同时采用多引擎存储架构——需要复杂扫描的元数据表使用 RocksDB，点查询为主的索引表使用高性能哈希引擎，近期版本在内存中通过 MVCC 保持约 5 秒的热窗口。在百度内部的生产环境中，Mantle 相比 Tectonic 和 InfiniFS 等系统将元数据访问延迟降低了 6.6\%--99.1\%。

SwitchFS\cite{xu2026switchfs}发表于 EuroSys'26，探索了利用可编程交换机加速分布式文件系统元数据更新的可能性。通过在 Tofino 2 等支持 P4 编程的交换芯片中嵌入元数据操作的冲突检测逻辑，SwitchFS 将部分一致性协调从服务器端卸载到网络层，减少了服务器间的往返延迟。这种 in-network computing 的思路虽然新颖，但对网络硬件有特殊要求，部署门槛较高。

另一个值得关注的方向是操作系统层面的文件系统优化。FetchBPF\cite{mishler2024fetchbpf}发表于 ATC'24，提出了利用 eBPF 在 Linux 内核中实现可定制预取策略的框架。用户可以编写 eBPF 程序定义文件访问的预取规则，由内核在 I/O 路径中直接执行，避免了用户态预取方案的上下文切换开销。FetchBPF 在顺序读工作负载上实现了接近理论带宽的吞吐量，对随机读场景的预取命中率也有改善。这一工作启示了本课题未来可探索的优化方向：在 FUSE 守护进程与内核之间的 \texttt{/dev/fuse} 通道上，通过 eBPF 程序预取即将被访问的元数据，减少守护进程的响应延迟。同期的 FBMM\cite{tabatabai2024fbmm}同样发表于 ATC'24，展示了将 Linux VFS 接口用于非传统目的的可能性——FBMM 将内存管理器实现为文件系统（Memory-manager File System），利用 VFS 的可扩展性在不修改内核核心代码的情况下引入新的内存管理策略。这种"文件系统即抽象层"的理念与本课题使用 FUSE 将 KV 存储暴露为 POSIX 接口的思路有异曲同工之处。

龚拓宇\cite{gong2020rocksmeta}在 2020 年的清华大学硕士论文中，结合 RDMA 技术与 RocksDB 设计了分布式元数据服务原型 RocksMeta，在单节点上实现了超过 10 万 IOPS 的元数据操作吞吐量。该工作重点解决了目录树结构与 KV 存储的适配问题，其键编码方案与本课题的设计思路有相似之处，但 RocksMeta 侧重网络传输优化（RDMA 旁路内核协议栈），而本课题侧重存储层的写放大优化（增量更新与懒折叠）。吴雨飞等人\cite{wu2025dfscache}则在 2025 年针对基于 TiKV 的分布式文件系统元数据缓存一致性问题，提出了并发广播与惰性删除相结合的轻量级维护机制，为客户端缓存失效策略提供了新的参考。

\subsubsection{现有研究不足与本课题定位}

将上述工作放在一起审视，有几个问题尚未被充分回答。

第一，事务与并发控制薄弱。TableFS\cite{ren2013tablefs}所用的 LevelDB 没有任何事务语义，并发写入只能靠粗粒度互斥锁兜底。IndexFS\cite{ren2014indexfs}引入了 WriteBatch 保证单次操作的原子性，但 \texttt{rename} 需要同时修改源目录和目标目录的多条记录，WriteBatch 无法覆盖跨操作的一致性约束。RocksDB 从 6.0 版本起便提供了成熟的悲观事务（PessimisticTransactionDB），支持行级锁和冲突检测，然而目前尚未见到将其系统性地应用于文件系统元数据操作的公开工作。

第二，父目录 inode 的写放大问题长期被忽视。TableFS、JuiceFS、CubeFS 在处理 \texttt{create} 或 \texttt{unlink} 时无一例外地采用读取-修改-写回模式。RocksDB 的 Merge Operator 提供了一种延迟合并的可能性，但如何将其语义对接到 POSIX 文件操作上，仍需细致的工程设计。增量更新在数据库领域（CRDT、delta state）已有成熟实践，将其移植到文件系统元数据场景的工作却不多。

第三，现有系统偏重分布式可扩展性，对单机场景关注不足。IndexFS、CubeFS、Tectonic 引入了 Raft 共识、一致性哈希、分区迁移等重型机制；对于只需要一台或少量节点的场景，这些机制带来的额外延迟和运维复杂度反而成了负担。表~\ref{tab:cloud_fs_compare}也反映出一个取舍：Ozone 和 Tectonic 虽用了 RocksDB，POSIX 兼容性却很弱甚至完全不支持——学术界更倾向于发表大规模系统，单机场景下的深度优化空间被低估了。

综合以上分析，本课题定位于：在单机环境下，以 RocksDB 为基础构建面向 POSIX 语义的元数据引擎原型（RucksFS），利用多列族、悲观事务和前缀迭代器等特性，设计适配文件系统访问模式的键编码方案与增量更新机制。与已有工作的差异集中在三点：（1）系统性地使用 PCC 事务为所有目录变更操作提供原子性保证，而不是依赖外部锁或仅用 WriteBatch；（2）通过 DeltaOp 增量日志与懒折叠专门解决父目录 inode 的写放大问题；（3）以 Rust 编写，基于 FUSE 异步运行时实现，在不修改内核、不依赖特殊硬件的前提下提升并发处理能力。表~\ref{tab:system_compare}给出了与相关系统的定位对比。

\subsection{论文的主要内容与结构}

\subsubsection{主要工作}

本课题围绕"基于 KV 存储的文件系统元数据管理"这一目标，完成了以下工作：

（1）设计了基于 RocksDB 多列族的元数据存储方案。将 inode 属性、目录项、增量记录分别存储在独立的列族中，使得不同类型的数据可以拥有各自的压缩策略和缓存配置。目录项列族的键遵循"以边为中心"的设计：以父 inode 号与子文件名的大端序拼接作为键，每条目录项对应目录树中一条父子边。文件操作转化为对边的增删——\texttt{create} 插入一条边，\texttt{unlink} 删除一条边，\texttt{rename} 删旧边、插新边——均为常数时间操作。大端序编码保证字节序与数值序一致，同一父目录的全部子边在键空间中自然相邻，\texttt{readdir} 和 \texttt{lookup} 均可通过前缀迭代器和点查高效完成。

（2）提出了增量更新与懒折叠（lazy folding）机制。文件创建和删除操作不直接修改父目录 inode，而是以 DeltaOp 的形式追加增量记录。\texttt{getattr} 和 \texttt{readdir} 操作在读取时将增量记录与基值在线合并返回。后台压缩线程在增量条目超过 32 条时触发折叠，将累积的变更合入基值。这种方式在高并发写入场景下减少了 KV 层面的读-修改-写次数。

（3）实现了完整的 POSIX 文件操作集合，包括 \texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rmdir}、\texttt{rename}、\texttt{readdir}、\texttt{setattr}、\texttt{read}、\texttt{write} 等。所有变更操作通过 RocksDB 的 PCC 事务（AtomicWriteBatch）保证原子性，事务冲突时以 \texttt{EAGAIN} 语义自动重试，最多 3 次。

（4）基于 Rust 的 fuser 0.15 库\cite{fuser2023crate}实现了 FUSE 客户端，采用多线程请求分派模式。挂载选项启用 \texttt{default\_permissions} 和 \texttt{allow\_other}，将 POSIX 权限检查委托给内核 VFS 层。

（5）搭建了多层次的测试框架：约 192 个单元测试覆盖核心逻辑，pjdfstest 套件验证 POSIX 兼容性，自定义的元数据基准测试（\texttt{create}/\texttt{stat}/\texttt{unlink}/\texttt{readdir} 吞吐量与延迟）用于性能对比。在同等硬件环境下，与 JuiceFS、CubeFS、NFS 进行了对比评测。

\subsubsection{论文结构}

本论文共分为五章，各章内容安排如下：

第一章为绪论。介绍了课题的研究背景，分析了将 KV 存储用于文件系统元数据面临的技术挑战，梳理了国内外相关工作的发展脉络，给出了本课题的目标和定位。

第二章为相关技术基础。介绍 FUSE 框架的工作原理与请求处理流程，RocksDB 的 LSM-tree 架构、列族与事务机制，POSIX 文件系统语义的关键约束，以及悲观并发控制（PCC）的基本概念。

第三章为系统设计与实现。阐述 RucksFS 的整体架构（五模块 workspace 设计）、元数据键编码方案、增量更新与懒折叠机制、PCC 事务封装与冲突重试策略，以及各 POSIX 操作的实现逻辑。

第四章为实验与结果分析。描述测试环境配置，展示 RucksFS 与 JuiceFS、CubeFS、NFS 在元数据操作吞吐量、延迟、POSIX 合规性方面的对比数据，并对结果进行分析和讨论。

第五章为总结与展望。回顾本课题的主要成果，讨论当前设计的局限性，并给出分布式扩展、gRPC 元数据服务和智能预取等方向的改进思路。
