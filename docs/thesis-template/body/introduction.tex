\section{绪\hspace{1em}论}

文件系统的元数据管理——包括 inode 属性的维护、目录项的组织以及文件操作的事务性保证——直接决定了存储系统在高并发场景下的吞吐上限。当传统 B+ 树引擎面对每秒数万次的小文件创建时，就地修改带来的随机 I/O 和目录锁串行化问题会迅速成为瓶颈。基于 LSM-tree 的键值存储以顺序写入取代随机写入，为这一问题提供了有竞争力的替代方案。

\subsection{课题背景、目的与意义}

\subsubsection{研究背景}

大规模模型训练、容器化微服务和云端数据湖等场景有一个共同特征：它们产生并消费海量小文件。一次分布式训练作业可能在 checkpoint 阶段写出数十亿个参数分片；Kubernetes 集群中的 Pod 在启动过程中需要创建和读取大量配置文件与临时卷。百度沧海存储团队在 Mantle\cite{li2025mantle}中给出了一组直观的数据：其云对象存储管理的文件对象数已达万亿量级，且元数据请求规模仍在逐年数倍增长。当文件数量攀升至这一量级时，系统的性能瓶颈通常不在数据传输带宽，而在元数据路径上。Ren 和 Gibson\cite{ren2013tablefs}的测量表明，\texttt{stat}、\texttt{open}、\texttt{readdir} 等元数据操作占文件系统全部 I/O 操作的 60\%--80\%。单个热点目录下的文件数一旦达到百万级，元数据查询与更新的延迟就会成为系统吞吐的天花板。

现有的本地文件系统（ext4、XFS）采用 B+ 树或其变体（如 ext4 的 H-tree）组织目录项和 inode 索引。要理解这些传统方案在高频元数据写入下的瓶颈，不妨追踪一次最简单的文件创建操作——\texttt{create("/mnt/dir/newfile")}——在 ext4 上的完整磁盘写入路径。

这样一次看似简单的操作，在 ext4 内部至少涉及 5 类磁盘区域、总计至少 6 次写入：

\begin{enumerate}
    \item \textbf{Journal 事务起始}。ext4 在 ordered 模式下首先向日志区域写入一条事务描述符块（descriptor block），标记本次操作的开始。日志区域位于分区上的固定偏移量，与用户数据不相邻。
    \item \textbf{inode bitmap}。从新文件所属的块组中查找空闲 inode 位——读取 inode 位图块（4\,KiB），翻转对应比特位，写回。位图块的磁盘位置取决于块组编号，通常与上一步的日志区域相距甚远。
    \item \textbf{inode 表}。在 inode 表中初始化新分配的 inode 条目：写入 mode、uid、gid、时间戳等共 256 字节的属性数据。inode 表位于该块组的固定偏移处，又是一次不同位置的写入。
    \item \textbf{目录数据块}。在父目录 \texttt{/mnt/dir} 的数据块中插入一条目录项（dirent）。ext4 的 H-tree 是 B+ 树的哈希变体，插入时可能需要读取并修改叶子节点的数据块，若叶节点已满还要分裂出新块。目录数据块散布在父目录所属的块组中。
    \item \textbf{父目录 inode}。更新父目录的 \texttt{mtime}（修改时间）和 \texttt{ctime}（状态变更时间），如果创建的是子目录还要递增 \texttt{nlink}（链接计数）。父目录 inode 与新文件 inode 可能不在同一个块组中。
\end{enumerate}


最后，ext4 将上述所有修改的元数据块写入 journal 区域，追加一条 commit 块，调用 \texttt{fsync} 确保日志落盘，然后才将实际数据写回原位（称为 checkpoint）。单次文件创建总共产生至少 6 次磁盘写入，分布在 inode bitmap、inode 表、目录数据块、父目录 inode 和 journal 等互不相邻的磁盘区域上——每一次都是随机 I/O。图~\ref{fig:ext4-create-path} 展示了这一过程。

问题的本质在于粒度不匹配。一次文件创建实际需要写入的有效数据不超过 300 字节（256 字节 inode + 几十字节目录项），但 ext4 的操作粒度是 4\,KiB 的磁盘页——即使只改了 256 字节的 inode，也要读出整个 4\,KiB 页、修改、写回。Journal 模式下这个页还要被写两遍（一次写 journal，一次写原位置）。如果目录的 H-tree 发生节点分裂，写入量还会翻倍。对单个操作而言这些开销可以容忍，但当每秒需要执行数万次 create 时，写放大的累积效应就不可忽视了。并发场景下问题更严重：ext4 对目录数据块的修改受目录级互斥锁（\texttt{i\_rwsem}）保护，同一目录下的所有文件创建操作必须串行通过这把锁。即使底层存储设备（如 NVMe SSD）能提供数十万 IOPS，目录级锁也会将单目录下的创建吞吐量限制在远低于硬件能力的水平上。

基于 LSM-tree（Log-Structured Merge-tree）\cite{oneil1996lsmtree}的键值（KV）存储为上述问题提供了一条不同的技术路线。KV 存储将操作粒度从 4\,KiB 的磁盘页细化到单条记录：一个目录项就是一个键值对，写入时只做一次内存追加，没有 bitmap 翻转、没有 journal 双写、没有 4\,KiB 对齐的页填充。MemTable 攒满后一次性顺序刷盘，将原本分散在多个磁盘位置的随机写归并为连续的顺序 I/O。TableFS\cite{ren2013tablefs}在 2013 年率先将全部元数据存入 LevelDB\cite{google2011leveldb}并通过 FUSE 挂载，小文件创建速度达到同期 ext4 的 2--3 倍，首次验证了这一方案的可行性。此后 RocksDB\cite{dong2021rocksdb}在 LevelDB 基础上增加了列族、事务和前缀布隆过滤器等关键特性，Apache Ozone\cite{apache2021ozone}和 Facebook Tectonic\cite{pan2021tectonic}先后在 EB 级生产环境中部署了以 RocksDB 为核心的元数据引擎。KV 元数据方案的可行性已得到充分验证，但其在键编码设计、写放大控制和事务保障等方面仍有开放的设计问题。

\textbf{问题一：树形目录到扁平键空间的映射。}
KV 引擎的数据模型是字节串到字节串的扁平映射，而文件系统的命名空间是以根目录为起点的层次化树。如何设计键编码，使得文件创建、查询和遍历都能高效完成，是首要的设计问题。

以 \texttt{create} 为例。在 ext4 中，一次文件创建需要写入 6 个分散的磁盘区域（如 1.1.1 节所述）；而在 KV 存储中，如果将每条目录项建模为一个键值对，\texttt{create} 就简化为插入一条记录——一次内存追加即可完成，不涉及 bitmap、journal 或页结构维护。这正是 KV 方案能够在写入密集场景下超越 B+ 树的根本原因。但要实现这一简化，键的编码方式至关重要。

键编码需要同时满足三类操作的需求：（1）\texttt{create} 和 \texttt{unlink} 要求插入和删除单条目录项的代价为 $O(1)$；（2）\texttt{readdir} 要求同一父目录下的所有子项在键空间中连续排列，以便通过前缀迭代器一次扫出；（3）\texttt{lookup} 要求给定父 inode 号和文件名后，能在 $O(1)$ 时间内定位到对应条目。如果直接以全路径（如 \texttt{/a/b/c}）作为键，\texttt{create} 虽然仍是 $O(1)$，但 \texttt{rename} 操作就需要递归修改整棵子树的所有键，开销随子树规模线性增长。TableFS\cite{ren2013tablefs}以“父 inode 号 + 文件名”拼接作为键，使 \texttt{create}、\texttt{lookup}、\texttt{rename} 均为 $O(1)$，但 inode 号使用原生字节序存储，导致同一目录下的条目在 LSM-tree 中未必物理相邻，无法利用前缀布隆过滤器加速 \texttt{readdir}。键编码还必须保持字节序与数值序的一致性（byte-order preserving），否则范围查询的结果将不符合预期。

\textbf{问题二：父目录更新引发的写放大。}
POSIX 语义\cite{ieee2017posix}要求目录项发生变化时同步更新父目录的 \texttt{mtime} 和 \texttt{ctime}；仅在 \texttt{mkdir} 或 \texttt{rmdir} 这类子目录增删时，父目录的 \texttt{nlink} 才会额外变化。在 KV 存储中，这意味着高频的 \texttt{create}、\texttt{unlink}、\texttt{mkdir}、\texttt{rmdir} 都会反复触碰同一条父目录 inode 记录。对于热点目录（如 \texttt{/tmp}、CI 构建输出目录、模型训练的 checkpoint 目录），假设 1 秒内发生 1{,}000 次文件创建，父目录 inode 就要被读写 1{,}000 次，其中绝大部分读取到的值在写回前就已过期。在 RocksDB 中，对同一 key 的密集 \texttt{Get+Put} 序列还会加速 MemTable 填满和 L0 文件堆积，触发额外的 compaction，形成写放大的连锁效应。

\textbf{问题三：并发目录操作的正确性。}
多个线程同时操作同一目录时，检查条件与执行操作之间的时间窗口会引发 TOCTOU（Time-of-Check-to-Time-of-Use）竞态。例如，线程 A 执行 \texttt{rename} 时确认目标路径不存在，但在写入新目录项之前，线程 B 已在该路径上创建了同名文件，导致目录项冲突或数据丢失。传统内核文件系统通过 VFS 层的 inode 互斥锁串行化目录操作来规避此问题；在 KV 引擎上没有这层保护，需要依赖存储层自身的事务机制。事务的锁粒度——锁整个目录还是只锁涉及的 key——直接影响并发度与实现复杂度之间的取舍。

\subsubsection{课题目的与意义}

本课题的目标是设计并实现一个基于 RocksDB 的用户态 FUSE 文件系统 RucksFS，针对上述三个问题给出完整的解决方案，并通过实验验证其正确性和性能。

\textbf{技术选型。}
选择 RocksDB 作为元数据存储引擎，是因为它的特性恰好对应上述三个问题的解决需求：Column Family 机制支持将不同类型的元数据（inode 属性、目录项、增量记录）隔离存储并独立配置；前缀迭代器和前缀 Bloom 过滤器为目录遍历提供了高效的前缀扫描能力；悲观事务接口（TransactionDB）提供行级锁和冲突检测，可用于封堵 TOCTOU 窗口。选择 FUSE 作为用户态接口框架，是出于工程上的务实考量——用户态开发迭代快，与 JuiceFS\cite{juicefs2021}、CubeFS\cite{liu2019cubefs}等主流系统的接入方式一致，且 FUSE 层的文件操作逻辑在本质上等价于内核 VFS 接口，必要时可平滑迁移到内核态。选择 Rust\cite{klabnik2023rust}作为实现语言，是因为其所有权和生命周期机制能在编译期排除内存安全问题，对文件系统这类可靠性要求极高的基础设施具有切实价值，同时 Rust 的 async/await 生态（tokio）与 FUSE 异步处理模式契合。

\textbf{解决思路。}
针对键编码问题，本课题采用“以边为中心”的设计——以父 inode 号与子文件名的大端序拼接作为目录项键，使文件操作转化为对边的常数时间增删。针对父目录写放大问题，引入增量更新（DeltaOp）机制——写入时只追加属性变化量，不触碰基础 inode 值，后台线程定期折叠积累的增量。针对并发正确性问题，使用 RocksDB 悲观事务将检查与修改封装在同一原子操作中，按 inode ID 排序加锁以预防死锁。这三项设计的细节将在第三章展开。

\textbf{课题意义。}
在技术层面，本课题填补了 TableFS\cite{ren2013tablefs}以来在“KV 元数据引擎 + 完整事务保障”方向上的空缺——TableFS 基于 LevelDB 且不具备事务能力，IndexFS\cite{ren2014indexfs}更强调大规模分布式扩展而非 POSIX 文件操作在嵌入式 KV 上的细粒度事务语义。RocksDB 自 6.0 版本起已提供成熟的悲观事务接口，但尚未有公开工作将其系统性地应用于文件系统元数据操作。在工程层面，RucksFS 以 Rust 编写，形成了由 FUSE 客户端、MetadataServer 和 DataServer 组成的分布式原型系统，可作为后续研究（如元数据分片、副本容错、智能预取等方向）的基础平台。

\textbf{预期交付物。}
（1）按 FUSE 客户端、元数据服务（MetadataServer）和数据服务（DataServer）解耦设计的 RucksFS 原型系统，并实现基于 gRPC 的远程通信与分离部署；（2）pjdfstest POSIX 合规性测试结果；（3）在同等硬件条件下与 JuiceFS、NFS 等系统的元数据吞吐量对比评测数据。


\subsection{国内外研究现状}

用 KV 引擎管理文件系统元数据的研究始于 2013 年，至今已积累了从单机原型到 EB 级生产系统的丰富实践。

\subsubsection{基于KV存储的元数据管理研究}

TableFS\cite{ren2013tablefs}是这一方向最早的系统性工作。2013 年，Carnegie Mellon University 的 Ren 和 Gibson 将 ext4 的目录项和 inode 信息整体存入 LevelDB，并通过 FUSE 挂载为标准文件系统。TableFS 的设计思路直截了当：以（父 inode 号, 文件名）的拼接作为 LevelDB 的键，以序列化后的 \texttt{struct stat} 加上可选的内联数据作为值。在小文件（$<$ 4~KB）场景下，TableFS 的 \texttt{create} 吞吐量是同期 ext4 的 2--3 倍，这一结果首次验证了 KV 存储用于元数据管理的可行性。但 TableFS 也暴露了若干缺陷：（1）\texttt{readdir} 操作需要扫描 LevelDB 的全部键空间并过滤出指定父目录的条目——当文件系统中的文件总数增长到百万级别时，\texttt{readdir} 的延迟急剧上升；（2）LevelDB 本身不提供任何事务语义，多线程写入时的正确性完全依赖外部加锁；（3）键编码方案中父目录 inode 号使用原生字节序存储，导致同一目录下的条目在 LSM-tree 中不一定连续排列，无法利用前缀迭代器或前缀布隆过滤器加速查找。此外，TableFS 没有考虑父目录 inode 更新带来的写放大问题。

同一研究团队在次年推出了 IndexFS\cite{ren2014indexfs}，将 TableFS 的思路扩展到分布式环境。IndexFS 在每个元数据节点上运行一个 LevelDB 实例，目录项通过 GIGA+ 动态哈希分区算法分布到不同节点。其关键贡献是引入了列式目录布局（columnar directory layout）：将 \texttt{readdir} 所需的信息（文件名列表）与 \texttt{getattr} 所需的信息（inode 属性）分开存储在不同的 SSTable 列中，使得目录遍历只需读取文件名列，跳过体积更大的 inode 属性数据。IndexFS 在 128 节点的集群上实现了每秒超过 100 万次 \texttt{create} 操作，获得了 SC'14 的最佳论文奖。不过 IndexFS 的部署要求相当苛刻：每个节点需要独立维护一个 LevelDB 实例和一套目录缓存状态机，还需要一个独立的 GIGA+ 组件来处理跨节点的目录分裂。对于中小规模的使用场景，这样的架构存在过度设计之嫌。

LocoFS\cite{li2017locofs}在 2017 年观察到文件访问的时间局部性和空间局部性，设计了一种松耦合的元数据分布策略，将热点目录的元数据根据哈希分配到固定服务器上。Xu 等人\cite{xu2014drop}在 2014 年提出了面向 EB 级规模的子树粒度元数据划分方案。这些工作侧重多节点元数据划分与扩展策略，与本课题关注的“分布式访问路径下，如何围绕 RocksDB 组织 POSIX 元数据操作”这一问题并不相同。

百度沧海·存储团队在 SOSP'25 上发表的 Mantle\cite{li2025mantle}是 KV 元数据管理这一技术路线的最新进展。Mantle 为云对象存储提供层级命名空间支持，底层持久化引擎 TafDB 采用三个列族组织元数据：AccessCF 以（父 inode 号 + 文件名）为键存储目录项，AttrCF 以 inode 号为键存储属性，DeltaCF 以（inode 号 + 时间戳）为键追加增量更新记录。其中 Delta Record 机制与本课题的设计动机直接相关：当多个客户端并发在同一目录下创建文件时，父目录的子项计数和 \texttt{mtime} 不再以读取-修改-写回的方式更新，而是各自追加一条增量记录，由后台线程异步将增量合并到 AttrCF 中的基准值。这种设计从根本上消除了热点目录属性的写竞争。Mantle 还在 TafDB 之上叠加了 IndexNode 内存缓存层，以加速长路径解析和权限检查，在百度内部的 EB 级数据湖上实现了相比旧方案 115 倍的吞吐量提升。不过 Mantle 依赖的 TafDB 是百度自研的类 Spanner 分布式事务数据库，IndexNode 与 TafDB 之间的缓存一致性协议也相当复杂——整套系统并不开源。本课题借鉴了 Mantle 的增量更新思路，但将其落到“Client + MetadataServer + DataServer”三段式分布式路径上：元数据仍由 MetadataServer 上的 RocksDB 统一维护，同时以 DeltaOp + 懒折叠降低热点目录的写竞争，用较低的系统复杂度复现其核心收益。

\subsubsection{云原生文件系统中的元数据方案}

在 KV 元数据研究逐步成熟的同时，工业界的分布式文件系统也在快速演进。本课题选择 JuiceFS 和 CubeFS 作为实验对比对象，这里简要介绍二者的元数据方案以及其他几个代表性系统的技术路线，为后续对比提供背景。

早期的分布式文件系统以集中式元数据管理为主：GFS\cite{ghemawat2003gfs}和 HDFS\cite{shvachko2010hdfs}将全部元数据驻留在单节点内存中，Ceph\cite{weil2006ceph}用动态子树划分将目录树分配到多个元数据服务器上。这些系统在 PB 级规模上运转良好，但元数据的可扩展性始终受到内存容量或子树迁移开销的制约。Apache Ozone\cite{apache2021ozone}和 Facebook Tectonic\cite{pan2021tectonic}在 2020 年前后将元数据引擎替换为 RocksDB，分别管理了 EB 级的数据，但两者都不提供完整的 POSIX 接口——Ozone 面向 S3 和 Hadoop API，Tectonic 面向 Facebook 内部的对象存储协议。

JuiceFS\cite{juicefs2021}是国内使用较广的开源 POSIX 文件系统。它的架构将元数据和数据完全分离：元数据存储在 Redis、TiKV 或 MySQL 等外部引擎中，数据块则写入 S3 兼容的对象存储。这种设计的优势在于部署灵活——用户可以根据自身条件选择最合适的后端组合。劣势则在于性能和可靠性完全取决于外部组件：Redis 的单线程模型在元数据操作超过每秒数万次时会成为瓶颈；TiKV 虽然吞吐量更高，却引入了 Raft 共识协议带来的额外延迟。JuiceFS 官方基准测试显示，Redis 后端在单客户端下的文件创建吞吐量约为 1,410 ops/s，而 MySQL 后端仅约 350 ops/s。JuiceFS 的 FUSE 客户端使用 Go 语言实现，Go 运行时的垃圾回收（GC）暂停在极端负载下曾被部分用户报告为延迟抖动的来源之一。

CubeFS\cite{liu2019cubefs}（最初名为 CFS）于 2019 年在 SIGMOD 上发表，后捐赠给 CNCF 并于 2024 年成为毕业项目。与 JuiceFS 依赖外部数据库不同，CubeFS 自带元数据引擎：每个元数据分区（MetaPartition）使用内存中的 B-tree 维护目录结构，多副本之间通过 Multi-Raft 协议保持一致性。CubeFS 的元数据在读路径上表现出色（内存 B-tree 的查找延迟在微秒级），但写路径受到 Raft 日志同步的制约——每次写操作至少需要等待多数副本确认，典型延迟在毫秒级。内存中的 B-tree 也限制了单节点可管理的元数据量：按每个 inode 约 300 字节内存估算，十亿文件需要约 280~GB 内存。

上述系统的元数据管理方案按技术路线可归为三类：内存驻留型（GFS、HDFS、CubeFS）、外部数据库型（JuiceFS）、嵌入式 KV 型（Ozone、Tectonic）。三类方案各有侧重，一个共同趋势是围绕分布式环境组织元数据服务：要么把一致性和事务交给外部数据库，要么把复制、分片和故障恢复做进文件系统自身。这也带来一个更具体的问题：在已经采用分布式部署的前提下，元数据引擎本身该如何围绕文件操作做定制，而不是完全受制于通用数据库或共识框架的语义与开销。

\subsubsection{元数据引擎的深度优化研究}

与单纯追求分片扩展不同，另一条研究路线更关注元数据引擎本身的执行效率，即先缩短单次操作的关键路径，再讨论系统如何扩展。SingularFS\cite{li2022singularfs}的成果表明，这一路线的性能上限远高于通常的直觉判断。

SingularFS 在 ATC'23 上展示了一个激进的设计选择：仅用单个元数据服务器支撑十亿级文件规模的分布式文件系统。其核心优化包括三个方面：（1）无日志元数据操作（log-free metadata operations），通过精心设计的持久化顺序消除崩溃一致性日志的开销；（2）层次化并发控制，对目录树的不同层级使用不同粒度的锁，避免粗粒度锁的串行化瓶颈；（3）NUMA 感知的 inode 分区，将 inode 地址空间按 NUMA 节点划分，减少跨节点内存访问。在配备 RDMA 网络和持久内存的硬件环境下，SingularFS 的单节点元数据吞吐量远超传统多节点方案。这一结果对本课题的启发不在于“把系统做成单机”，而在于即便系统整体以分布式方式部署，元数据服务内部仍值得围绕关键路径做深度优化。RucksFS 当前正是沿着这条思路，在 MetadataServer 的 RocksDB 事务、增量更新和键编码上做针对性设计，而不是把全部语义都外包给通用分布式 KV。

Wang 等人\cite{wang2023cfs}在 EuroSys'23 上发表的 CFS 对元数据服务中临界区（critical section）的范围做了精细化分析。传统方案在处理 \texttt{rename} 等涉及多个目录的操作时，往往采用粗粒度的目录级锁，导致不相关的操作被串行化。CFS 通过形式化方法识别出各个 POSIX 操作中真正需要互斥的 key 集合，将临界区缩减到最小。实验表明，这种优化将 \texttt{rename} 密集型工作负载的吞吐量提升了 3.2$\times$。本课题的 PCC 事务设计借鉴了类似思路——只对实际涉及的 key 加排他锁，而非锁定整个目录，从而在保证正确性的同时保留并发度。

Yang 等人\cite{yang2020batchfile}在 2020 年从实用角度分析了大规模目录中逐个执行 \texttt{stat} 和 \texttt{open} 效率低下的问题，设计了将 \texttt{readdir}+\texttt{stat} 复合操作在内核内合并执行的批量接口，减少了用户态/内核态切换次数。这一批量化思路直接影响了 RucksFS 的 \texttt{readdirplus} 实现——在一次 FUSE 调用中同时返回目录项和完整的 inode 属性，避免后续逐个 \texttt{getattr} 的往返开销。

操作系统层面的优化同样值得关注。FetchBPF\cite{mishler2024fetchbpf}发表于 ATC'24，利用 eBPF 在 Linux 内核中实现可定制的文件访问预取策略。用户编写的 eBPF 程序由内核在 I/O 路径中直接执行，避免了用户态预取方案的上下文切换开销，在顺序读工作负载上接近理论带宽。同期的 FBMM\cite{tabatabai2024fbmm}则展示了将 Linux VFS 接口用于非传统目的的可能性——将内存管理器实现为文件系统，利用 VFS 的可扩展性在不修改内核核心代码的情况下引入新策略。这种“文件系统即抽象层”的理念与本课题使用 FUSE 将 KV 存储暴露为 POSIX 接口的思路有异曲同工之处。FetchBPF 所展示的 eBPF 预取能力也为 RucksFS 未来的优化方向提供了参考——在 FUSE 的 \texttt{/dev/fuse} 通道上通过 eBPF 预取即将被访问的元数据，有可能进一步降低用户态守护进程的响应延迟。

MetaHive\cite{heidari2024metahive}在 2024 年针对 KV 存储引擎在元数据场景下的缓存效率问题展开研究。通用 KV 存储的缓存替换策略（如 LRU）未考虑文件系统的访问局部性——同一目录下的 inode 被连续访问的概率远高于随机 key。MetaHive 据此设计了元数据感知的缓存分区方案。这一发现与 RucksFS 的 InodeFoldedCache 设计动机一致：系统在应用层维护了专门的 LRU 缓存来存储折叠后的 inode 值，而非依赖 RocksDB 的通用块缓存。

在国内，龚拓宇\cite{gong2020rocksmeta}在 2020 年的清华大学硕士论文中结合 RDMA 技术与 RocksDB 设计了分布式元数据服务原型 RocksMeta，在单节点上实现了超过 10 万 IOPS 的元数据操作吞吐量。其键编码方案与本课题有相似之处，但侧重网络传输优化（RDMA 旁路内核协议栈），而本课题侧重存储层的写放大优化。范立衡等人\cite{fan2014kvmetadata}较早探索了键值存储用于元数据集群副本一致性的管理策略。吴雨飞等人\cite{wu2025dfscache}在 2025 年针对基于 TiKV 的分布式文件系统元数据缓存一致性问题，提出了并发广播与惰性删除相结合的轻量级维护机制。

\subsubsection{现有研究不足与本课题定位}

将上述工作放在一起审视，有几个问题尚未被充分回答。这些问题并非都是研究界完全忽视的盲点，
而更像是“部分被解决、但没有在分布式访问路径 + POSIX + 嵌入式 KV 的完整组合下被系统性回答”的空白。
具体可以归纳为三点。

第一，事务与并发控制薄弱。TableFS\cite{ren2013tablefs}所用的 LevelDB 没有任何事务语义，
并发写入只能靠粗粒度互斥锁兜底，这在单用户基准测试下没什么问题，
一旦移植到真实多进程负载（make 并行构建、rsync 同步、并发日志写入）就会暴露出竞态和一致性问题。
IndexFS\cite{ren2014indexfs}引入了 WriteBatch 保证单次操作的原子性，但 \texttt{rename}
需要同时修改源目录和目标目录的多条记录，WriteBatch 无法覆盖跨操作的一致性约束——
跨目录 rename 在 IndexFS 中仍需在上层自行加锁。
RocksDB 从 6.0 版本起便提供了成熟的悲观事务（PessimisticTransactionDB），
支持行级锁、\texttt{GetForUpdate} 和冲突检测，且借助大端键编码的前缀局部性可以把冲突范围限制在单个父目录内；
然而目前尚未见到将其系统性地应用于文件系统元数据操作的公开工作。

第二，父目录 inode 的写放大问题长期被忽视。
TableFS、JuiceFS、CubeFS 在处理目录变更时普遍采用读取-修改-写回模式——
普通文件的 \texttt{create}、\texttt{unlink} 需要更新父目录的 \texttt{mtime}、\texttt{ctime}，而 \texttt{mkdir}、\texttt{rmdir} 还会额外影响父目录的 \texttt{nlink}；这些字段往往需要被完整读出、修改、再写回。
在热点目录（例如存放数十万文件的临时目录）上，这种模式会制造明显的写竞争：
所有写入串行化到同一条父 inode 记录上，LSM-Tree 的优势无从发挥。
Mantle\cite{li2025mantle}的 Delta Record 机制是目前唯一在生产环境中系统性解决这一问题的方案，
但其依赖的 TafDB 分布式事务数据库和 IndexNode 缓存一致性协议使得方案难以直接迁移到本课题这种更轻量的三段式架构。
如何在 MetadataServer 本地的 RocksDB 上以较低复杂度实现类似的增量更新语义，
并保证 POSIX 操作的原子性和读取一致性，仍需探索。

第三，现有系统偏重通用分布式可扩展性，对专用元数据服务器的操作路径优化关注不足。
IndexFS、CubeFS、Tectonic 引入了 Raft 共识、一致性哈希、分区迁移等重型机制，
这些机制在 EB 级规模下是必要的基础设施，但它们也会把本可在专用元数据服务器内完成的简单目录操作拉长成更重的事务路径。
SingularFS 的实验数据已经表明，元数据引擎经过深度优化后的吞吐量，
可以远超很多传统方案——学术界更倾向于发表大规模分布式系统，
而“分布式部署 + 专用元数据引擎 + POSIX 合规”这一组合下的实现细节仍缺少公开、完整的原型验证。
本课题所处的毕业设计场景，正是这样一种需要真实分布式访问路径、但又希望控制系统复杂度的典型代表。

综合以上分析，本课题定位于：构建一个基于 RocksDB 的分布式文件系统原型（RucksFS），
由 FUSE Client、MetadataServer 和 DataServer 组成，通过 gRPC 连接客户端与服务端；其中元数据面以 RocksDB 为基础，利用多列族、悲观事务和前缀迭代器等特性，设计适配文件系统访问模式的键编码方案与增量更新机制。
与已有工作的差异集中在三点：
（1）系统性地使用 PCC 事务为所有目录变更操作提供原子性保证，
而不是依赖外部锁或仅用 WriteBatch；
（2）借鉴 Mantle 的 Delta Record 思路，通过 DeltaOp 增量日志与懒折叠解决父目录 inode 的写放大问题，
但将其约束在 MetadataServer 本地 RocksDB 内完成，不依赖外部通用分布式事务数据库和复杂缓存一致性协议；
（3）以 Rust 编写，基于 FUSE 异步运行时和 gRPC 通信实现端到端分布式访问路径，
在不修改内核、不依赖特殊硬件的前提下验证该架构下的元数据语义与性能表现。

\subsection{论文的主要内容与结构}

\subsubsection{主要工作}

本课题围绕“基于 KV 存储的文件系统元数据管理”这一目标，完成了以下工作：

（1）设计了基于 RocksDB 多列族的元数据存储方案。将 inode 属性、目录项、增量记录分别存储在独立的列族中，使得不同类型的数据可以拥有各自的压缩策略和缓存配置。目录项列族的键遵循“以边为中心”的设计：以父 inode 号与子文件名的大端序拼接作为键，每条目录项对应目录树中一条父子边。文件操作转化为对边的增删——\texttt{create} 插入一条边，\texttt{unlink} 删除一条边，\texttt{rename} 删旧边、插新边——均为常数时间操作。大端序编码保证字节序与数值序一致，同一父目录的全部子边在键空间中自然相邻，\texttt{readdir} 和 \texttt{lookup} 均可通过前缀迭代器和点查高效完成。

（2）提出了增量更新与懒折叠（lazy folding）机制。文件创建和删除操作不直接修改父目录 inode，而是以 DeltaOp 的形式追加增量记录。\texttt{getattr} 和 \texttt{readdir} 操作在读取时将增量记录与基值在线合并返回。后台压缩线程在增量条目超过 32 条时触发折叠，将累积的变更合入基值。这种方式在高并发写入场景下减少了 KV 层面的读-修改-写次数。

（3）实现了完整的 POSIX 文件操作集合，包括 \texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rmdir}、\texttt{rename}、\texttt{readdir}、\texttt{setattr}、\texttt{read}、\texttt{write} 等。所有变更操作通过 RocksDB 悲观事务完成加锁与冲突检测，并以原子提交保证多键更新的一致性；事务冲突时，MetadataServer 会先自动重试，最多 3 次，若仍失败则在 FUSE 层映射为 \texttt{EAGAIN}。

（4）基于 Rust 的 fuse3 库（版本 0.8，集成 tokio 异步运行时）实现了 FUSE 客户端，采用异步请求处理模式。挂载选项启用 \texttt{default\_permissions} 和 \texttt{allow\_other}，将 POSIX 权限检查委托给内核 VFS 层。

（5）搭建了多层次的测试框架：约 192 个单元测试覆盖核心逻辑，pjdfstest 套件验证 POSIX 兼容性，自定义的元数据基准测试（\texttt{create}/\texttt{stat}/\texttt{unlink}/\texttt{readdir} 吞吐量与延迟）用于性能对比。在同等硬件环境下，与 JuiceFS、NFS 等系统进行了对比评测。

\subsubsection{论文结构}

本论文共分为五章，各章内容安排如下：

第一章为绪论。介绍了课题的研究背景，分析了将 KV 存储用于文件系统元数据面临的技术挑战，梳理了国内外相关工作的发展脉络，给出了本课题的目标和定位。

第二章为相关技术基础。介绍 FUSE 框架的工作原理与请求处理流程，RocksDB 的 LSM-tree 架构、列族与事务机制，POSIX 文件系统语义的关键约束，以及悲观并发控制（PCC）的基本概念。

第三章为系统设计与实现。阐述 RucksFS 的整体架构（Client、MetadataServer、DataServer 及其七个 crate 的 workspace 模块划分）、元数据键编码方案、增量更新与懒折叠机制、PCC 事务封装与冲突重试策略，以及各 POSIX 操作的实现逻辑。

第四章为实验与结果分析。描述测试环境配置，展示 RucksFS 与 JuiceFS、NFS 等系统在元数据操作吞吐量、延迟、POSIX 合规性方面的对比数据，并对结果进行分析和讨论。

第五章为总结与展望。回顾本课题的主要成果，讨论当前设计的局限性，并给出元数据分片、副本容错、RPC 路径优化和智能预取等方向的改进思路。
