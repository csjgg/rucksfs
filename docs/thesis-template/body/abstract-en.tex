\abstracten

In distributed storage and cloud file services, metadata handling often becomes the system bottleneck as data volume and concurrency grow. Traditional metadata management commonly relies on page-based index structures represented by B+ trees. Under heavy metadata operations, random writes, write amplification, and hot-directory contention limit throughput growth. If metadata is moved to a Key-Value store built on a Log-Structured Merge-tree (LSM-tree), and key file-operation flows are redesigned accordingly, updates can be managed at record granularity and random writes can be reorganized into sequential batched writes, providing a practical path to mitigate metadata bottlenecks.

Following this direction, RucksFS is built with Key-Value metadata storage and core file operations implemented end to end. The architecture separates metadata management from file-data management through three components: a metadata server, a data server, and a client. The metadata server manages inode attributes and directory-entry indexes on top of RocksDB, with data organized into column families such as inodes and dir\_entries. Pessimistic Concurrency Control (PCC) transactions are used to preserve atomicity and consistency of critical operations. To address shared-parent-directory hot spots, DeltaOp introduces delta updates with lazy folding, reducing in-place inode rewrites and lock contention. The data server manages file data and persistence, providing read, write, truncate, delete, and flush interfaces. The client handles major file operations and cross-service orchestration, receives kernel requests through Filesystem in Userspace (FUSE), provides Portable Operating System Interface (POSIX) semantics to applications, and routes requests to backend servers. Overall, the implementation integrates Key-Value metadata storage, concurrency control, and core file operations into a verifiable engineering system.

For correctness, pjdfstest is used, and all 182 executable cases within the implemented interface scope passed. For performance, mdtest is run on a cluster with one 64-core server and multiple clients. With 32 concurrent clients, RucksFS improves create throughput by about 4.6$\times$ over NFS v4.2 on ext4. Under higher concurrency, DeltaOp is further validated: with concurrency 384, create and remove throughput reach 2.96$\times$ and 2.63$\times$ of the build without DeltaOp. These results show that the proposed design improves scalability of metadata-intensive operations while preserving semantic correctness.
