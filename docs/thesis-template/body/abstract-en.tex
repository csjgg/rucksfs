\abstracten

Metadata operations---file creation, deletion, renaming, and the like---typically account for 60\%--80\% of file system I/O, which makes the organisation of metadata a direct determinant of overall throughput. Traditional file systems manage directory entries and inode indexes with B+ trees at page granularity: even a single-entry change forces a page read, modification, and write-back. Under high-concurrency small-file workloads, this in-place update is further compounded by directory-level lock contention, so write amplification and concurrency bottlenecks arise at the same time. Key-Value (KV) storage lowers the operational granularity from a page to a single record---each directory entry is stored as an independent key-value pair and an insertion or deletion reduces to one write, eliminating the need to maintain page-level structures. On top of this, KV engines based on the Log-Structured Merge-tree (LSM-tree) aggregate scattered small writes in an in-memory MemTable and, once it fills, flush them to disk in sequential batches, turning the random small writes common in metadata workloads into sequential ones.

Building on this idea, this work designs and implements RucksFS, a user-space file system backed by RocksDB and exposed as a POSIX mount via FUSE, covering the common file operations \texttt{create}, \texttt{mkdir}, \texttt{unlink}, \texttt{rmdir}, \texttt{rename}, \texttt{readdir}, \texttt{setattr}, \texttt{read} and \texttt{write}. Implementing these operations over KV storage requires answering three design questions: how to model metadata as key-value structures, how to control write amplification on parent directories, and how to guarantee semantic correctness under concurrent access.

For storage modelling, the system partitions data across Column Families (CFs): inode attributes and directory entries occupy separate CFs, each tuned with its own Bloom filter and compaction policy. Keys in the directory-entry CF are formed by concatenating the parent inode number and the child filename in big-endian order, so every directory entry corresponds to an edge in the directory tree. File operations thus reduce to edge operations---\texttt{create} inserts an edge, \texttt{unlink} removes one, and \texttt{rename} deletes the old edge before inserting a new one---all at constant cost. By comparison, schemes that key by full path must rewrite every key in the subtree on \texttt{rename}, with cost scaling linearly in subtree size. Moreover, all child edges of the same parent are adjacent in key space, so directory traversal and single-entry lookups can be served by prefix scan and point query, respectively.

For write amplification, POSIX semantics require that \texttt{create} and \texttt{unlink} synchronously update the parent directory's timestamps and link count. Under high concurrency the same parent repeatedly becomes a write hot spot, and staying with read-modify-write makes long lock waits hard to avoid. To address this, a delta update and lazy folding mechanism (DeltaOp) is introduced: each write appends a small delta record without touching the base inode value, converting in-place rewrites into append writes; each read merges the base value and outstanding deltas on the fly so that callers always observe up-to-date attributes; a background thread periodically scans accumulated deltas and folds them back into the base value once a threshold is reached, keeping read-path overhead bounded while eliminating write contention on the parent directory.

For concurrency safety, when multiple threads execute \texttt{create}, \texttt{rmdir}, or \texttt{rename} on the same directory, a race window exists between the check and the modification: \texttt{create} must confirm the absence of a duplicate before inserting, \texttt{rmdir} must confirm that the directory is empty before deleting, and \texttt{rename} involves multi-key atomic modifications across directories. RocksDB Pessimistic Concurrency Control (PCC) transactions lock the involved keys one by one, wrapping both checks and mutations inside a single atomic operation and thereby eliminating these races.

A full pjdfstest run in a distributed deployment (client and MetadataServer on separate nodes) executes all 398 cases. Of these, 167 depend on \texttt{mknod} and \texttt{mkfifo}, which this work deliberately leaves unimplemented; 1 case exceeds the DataServer's current 64\,MiB per-file development limit; a further 48 are skipped by the pjdfstest harness itself (e.g., \texttt{utimensat}, \texttt{remount}), unrelated to the metadata design. The remaining 182 cases constitute the in-scope set for this work and all pass---a 100\% pass rate within scope.

For performance, the test cluster consists of one 64-core server and up to 96 two-core clients, driven by mdtest. Under $N \in \{2, 8, 32\}$, RucksFS is compared head-to-head with NFS v4.2 and JuiceFS+TiKV. Against JuiceFS+TiKV, RucksFS is roughly 1.5--1.7$\times$, 1.8--2.2$\times$, and 1.2$\times$ faster on \texttt{create}, \texttt{remove}, and \texttt{stat} respectively, reflecting two effects: a single-node direct-to-RocksDB design does not pay the cost of TiKV's Raft and two-phase commit, and DeltaOp relieves write contention on shared parent directories. Against NFS v4.2, at $N=2$ RucksFS is slightly slower because each \texttt{create}, with attribute cache disabled, now takes two extra lookup RPCs; at $N=32$, NFS is bounded by ext4's \texttt{i\_rwsem}, and RucksFS overtakes it on \texttt{create} by roughly 4.6$\times$.

At $N \geq 64$, neither NFS nor JuiceFS+TiKV sustains reliable numbers. In this range the comparison shifts to an internal one between the Delta and NoDelta builds of RucksFS, sweeping $\mathrm{np} = N \times r$. Once $\mathrm{np}$ exceeds 96, NoDelta cannot continue to scale due to repeated transaction conflicts and its throughput stabilises around 12{,}000\,ops/s, whereas Delta continues to scale to 38{,}000\,ops/s, with \texttt{create} and \texttt{remove} ratios reaching 2.96$\times$ and 2.63$\times$ respectively at $\mathrm{np}=384$. Under easy mode, in which each process uses an independent parent directory, the two variants are indistinguishable, confirming that the gain originates from eliminating write conflicts on shared parent directories.
