\section{元数据组织与事务模型}
\label{sec:design}

RucksFS 是一个用 Rust 实现的分布式文件系统原型。系统由三部分组成：FUSE 客户端提供 POSIX 入口，MetadataServer 负责命名空间和 inode 元数据，DataServer 负责文件数据的读写；三者之间通过 gRPC 通信。其中 MetadataServer 承担了元数据路径上的关键设计，本章先把这部分内部结构讲清楚；客户端如何在 MetadataServer 和 DataServer 之间协调、DataServer 的数据路径如何组织、FUSE 层的可见语义与错误处理，则统一放到第~\ref{sec:impl}~章展开。

本章围绕三件事：目录项如何映射到 RocksDB 键空间，目录修改在并发下如何保持正确性，以及核心 POSIX 操作如何在这套键编码和事务机制上实现。

\subsection{设计目标}

在进入具体实现之前，需要先把几项从一开始就希望系统具备的性质确定下来，它们会在后续章节反复影响具体决策。

接口层面保留标准的 POSIX 路径。应用程序通过普通的 \texttt{open()}、\texttt{rename()}、\texttt{unlink()} 等系统调用访问挂载点，不需要为 RucksFS 改写任何代码。选择 FUSE 作为对外接口层的直接原因即在于此。

服务层面将元数据与文件数据显式分离。二者的访问模式并不相同：元数据多为小粒度随机写入，且与事务、并发控制紧密相关；文件数据以顺序 I/O 为主，读写路径更短。在服务边界上将两者拆开，既让各自的实现可以按自身特点组织，也使本文真正想研究的元数据路径不被数据块管理的细节干扰。

并发正确性方面需要经得起检验。名字唯一、目录非空判断、跨目录重命名的原子性、延迟删除等语义不能依赖“多数情况下不会冲突”这种经验假设，需要用事务边界把它们固定下来。这一点是采用 RocksDB 悲观事务的直接动机。

性能瓶颈方面，本文希望把可能成为瓶颈的点集中到少数关键路径上——目录项查找、事务提交与父目录属性更新。这几条路径之所以值得单独优化，是因为其行为直接决定了高并发下的吞吐；其他路径即便暂未细致调优，也不会影响整体结论。

\subsection{系统整体架构}
\label{sec:architecture}

RucksFS 的运行时由三个角色组成：部署在客户端机器上的 FUSE 客户端、负责命名空间和 inode 状态的 MetadataServer，以及负责文件内容读写的 DataServer。客户端收到内核的 FUSE 请求后，并非统一转发到同一个后端，而是先经过本地的 \texttt{VfsCore} 路由层，再按请求类型决定交给元数据面还是数据面。图~\ref{fig:overall_arch} 给出了整体的请求路径。

从请求走向来看，这种拆分对应三类路径。

第一类是元数据路径。\texttt{lookup}、\texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rename}、\texttt{readdir} 等请求经 FUSE 客户端进入 \texttt{VfsCore}，随后被转交给 MetadataServer，再由其访问 RocksDB。

第二类是数据路径。\texttt{read}、\texttt{write}、\texttt{truncate}、\texttt{flush} 等请求由 \texttt{VfsCore} 选定对应的 DataServer，交由原始磁盘后端执行。

第三类是协调路径。有些操作天然跨越元数据面和数据面。例如客户端写完数据后需要通知 MetadataServer 更新文件大小和 \texttt{mtime}；又例如 \texttt{unlink} 或覆盖式 \texttt{rename} 完成后，MetadataServer 会在响应结构中告知客户端哪些 inode 需要在 DataServer 侧清理数据。

客户端不直接持有 RocksDB，也不直接管理原始磁盘布局，只负责在 FUSE 请求、MetadataServer、DataServer 三者之间完成调用串联和结果汇总。这样做的直接好处是，文件系统的正确性主要集中在元数据事务设计和客户端协调逻辑内，模块边界清楚，出现问题时也更容易定位到具体的服务。

整体布局确定之后，本章后续几节按“元数据在 RocksDB 键空间中如何组织”、“事务模型如何保证并发正确性”、“核心 POSIX 操作如何映射到键空间和事务上”的顺序展开；“客户端如何协调元数据面和数据面”这一部分留到第~\ref{sec:impl}~章。

\subsection{元数据模型与存储布局}
\label{sec:datamodel}

本节先给出面向实现的元数据整体布局，再分别展开列族划分、键编码和值编码方面的细节。整体布局部分只回答三个问题：系统保存哪些对象，这些对象分别落在哪个列族，以及它们的 key/value 编码是什么样子。至于为何这样划分、这种编码对各类操作意味着什么，留到后面的小节展开。

\subsubsection{元数据方案总览}

RucksFS 以一棵目录树作为命名空间的抽象模型。用户看到的是按目录组织的层次结构，但真正需要持久化的核心对象只有两类：inode 和 dentry（目录项）。inode 代表文件系统对象本体，保存类型、权限、大小、链接计数、时间戳等属性；dentry 代表目录树上一条父子边，记录“某个父目录下的某个名字指向哪个 inode”。普通文件、目录、符号链接都先由 inode 标识，再通过 dentry 挂接到目录树中。

由此，路径本身不是单独保存的对象，而是通过一系列 dentry 逐级串联得到。以 \texttt{/docs/paper.tex} 为例，先在根目录下找 \texttt{docs} 对应的 dentry，取得目录 inode；再在该目录下找 \texttt{paper.tex} 对应的 dentry，最终定位到目标 inode。围绕这两个核心对象，RucksFS 在 RocksDB 中分别组织 inode 记录和 dentry 记录，并补充数据位置、属性增量和系统状态等辅助元数据。图~\ref{fig:cf_kv_layout} 给出这一方案的整体视图。

\subsubsection{列族划分与设计考量}

有了上述的整体布局，下一个问题是：为何这些对象要如此分组，而不是全部放入一个统一键空间？本文的做法是按访问模式将 RocksDB 实例拆分为 4 个列族——\texttt{inodes}、\texttt{dir\_entries}、\texttt{delta\_entries}、\texttt{system}——分别承担 inode 点查、目录项存储、属性增量追加、系统状态保存这几类不同任务。

\texttt{inodes} 列族承担“对象本体”的点查负载。inode 基础属性、\texttt{DataLocation} 和符号链接目标在语义上并不完全同类，但访问模式一致——都是按 inode 直接定位的点查——因此放在同一列族中。这样就无需为少量辅助元数据额外维护一套 MemTable 和压缩状态。

\texttt{dir\_entries} 列族承担命名空间的边关系。\texttt{lookup} 需要按“父 inode + 名字”定位单条目录项，\texttt{readdir} 需要按父目录枚举全部子项，这两种访问都依赖前缀局部性。因此该列族启用了 9 字节的前缀提取器，让同一父目录下的目录项按前缀连续排列。相比“目录项和 inode 混在同一列族”，这种方式更有利于目录遍历和前缀 Bloom 过滤器生效；相比“使用完整路径作为键”，又避免了跨目录 \texttt{rename} 对整棵子树的改写。

\texttt{delta\_entries} 列族承担热点目录的增量写入。其目标不是长期保存完整状态，而是承接高频的 \texttt{mtime}/\texttt{ctime}/\texttt{nlink} 变化，避免每次目录修改都重写父目录 inode 基值。由于该列族中的条目短小、生命周期也短，并且会被后台折叠线程定期消费，这里刻意关闭了压缩，以省去热路径上的编解码开销。

\texttt{system} 列族仅存放 inode 分配器等低频系统元信息。单独隔离是为了避免它与热点目录项共用一套压缩和缓存行为。访问频率虽然远低于其他列族，但它仍需与普通元数据一起走原子提交，因此留在同一个 RocksDB 实例内，不额外引入外部存储。

综上，这套列族划分按访问模式组织布局，而不是按语义模块简单切分。列族本身也是元数据关键路径优化的一部分，因此先把“为什么这么分”讲清楚，再讨论“每类记录如何编码”。

这种设计遵循的是“以边为中心”的命名空间建模思路。目录树中的每一条父子边，都对应 \texttt{dir\_entries} 列族中的一条记录。\texttt{lookup(parent, name)} 退化为点查一条目录项键；\texttt{readdir(parent)} 退化为扫描同一前缀下的全部目录项键；\texttt{rename} 则退化为删除旧边、插入新边。与使用全路径作为键相比，这种建模方式避免了跨目录重命名时对子树的全量重写。第~\ref{sec:pjdfstest}~节中 pjdfstest 的 \texttt{rename}、\texttt{readdir}、\texttt{lookup} 用例将直接检验这一建模方式的正确性。

\textbf{与 TableFS 键编码方案的对比。} 最早将 KV 存储应用于文件系统元数据的 TableFS\cite{ren2013tablefs}同样采用“父 inode + 文件名”作为目录项键。但 TableFS 使用原生字节序编码 inode 编号，导致同一父目录下的所有目录项在 LSM-tree 的键空间中不一定物理相邻——因为 LevelDB 按字节序比较键，inode 编号 256 的字节序表示为 \texttt{\textbackslash x01\textbackslash x00\textbackslash x00\textbackslash x01}，inode 编号 257 表示为 \texttt{\textbackslash x01\textbackslash x00\textbackslash x00\textbackslash x02}，两者之间可能隔着大量其他 inode 的键。RucksFS 采用大端序编码，使 inode 编号的大小顺序与字典序一致，同一父目录下的所有子项天然在键空间中连续排列。这一差异在 \texttt{readdir} 场景下尤为关键：TableFS 需要在扫描时额外过滤父目录编号，而 RucksFS 只需一次前缀迭代即可获得全部目录项。此外，大端序也使得前缀 Bloom 过滤器在同一父目录内的假阳性率更低。

\subsubsection{Value 编码与定长序列化}

在实现层，inode 的基础属性由 \texttt{FileAttr} 抽象表示，覆盖 inode 编号、文件大小、模式位、链接计数以及访问、修改和状态变更时间戳这几项上层接口最常使用的字段。MetadataServer 在 RocksDB 中并不使用通用序列化框架，而是按手工布局对这些字段进行定长编码：\texttt{InodeValue} 共 57 字节，首字节保留为格式版本号，之后依次是 inode 编号、文件大小、模式位、链接计数和三类时间戳。定长布局有两点直接好处：读写路径只需按偏移切片，无需承担变长解码开销；调试时也可以直接从 RocksDB 中观察二进制布局，问题定位更直接。代价是模式演进需要显式处理——若未来确实需要扩展字段，可以借首字节的版本号引入兼容逻辑，而不必打破既有键空间。

目录项与增量记录的 value 另作设计。目录项 value 固定为 12 字节的 \texttt{[child\_inode: u64 BE][child\_mode: u32 BE]}，把 \texttt{mode} 和子 inode 编号放在一起，使 \texttt{readdir} 在拿到目录项时即可判断子对象类型，无需再回查一次 inode。增量 value 采用短小的 \texttt{DeltaOp} 编码，以使热点目录上的属性更新尽可能表现为小对象追加写。

\subsection{事务模型与元数据一致性}
\label{sec:txn_model}

\subsubsection{事务边界与原子提交}

RucksFS 的大多数元数据操作都不是单键更新。以 \texttt{create} 为例，它至少需要同时修改目录项、目标 inode、数据位置以及父目录属性；\texttt{rename} 往往同时涉及源目录、目标目录、被移动对象和潜在被覆盖对象。因此事务模型首先要回答的不是“如何加锁”，而是“哪些修改必须作为一个不可分割的整体提交”。

本文的做法是不将 RocksDB 原生事务接口直接散落在业务代码中调用，而是通过 \texttt{StorageBundle} 和 \texttt{AtomicWriteBatch} 将跨列族写入封装起来。调用方先使用 \texttt{StorageBundle::begin\_write()} 开启一个事务上下文，向其中追加 \texttt{BatchOp}，最后统一提交。这样做的好处是，上层逻辑可以围绕“目录项是否存在”、“应该写哪些 inode 记录”、“父目录需要追加哪些增量”来组织动作，无需显式处理不同列族的底层细节。提交时底层最终会调用 RocksDB 事务的 \texttt{commit()}，把跨 \texttt{inodes}、\texttt{dir\_entries}、\texttt{delta\_entries} 的修改作为一个原子单元持久化。

\subsubsection{悲观并发控制与冲突处理}

仅有原子提交还不够，并发目录操作的正确性同样需要处理。这里使用的是 RocksDB TransactionDB 的悲观并发控制（PCC）接口，关键操作是 \texttt{get\_for\_update}：事务在读取目录项键或 inode 键的同时获取排他锁，使“检查条件”和“执行修改”落在同一个事务视图内。对文件系统而言这一点较为重要——\texttt{create}、\texttt{rmdir}、\texttt{rename} 都存在典型的 TOCTOU 风险，如果先在事务外检查、再进入事务修改，就会留下竞态窗口。

多对象操作还会带来死锁。本文的做法是：凡可能涉及多个 inode 的事务，都先将相关 inode 编号收集并排序，再按升序逐个加锁。\texttt{rename} 是最典型的例子——源目录、目标目录、被移动对象、潜在被覆盖对象都可能同时进入同一个事务。统一的加锁顺序将“先锁谁、后锁谁”确定为一条规则，不同线程就不会因加锁顺序不一致而形成循环等待。当前实现还为锁等待设置了 5 秒超时；若事务冲突一时无法消解，MetadataServer 会先在服务端有限重试，最多 3 次，并采用指数退避；重试仍失败时，才以 \texttt{TransactionConflict} 向上返回，在 FUSE 层映射为 \texttt{EAGAIN}。

\subsubsection{增量更新与读写一致性}

事务并不能解决所有热点问题。目录修改中还有一个较为隐蔽的瓶颈：父目录属性更新。按 POSIX 语义，\texttt{create}、\texttt{unlink}、\texttt{mkdir}、\texttt{rmdir} 都会动到父目录的 \texttt{mtime}/\texttt{ctime}，涉及子目录的操作还会动 \texttt{nlink}。如果每次都直接读出父目录 inode 再写回，热点目录下的冲突就会集中到同一条基值记录上，LSM-tree 的顺序追加优势也无法体现。

本文的做法是把这类更新从“基值重写”改为“增量追加”。写路径不直接修改父目录在 \texttt{inodes} 中的基础值，而是把变化编码为一条 \texttt{DeltaOp}，追加到 \texttt{delta\_entries} 中。目前的增量类型有 \texttt{IncrementNlink}、\texttt{SetMtime}、\texttt{SetCtime}、\texttt{SetAtime}。普通文件的创建和删除只影响父目录时间戳；\texttt{mkdir}、\texttt{rmdir} 和跨目录移动目录时会额外引入 \texttt{nlink} 增量。这样热点目录上的高频更新就从“反复覆盖同一条 inode 基值”变为“向独立的增量记录追加”。

读取路径需要负责把“基础值 + 增量”重新合成为最新视图。MetadataServer 在加载 inode 时，优先从 \texttt{InodeFoldedCache} 读取已经折叠好的结果；缓存未命中则先读取 \texttt{inodes} 中的基础值，再扫描属于该 inode 的 \texttt{DeltaOp}，按顺序合并。为了避免读路径的折叠成本无限增长，后台的 \texttt{DeltaCompactionWorker} 会周期性检查 dirty inode，当增量条目超过阈值时把折叠后的新基值写回 \texttt{inodes}，同时删除已消费的增量。当前默认阈值 32 条、检查周期 5 秒。其思路可以概括为：用可控的读路径折叠成本，换取写路径上的冲突减少。和 Mantle 的 Delta Record 相比，本文的 DeltaOp 沿用了“以增量换写放大”的基本思路，但将追加、折叠和提交都限制在同一个 RocksDB 实例内完成，实现上更为轻量。

这套机制的收益在低并发下并不明显：父目录上的写争抢需要达到一定密度，覆盖式更新和增量追加的差距才会拉开。第~\ref{sec:delta-vs-nodelta}~节将使用 Delta 和 NoDelta 两种编译变体做对照实验以衡量这一点——在共享父目录的元数据负载下，当到达服务端的请求速率跨过事务冲突阈值之后，关闭 DeltaOp 的构建吞吐出现明显下降，而开启 DeltaOp 的构建仍能继续扩展。

\subsubsection{删除语义与句柄生命周期}

除了事务原子性，元数据一致性还体现在删除语义上。POSIX 允许目录项已经从命名空间中消失，已打开的句柄仍然能继续访问原文件。仅靠一次 \texttt{unlink} 事务不足以完成这件事，因为“名字空间可见性”和“数据回收时机”并不总在同一时刻。本文因此在 MetadataServer 内维护了 \texttt{open\_handles} 和 \texttt{pending\_deletes} 两类状态。\texttt{open()} 成功后对应 inode 的打开计数加一；\texttt{release()} 时减一。

当 \texttt{unlink} 或覆盖式 \texttt{rename} 把某个普通文件的 \texttt{nlink} 降为 0 时，MetadataServer 并不一律立刻删除数据。若该 inode 还有打开句柄，系统只将其加入 \texttt{pending\_deletes}；等最后一个句柄关闭时，再通知客户端到 DataServer 侧真正清理数据。这样目录项删除、链接计数归零、物理数据回收就分为三个阶段：名字先从命名空间中消失，可达性由 \texttt{nlink} 表示，物理删除则推迟到没有进程持有该对象时再执行。从外部来看，这对应了 POSIX 要求的“名字已删除、但已有句柄仍然有效”的语义。pjdfstest 中的 \texttt{unlink::open\_file\_not\_freed} 用例直接检验了这一行为，结果见第~\ref{sec:pjdfstest-semantic}~节。

\subsection{核心操作实现}
\label{sec:posix_impl}

RucksFS 的大多数元数据操作走同一条主线：先定位少量相关的目录项键和 inode 记录，在同一事务内完成检查和修改，按需向父目录追加属性增量。本节按操作类别逐一说明这条主线如何落到具体流程上，同时也说明前述键编码、PCC、DeltaOp 三件事如何在这些操作中组合复用。

\subsubsection{create 与 mkdir}

\texttt{create} 的难点不在于“写入一个新 inode”，而在于需要同时满足三个条件：目标名字在此之前不存在；新对象以完整状态出现；父目录的属性更新与前两者保持原子一致。若将这些动作拆成多个独立步骤，就容易出现中间状态——例如 inode 已经建立但目录项尚未插入，或者同名检查通过后被其他并发事务抢先插入。算法~\ref{alg:create} 给出当前实现的 \texttt{create} 流程。

\begin{algorithm}[htbp]
    \caption{\texttt{create(parent, name, mode, uid, gid)} 的核心流程}
    \label{alg:create}
    \begin{algorithmic}[1]
        \State 开启事务 batch
        \State 对目录项键 \texttt{D[parent, name]} 执行 \texttt{get\_for\_update}
        \If{目录项已存在}
            \State 返回 \textsc{EEXIST}
        \EndIf
        \State 分配新 inode 编号，构造普通文件的 \texttt{InodeValue}
        \State 向 \texttt{inodes} 列族写入新 inode 基值
        \State 向 \texttt{dir\_entries} 列族写入父目录到新文件的映射
        \State 向 \texttt{inodes} 列族写入该 inode 的 \texttt{DataLocation}
        \State 向 \texttt{delta\_entries} 追加父目录的 \texttt{SetMtime} 与 \texttt{SetCtime}
        \State 提交事务
        \State 更新本地缓存并返回新文件属性
    \end{algorithmic}
\end{algorithm}

从键空间组织的角度看，\texttt{create} 能够直接体现“以边为中心”建模的好处：不需要改写整条路径，也不需要搬移子树，只需在 \texttt{dir\_entries} 中插入一条新边，并在 \texttt{inodes} 中补齐目标对象的基础记录和数据位置记录。新对象的可见性由一条目录项记录和一组 inode 记录共同决定，而不是由单一结构承载。

另一方面，父目录属性并不走“覆盖写回基础 inode”这条路径，而是追加两条 \texttt{DeltaOp}——一条 \texttt{SetMtime}、一条 \texttt{SetCtime}。这样一来，\texttt{create} 就从“目录项插入 + 父目录基值重写”的组合，转换为“目录项插入 + 若干新记录写入 + 增量追加”的组合。前者在热点目录下容易将竞争集中到父目录 inode 上；后者更贴合 LSM-tree 擅长的追加写模式。

\texttt{mkdir} 与 \texttt{create} 共用同一条事务骨架，但比普通文件创建多了一层目录相关的语义约束。新对象的类型位是目录，初始 \texttt{nlink} 为 2；父目录除了更新时间戳之外，还需要追加 \texttt{IncrementNlink(+1)}，因为多了一个子目录。普通文件创建不会改变父目录 \texttt{nlink}，而目录创建会——这一差异看似细微，但直接决定了后续 \texttt{rename}、\texttt{rmdir} 以及一致性检查的正确性。

\subsubsection{unlink、rmdir 与 release}

若将 \texttt{create} 理解为“建立一条新边”，则 \texttt{unlink} 和 \texttt{rmdir} 分别对应“删除一条普通文件边”和“删除一条目录边”。两者的共同点是都需要在事务内同时处理目录项可见性、目标对象状态和父目录属性变化；差异则来自 POSIX 对普通文件和目录所加的不同约束。

\texttt{unlink} 的核心流程是：锁定目标目录项，确认其不是目录，删除该条目录项记录，将目标 inode 的 \texttt{nlink} 减一，再根据链接计数和打开句柄的状态决定是否进入延迟删除分支。关键在于目录项删除和对象状态更新处于同一事务内，不会出现“名字已消失但链接计数尚未更新”这种中间状态。父目录的 \texttt{mtime}/\texttt{ctime} 同样不直接重写基值，继续走增量日志。

\texttt{rmdir} 的难点在于空目录检查。若写成“先扫描一遍目录、确认为空、再开启事务删除”，检查结果在事务开启前就可能失效。本文的做法是把“是否为空”的判断也放入事务，用一致性视图下的前缀扫描完成。若目录非空，直接返回 \texttt{ENOTEMPTY};若为空，则删除目标目录项和目标 inode，并向父目录追加 \texttt{IncrementNlink(-1)} 及一组时间戳增量。空目录约束由事务本身保证，不依赖上层约定。

\texttt{release} 对应延迟删除的最后一步——物理数据回收。\texttt{unlink} 和覆盖式 \texttt{rename} 可以使名字从命名空间中消失，但只要仍有进程持有打开句柄，物理数据就不能立刻清理。每次客户端关闭文件句柄时,\texttt{release} 都会将打开计数减一；若某个 inode 之前已进入 \texttt{pending\_deletes},而此时计数恰好降为 0,MetadataServer 才会把该 inode 返回给客户端，通知它到 DataServer 侧完成真正的数据删除。这样,“名字可见性”、“元数据可达性”和“物理数据回收”被清晰地分为三个阶段。

\subsubsection{rename}

\texttt{rename} 是前述几项设计决策综合体现最为充分的操作。它既依赖“以边为中心”的键建模，也依赖统一的加锁顺序、覆盖删除语义以及父目录属性增量，可以视作整条元数据路径中最能体现系统设计取舍的操作。它可能同时牵涉源目录、目标目录、被移动对象、潜在被覆盖对象这四类状态，并且要求整个过程对外原子可见。算法~\ref{alg:rename} 概括了这一流程。

\begin{algorithm}[htbp]
    \caption{\texttt{rename(old\_parent, old\_name, new\_parent, new\_name)} 的核心流程}
    \label{alg:rename}
    \begin{algorithmic}[1]
        \State 读取并锁定源目录项
        \State 若目标目录项存在，则读取其 inode 信息
        \State 收集全部相关 inode 编号并排序，按序执行 \texttt{get\_for\_update}
        \State 检查源对象与目标对象的类型兼容性
        \If{目标存在且为非空目录}
            \State 返回 \textsc{ENOTEMPTY}
        \EndIf
        \If{目标存在}
            \State 按覆盖删除逻辑处理目标对象
        \EndIf
        \State 删除源目录项，写入目标目录项
        \State 更新被移动对象的 \texttt{ctime}
        \If{移动的是目录且跨父目录}
            \State 对旧父目录追加 \texttt{IncrementNlink(-1)}
            \State 对新父目录追加 \texttt{IncrementNlink(+1)}
        \EndIf
        \State 对涉及的父目录追加时间戳增量
        \State 提交事务并返回被覆盖对象列表
    \end{algorithmic}
\end{algorithm}

从实现角度看,\texttt{rename} 的主体可以概括为“删除旧边、插入新边，并按需处理被覆盖对象”。这是边中心建模的直接好处：即便是跨目录移动，系统也只需要改动少量与源路径、目标路径直接相关的记录，而不需要像全路径键那样对子树做级联改写。\texttt{rename} 的复杂性主要来自语义约束，而非底层数据搬移。

排序加锁是 \texttt{rename} 避免死锁的关键，但其意义不止于此。其更关键的作用在于，统一加锁顺序让源目录、目标目录和被覆盖对象可以放在同一个一致的事务视图下，使“检查目标是否存在”、“检查目标目录是否为空”、“删除旧目录项”、“写入新目录项”、“处理覆盖删除”这一系列动作共享同一个一致性视图。在外部观察者看来,\texttt{rename} 表现为一次原子切换，不会暴露“旧名字已删除、新名字尚未建立”这类中间状态。

还有一个容易忽略的细节：只有在目录跨父目录移动时才需要调整父目录 \texttt{nlink}。移动普通文件不会影响任何目录的 \texttt{nlink};同一父目录内对目录重命名也不会影响父目录 \texttt{nlink}。这些差异直接决定了增量日志中需要追加哪些 \texttt{DeltaOp},也说明 \texttt{rename} 的正确性不仅取决于“目录项是否改动正确”,还取决于相关目录属性是否被同步维护。

\subsubsection{readdir、open 与写入协调}

\texttt{readdir} 最能体现键空间的局部性。在给定父目录 inode 之后,MetadataServer 构造前缀 \texttt{[D][parent: u64 BE]},利用 RocksDB 的前缀迭代器即可顺序扫描该目录下的所有 dentry。由于目录项 value 已包含子对象的 \texttt{mode},普通 \texttt{readdir} 无需为每个条目再回查一次 inode 即可获取文件类型。由此可见，在目录项 value 中额外保存一个 \texttt{child\_mode} 并非冗余，其目的正是降低目录遍历的读放大。

\texttt{open} 是元数据面与数据面的衔接点。它本身不负责数据读写，而是将“该 inode 的数据应去哪一台 DataServer 读写”这一信息交给客户端。因此 \texttt{open} 返回的不只是句柄，还包括对应的 \texttt{DataLocation}。这样后续的 \texttt{read}、\texttt{write}、\texttt{truncate} 就不必每次都回 MetadataServer 重新询问路由，可以直接访问正确的数据后端。

写入完成后的元数据更新同样体现出 RucksFS 的职责划分：DataServer 只负责把字节写入正确位置，并不主动维护文件长度或 \texttt{mtime}；这些语义统一由 MetadataServer 掌握。客户端在 \texttt{write\_data} 成功之后会显式调用 \texttt{report\_write(inode, new\_size, mtime)} 回写元数据。这样做的好处是边界清晰：数据路径保持简单，元数据语义集中维护；代价是客户端需要多做一次协调动作。客户端如何具体承担这一协调职责，以及 DataServer 在另一侧如何组织数据路径，留到第~\ref{sec:impl}~章展开。

\subsection{本章小结}

本章围绕 inode 与 dentry 这两个核心对象，集中讨论 MetadataServer 内部的元数据组织与事务模型。元数据按访问模式被划分到 \texttt{inodes}、\texttt{dir\_entries}、\texttt{delta\_entries}、\texttt{system} 四个列族；目录树通过“以边为中心”的键编码映射到 RocksDB 键空间，使 \texttt{create}、\texttt{unlink}、\texttt{rename} 等操作对应到对一条边的增删；事务部分借助 RocksDB 悲观锁和统一加锁顺序保证并发正确性；父目录属性则通过 DeltaOp 增量记录减少热点冲突。在这套键编码、事务与 DeltaOp 的组合下，\texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rmdir}、\texttt{rename}、\texttt{readdir}、\texttt{open} 等操作都能在同一条主线下落地。

本章的讨论仅限于 MetadataServer 内部。一次完整的用户请求还需要 FUSE 客户端、MetadataServer 与 DataServer 三者协同，这部分跨服务的路由、编排、数据路径、FUSE 可见语义以及错误处理，将在第~\ref{sec:impl}~章单独展开。
