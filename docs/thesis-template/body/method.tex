\section{系统设计与实现}
\label{sec:design}

RucksFS 是一个采用 Rust 实现的分布式文件系统原型。系统以 FUSE 客户端作为 POSIX 入口，以 MetadataServer 负责命名空间与 inode 元数据，以 DataServer 负责文件数据 I/O，并通过 gRPC 连接客户端与服务端。设计的关键不在于把“文件系统功能”简单拆成若干模块，而在于围绕元数据关键路径组织三件事：目录项如何映射到 RocksDB 键空间，目录修改如何在并发下保持正确性，以及客户端如何在元数据服务与数据服务之间完成协调。

\subsection{设计目标}

本文实现 RucksFS 时遵循了四个设计目标。

第一，客户端必须提供标准 POSIX 访问入口。应用程序通过普通的 \texttt{open()}、\texttt{rename()}、\texttt{unlink()} 等系统调用访问挂载点，而不需要为 RucksFS 单独改写接口或调用方式。

第二，元数据与文件数据在服务层显式分离。这不仅因为二者的访问模式不同，也因为本文的研究重点本来就放在元数据路径上。元数据更新涉及目录项组织、inode 属性维护、事务提交与并发控制，更适合放在 MetadataServer 中围绕 RocksDB 做针对性设计；文件数据访问则主要表现为按偏移读写与刷盘，更接近顺序 I/O 路径，交由 DataServer 单独处理更容易把问题边界划清。这样的拆分可以避免数据读写细节干扰本文对元数据组织方式的分析。

第三，目录操作必须在并发下保持可解释的正确性。名字唯一、目录非空检查、跨目录重命名原子性以及延迟删除等语义，都不能依赖“通常不会冲突”这样的经验假设，而需要在事务边界内被明确保证。

第四，系统应尽量把元数据性能瓶颈集中到少数可分析、可优化的关键路径上，例如目录项查找、事务提交与父目录属性更新，而不是把大量文件系统语义完全外包给通用数据库后，再把性能与正确性问题留给黑盒后端。

\subsection{系统整体架构}
\label{sec:architecture}

RucksFS 的运行时由三个主要角色组成：挂载在客户端机器上的 FUSE 客户端、负责命名空间与 inode 状态的 MetadataServer，以及负责文件内容读写的 DataServer。客户端接收来自内核的 FUSE 请求后，并不会把所有操作都转发到同一后端，而是先经过本地的 \texttt{VfsCore} 路由层，再决定请求属于元数据面还是数据面。图~\ref{fig:overall_arch}给出了系统的整体请求路径。

从请求走向上看，这种拆分对应三条不同路径。

\begin{enumerate}
    \item \textbf{元数据路径。} 诸如 \texttt{lookup}、\texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rename} 和 \texttt{readdir} 等请求，经 FUSE 客户端进入 \texttt{VfsCore} 后，被转发给 MetadataServer，再由后者访问 RocksDB。
    \item \textbf{数据路径。} \texttt{read}、\texttt{write}、\texttt{truncate} 与 \texttt{flush} 等请求，由 \texttt{VfsCore} 选择对应的 DataServer，并交由原始磁盘后端执行。
    \item \textbf{协调路径。} 某些操作天然跨越元数据面与数据面，例如客户端写入文件数据后，还需要通知 MetadataServer 更新文件大小和 \texttt{mtime}；\texttt{unlink} 或覆盖式 \texttt{rename} 完成后，MetadataServer 还会通过响应结构告知客户端哪些 inode 需要在 DataServer 侧清理数据。
\end{enumerate}

客户端不直接持有 RocksDB，也不直接管理原始磁盘布局。它只负责在 FUSE 请求、MetadataServer 和 DataServer 之间做语义编排。这样做的直接好处是，文件系统的正确性主要集中在元数据事务设计与客户端协调逻辑中，边界较清晰，调试路径也更可控。

\subsection{元数据模型与存储布局}
\label{sec:datamodel}

\subsubsection{核心对象模型}

RucksFS 采用典型的 inode 模型。每个文件系统对象都有一个全局唯一的 inode 编号，普通文件、目录和符号链接都以这种方式被标识。inode 保存的是对象属性，例如类型、权限、大小、链接计数以及时间戳，而不是路径名字。

路径中的名字由目录项维护。目录中的每一条目录项，本质上都是“名字到 inode 的映射关系”：父目录记录一个名字，再指出这个名字对应哪个 inode。也就是说，文件系统真正持久化的是 inode 和目录项，而不是把整条路径作为一个整体对象保存下来。

因此，路径解析就是一个逐层查找目录项的过程。以 \texttt{/docs/paper.tex} 为例，系统先在根目录下查找名字 \texttt{docs}，得到其对应的目录 inode；再进入该目录，继续查找名字 \texttt{paper.tex}，最终定位到目标文件 inode。这样的组织方式使 \texttt{lookup}、\texttt{readdir} 和 \texttt{rename} 都可以围绕“目录项映射”来实现，也为后续把目录树映射到键值空间提供了基础。

在实现中，对象属性由 \texttt{FileAttr} 抽象表示，目录遍历结果由 \texttt{DirEntry} 表示。前者面向 inode 本体，后者面向“目录中的一条边”。表~\ref{tab:fileattr}列出了 \texttt{FileAttr} 中最重要的字段。

\subsubsection{列族划分}

列族划分是 RucksFS 元数据设计中的核心决策之一。若把 inode、目录项、增量日志和系统状态全部放进一个统一键空间，点查、前缀扫描与高频追加写就会共享同一组 MemTable、SST 文件和压缩策略，最终在缓存命中率、Compaction 行为与读写放大之间相互牵制。RucksFS 因此没有采用“所有元数据混放”的做法，而是按访问模式把 RocksDB 实例拆成 4 个列族。表~\ref{tab:columnfamilies}先给出列族名称与主要内容。

\texttt{inodes} 列族承担“对象本体”的点查负载。inode 基础属性、\texttt{DataLocation} 以及符号链接目标都被放在这一列族中，不是因为它们在语义上完全相同，而是因为它们都表现为按 inode 直接定位的点查访问。把这些内容放在同一列族，可以避免为了少量辅助元数据再额外维护新的 MemTable 和压缩状态。

\texttt{dir\_entries} 列族承担命名空间边关系。\texttt{lookup} 需要按“父 inode + 名字”定位单条目录项，\texttt{readdir} 需要按父目录枚举全部子项，这两种访问都依赖前缀局部性。因此该列族启用了 9 字节前缀提取器，使同一父目录下的目录项可以按前缀连续排列。与把目录项和 inode 混放在一起相比，这种做法更有利于目录遍历与前缀 Bloom 过滤器生效；与一些把完整路径作为 key 的方案相比，它又避免了跨目录 \texttt{rename} 时对子树全量改写。

\texttt{delta\_entries} 列族承担热点目录的增量写入。其设计目标不是长期保存完整状态，而是承接高频的 \texttt{mtime}/\texttt{ctime}/\texttt{nlink} 变化，避免每次目录修改都去重写父目录 inode 基值。Delta 的 value 并不是通用 JSON 或 protobuf，而是极短的二进制 \texttt{DeltaOp}，例如 \texttt{IncrementNlink(i32)} 用 5 字节表示，\texttt{SetMtime(u64)} 等时间戳操作用 9 字节表示。由于该列族条目短小、生命周期短，而且会被后台折叠线程周期性消费，因此这里刻意关闭压缩，以减少热路径上的编解码负担。

\texttt{system} 列族只保存 inode 分配器等低频系统元信息。将这部分内容单独隔离，可以避免分配器状态与热点目录项共享同一压缩和缓存行为。虽然它在访问频率上远低于其他列族，但仍然需要与普通元数据一起参与原子提交，因此保留在同一 RocksDB 实例中，而不是另起一套外部存储。

这种列族拆分体现的是“按访问模式设计元数据布局”，而不是形式上的模块化。它与把所有元数据塞进单一 KV 空间的方案不同，也与把元数据语义完全交给外部数据库默认布局的思路不同。对本文而言，列族设计本身就是元数据关键路径优化的一部分，因此需要在本章明确展开，而不能仅在实现细节中一笔带过。

这种设计遵循的是“以边为中心”的命名空间建模思路。目录树中的每一条父子边，都对应 \texttt{dir\_entries} 列族中的一条记录。\texttt{lookup(parent, name)} 退化为点查一条目录项键；\texttt{readdir(parent)} 退化为扫描同一前缀下的全部目录项键；\texttt{rename} 则退化为删除旧边、插入新边。与使用全路径作为键相比，这种建模方式避免了跨目录重命名时对子树全量重写。

除 key 之外，不同类型 value 的编码也经过了针对性设计。inode value 采用 57 字节的定长二进制布局，依次保存版本号、inode 编号、文件大小、模式位、链接计数以及三类时间戳；目录项 value 固定为 12 字节，即 \texttt{[child\_inode: u64 BE][child\_mode: u32 BE]}；delta value 则采用极短的 \texttt{DeltaOp} 编码。把 \texttt{mode} 一并放入目录项 value，可以让 \texttt{readdir} 在不额外回查 inode 的情况下直接得到对象类型；而定长 inode value 则有利于在点查路径上减少解析开销。
\subsubsection{固定长度 inode 序列化}

MetadataServer 在 RocksDB 中存储的 inode 基础值并不使用通用序列化框架，而是采用固定长度的 \texttt{InodeValue} 手工编码。当前布局总长度为 57 字节，包含版本号、inode 编号、文件大小、模式位、链接计数以及三类时间戳。采用固定长度编码有两个直接好处：其一，读写路径只需要按偏移切片，不必承担变长解码开销；其二，调试时可以直接从 RocksDB 中观察二进制布局，问题定位更直接。

固定长度编码也意味着模式演进需要显式处理。为此，\texttt{InodeValue} 的第一个字节保留为格式版本号。只要未来确实需要扩展字段，就可以通过版本字段引入兼容逻辑，而不必打破既有键空间。

\subsection{事务模型与元数据一致性}
\label{sec:txn_model}

\subsubsection{原子批量写抽象}

RucksFS 并不直接在上层业务代码里拼装 RocksDB 原生事务，而是通过 \texttt{StorageBundle} 和 \texttt{AtomicWriteBatch} 进行封装。\texttt{StorageBundle::begin\_write()} 返回一个跨列族的批量写对象，调用者向其中追加 \texttt{BatchOp}，最后统一提交。这样做的目的有两点：一是把列族差异藏在存储层内部，二是为后续在事务中执行 \texttt{get\_for\_update}、目录是否为空检查等操作提供统一入口。

当前存储后端由 \texttt{RocksStorageBundle} 实现，其内部持有 \texttt{Arc<TransactionDB>}。提交时，批量写对象最终调用的是 RocksDB 事务的 \texttt{commit()}，从而把跨 \texttt{inodes}、\texttt{dir\_entries} 和 \texttt{delta\_entries} 的多条修改一起持久化。

\subsubsection{悲观事务与加锁顺序}

MetadataServer 的目录操作依赖 RocksDB TransactionDB 的悲观事务接口。关键方法是 \texttt{get\_for\_update}：一旦在事务内读取某个目录项键或 inode 键，就同时获取该键的排他锁。RucksFS 还显式配置了 5 秒的锁等待超时，并关闭 RocksDB 的死锁检测；原因不是忽略死锁风险，而是系统已经在上层显式约束了加锁顺序。

对于可能涉及多个 inode 的操作，MetadataServer 会先收集全部相关 inode 编号，再按升序依次加锁。\texttt{rename} 是最典型的场景：源目录、目标目录、被移动对象以及潜在被覆盖对象，都可能同时出现在同一事务中。统一的加锁顺序可以避免线程 A 先锁源目录、线程 B 先锁目标目录时形成循环等待。

事务冲突不是直接向外暴露原始 RocksDB 错误，而是通过统一的重试逻辑处理。MetadataServer 对临时冲突最多重试 3 次，退避时间按指数增长；若多次重试后仍然失败，错误再向上返回，并在 FUSE 层映射为 \texttt{EAGAIN}。这样既保留了并发正确性，也避免把偶发冲突直接放大为用户可见错误。

\subsubsection{增量更新与懒折叠}

若每次 \texttt{create}、\texttt{unlink}、\texttt{mkdir} 和 \texttt{rmdir} 都直接读取并重写父目录 inode，高频热点目录很快就会成为争用中心。RucksFS 因此没有把父目录时间戳与链接计数更新写回 \texttt{inodes} 基值，而是将这些变化编码为 \texttt{DeltaOp}，追加到 \texttt{delta\_entries} 列族中。

当前实现中的增量类型包括 \texttt{IncrementNlink}、\texttt{SetMtime}、\texttt{SetCtime} 和 \texttt{SetAtime}。其中，普通文件的创建与删除只会追加父目录 \texttt{mtime}/\texttt{ctime} 变化；\texttt{mkdir}、\texttt{rmdir} 以及跨目录移动目录时，才会额外追加 \texttt{nlink} 增量。这样做的直接收益是，热点父目录不再需要在每次操作中执行“读完整 inode，再改几个字段，再写回完整 inode”的流程。

读取路径则通过“基值 + 增量在线折叠”恢复最新状态。MetadataServer 在加载 inode 时，先尝试从 \texttt{InodeFoldedCache} 获取已经折叠好的结果；若缓存未命中，则先读 \texttt{inodes} 列族中的基础值，再按前缀扫描 \texttt{delta\_entries} 中属于该 inode 的全部增量，并依次执行折叠。折叠后的结果会重新写入缓存，以避免后续重复扫描。

后台的 \texttt{DeltaCompactionWorker} 负责把积累的增量合并回基础值。写路径在追加增量后会把对应 inode 标记为 dirty；后台线程周期性检查 dirty 集合，当某个 inode 的增量数量超过阈值时，便读取基值、折叠增量，并通过原子提交把“写回新基值”和“删除已消费增量”一起完成。当前默认阈值为 32 条，检查周期为 5 秒。这一机制的本质，是用读路径上的有限折叠成本换取写路径上的冲突削减。

\subsubsection{延迟删除与句柄生命周期}

POSIX 允许文件在目录项已经被移除后继续被打开的句柄访问。RucksFS 为此在 MetadataServer 内维护了 \texttt{open\_handles} 与 \texttt{pending\_deletes} 两类状态。\texttt{open()} 成功后，MetadataServer 递增该 inode 的打开计数，并把返回句柄与 \texttt{DataLocation} 一并交给客户端；\texttt{release()} 则递减打开计数。

当 \texttt{unlink} 或覆盖式 \texttt{rename} 使某个普通文件的 \texttt{nlink} 降到 0 时，MetadataServer 不一定立刻要求客户端删除数据。若该 inode 仍有打开句柄，系统只把它加入 \texttt{pending\_deletes}，并在最后一个句柄关闭时再通知客户端执行数据删除。这样才能满足“名字已经不可见，但已有句柄仍可继续使用”的 POSIX 语义。

\subsection{核心操作实现}
\label{sec:posix_impl}

\subsubsection{create 与 mkdir}

\texttt{create} 的关键不是写入新 inode，而是确保“同名检查”和“目录项插入”处于同一事务中。算法~\ref{alg:create}给出了当前实现中的 \texttt{create} 流程。

\begin{algorithm}[htbp]
    \caption{\texttt{create(parent, name, mode, uid, gid)} 的核心流程}
    \label{alg:create}
    \begin{algorithmic}[1]
        \State 开启事务 batch
        \State 对目录项键 \texttt{D[parent, name]} 执行 \texttt{get\_for\_update}
        \If{目录项已存在}
            \State 返回 \textsc{EEXIST}
        \EndIf
        \State 分配新 inode 编号，构造普通文件的 \texttt{InodeValue}
        \State 向 \texttt{inodes} 列族写入新 inode 基值
        \State 向 \texttt{dir\_entries} 列族写入父目录到新文件的映射
        \State 向 \texttt{inodes} 列族写入该 inode 的 \texttt{DataLocation}
        \State 向 \texttt{delta\_entries} 追加父目录的 \texttt{SetMtime} 与 \texttt{SetCtime}
        \State 提交事务
        \State 更新本地缓存并返回新文件属性
    \end{algorithmic}
\end{algorithm}

\texttt{mkdir} 与 \texttt{create} 的差异主要体现在两个地方。第一，新对象的类型位为目录，且初始 \texttt{nlink} 为 2；第二，父目录除了更新时间戳之外，还需要追加 \texttt{IncrementNlink(+1)}，因为新增了一个子目录。普通文件创建不会改变父目录 \texttt{nlink}，这一点在后续实验分析中很重要。

\subsubsection{unlink、rmdir 与 release}

\texttt{unlink} 先锁定目标目录项，再锁定被删除对象的 inode。若目标实际上是目录，则直接返回 \texttt{EISDIR}。对普通文件而言，\texttt{unlink} 会删除目录项、递减目标 inode 的 \texttt{nlink}，并根据结果决定是否进入延迟删除分支。父目录的 \texttt{mtime}/\texttt{ctime} 更新则通过增量日志完成。

\texttt{rmdir} 的关键检查在于“目录是否为空”。RucksFS 并不是在事务外先做一次普通扫描，再进入事务删除目录项；相反，它通过事务内的前缀扫描接口在一致性视图里完成“是否为空”的检查，然后才决定是否执行删除。若目录非空，则返回 \texttt{ENOTEMPTY}；若目录为空，则删除目标目录项与目标 inode，并对父目录追加 \texttt{IncrementNlink(-1)} 以及时间戳增量。

\texttt{release} 并不对应普通元数据变更，但它是延迟删除语义闭环中不可缺少的一环。每次客户端关闭文件句柄时，\texttt{release} 都会递减打开计数；若某个 inode 先前已经进入 \texttt{pending\_deletes} 且此时计数降为 0，MetadataServer 就把该 inode 返回给客户端，通知后者继续在 DataServer 侧执行最终的数据删除。

\subsubsection{rename}

\texttt{rename} 是当前实现中最复杂的目录操作。它可能同时涉及源目录、目标目录、被移动对象以及潜在被覆盖对象四类状态，且要求整个过程原子可见。算法~\ref{alg:rename}概括了这一流程。

\begin{algorithm}[htbp]
    \caption{\texttt{rename(old\_parent, old\_name, new\_parent, new\_name)} 的核心流程}
    \label{alg:rename}
    \begin{algorithmic}[1]
        \State 读取并锁定源目录项
        \State 若目标目录项存在，则读取其 inode 信息
        \State 收集全部相关 inode 编号并排序，按序执行 \texttt{get\_for\_update}
        \State 检查源对象与目标对象的类型兼容性
        \If{目标存在且为非空目录}
            \State 返回 \textsc{ENOTEMPTY}
        \EndIf
        \If{目标存在}
            \State 按覆盖删除逻辑处理目标对象
        \EndIf
        \State 删除源目录项，写入目标目录项
        \State 更新被移动对象的 \texttt{ctime}
        \If{移动的是目录且跨父目录}
            \State 对旧父目录追加 \texttt{IncrementNlink(-1)}
            \State 对新父目录追加 \texttt{IncrementNlink(+1)}
        \EndIf
        \State 对涉及的父目录追加时间戳增量
        \State 提交事务并返回被覆盖对象列表
    \end{algorithmic}
\end{algorithm}

排序加锁是 \texttt{rename} 能够避免死锁的关键。若线程 A 正在执行 \texttt{rename(dir1/a, dir2/b)}，线程 B 同时执行 \texttt{rename(dir2/c, dir1/d)}，只要双方都按 inode 编号升序加锁，就不会形成“你等我、我等你”的循环等待。

另一个容易被忽视的细节是，目录跨父目录移动时才需要调整父目录 \texttt{nlink}。移动普通文件不会影响任何目录的 \texttt{nlink}；同一父目录内对目录重命名，也不会影响父目录 \texttt{nlink}。这些细节都直接决定了增量日志中应追加哪些 \texttt{DeltaOp}。

\subsubsection{readdir、open 与写入协调}

\texttt{readdir} 完全依赖 \texttt{dir\_entries} 列族的前缀扫描能力。给定父目录 inode 后，MetadataServer 构造前缀 \texttt{[D][parent: u64 BE]}，调用 RocksDB 前缀迭代器读取全部同前缀目录项值，并从目录项值中直接提取子 inode 与 \texttt{mode}。由于目录项值中已经携带对象类型，普通 \texttt{readdir} 不必逐项回查 inode。当前实现返回的是持久化目录项本身，不额外在元数据层存储 \texttt{.} 与 \texttt{..} 两个伪条目。

\texttt{open} 的职责与本地文件系统有所不同。它不仅要为客户端分配句柄，还要返回该 inode 对应的 \texttt{DataLocation}，使客户端后续的 \texttt{read} 和 \texttt{write} 可以直接路由到正确的 DataServer。当前实现里，\texttt{VfsCore} 维护了句柄到 \texttt{server\_id} 的映射；\texttt{read}、\texttt{write} 与 \texttt{truncate} 则根据该映射选择数据后端。

写入完成后的元数据更新由客户端主动协调。DataServer 成功完成 \texttt{write\_data} 后，客户端并不会假设 MetadataServer 已经自动得知文件大小变化，而是显式调用 \texttt{report\_write(inode, new\_size, mtime)}。这样做的结果是，数据路径可以保持尽量简洁，而文件长度与修改时间仍由 MetadataServer 统一维护。

\subsection{数据服务与客户端协调}
\label{sec:data_path}

\subsubsection{VfsCore 路由层}

\texttt{VfsCore} 是客户端内部最关键的协调层。它实现 \texttt{VfsOps}，对 FUSE 层暴露的是完整的 POSIX 风格接口；对下游则同时持有一个 \texttt{MetadataOps} 对象和一个或多个 \texttt{DataOps} 对象。元数据请求由 \texttt{VfsCore} 直接转发给 MetadataServer；数据请求则根据句柄映射或默认 DataServer 路由到相应 \texttt{DataOps} 实现。

更重要的是，\texttt{VfsCore} 还承担“补齐跨服务语义”的职责。MetadataServer 返回的 \texttt{UnlinkResponse}、\texttt{RenameResponse} 和 \texttt{ReleaseResponse} 中，会携带 \texttt{purged\_inodes} 列表；客户端收到这些列表后，再调用 DataServer 的 \texttt{delete\_data} 完成数据侧清理。\texttt{setattr} 若导致文件缩短，也是在客户端收到 \texttt{needs\_truncate} 后，由 \texttt{VfsCore} 进一步调用 DataServer 执行真实的数据截断。

换言之，RucksFS 并没有把所有“涉及元数据与数据两侧的复合语义”都压到 MetadataServer 内部，而是通过 \texttt{VfsCore} 在客户端做显式协调。这种设计让 MetadataServer 保持“只理解元数据事务”的边界，同时又没有牺牲最终的 POSIX 表现。

\subsubsection{DataServer 与原始磁盘后端}

DataServer 的实现相对直接。它只是对 \texttt{DataStore} 的封装层，向上实现 \texttt{DataOps}，向下把请求交给具体的数据后端。当前唯一的数据后端是 \texttt{RawDiskDataStore}，它把所有 inode 的数据映射到一个连续的 \texttt{data.raw} 文件中，并用公式
\[
\text{disk\_offset} = \text{inode} \times \text{max\_file\_size} + \text{file\_offset}
\]
计算绝对偏移。随后，读取和写入分别通过 \texttt{pread} 与 \texttt{pwrite} 完成。

这种设计很朴素，但有两个现实好处。第一，数据路径不需要再经过 LSM-tree，也不需要维护 extent 树或块分配器，读写语义足够直接。第二，\texttt{pread}/\texttt{pwrite} 不共享文件游标，不同线程可以并发访问同一后端文件而不需要额外的全局锁。其不足也同样明显：每个 inode 会被分配一个固定大小的逻辑地址空间，空间利用率不如成熟块映射方案高。

当前实现中的 \texttt{delete\_data} 仍然是空操作。原因不是忽略删除语义，而是 inode 编号目前单调递增且不复用，旧数据区域一旦失去目录项与 inode 的可达性，就不会再被后续对象引用。这样做避免了在 \texttt{unlink} 热路径上执行大块零填充，但也意味着物理空间回收尚未实现，这属于后续需要补足的工程能力。

\subsubsection{挂载选项与客户端实现细节}

FUSE 客户端的挂载选项会直接影响访问语义与系统表现。当前实现通过 \texttt{fuse3} 设置了 \texttt{fs\_name("rucksfs")}、\texttt{force\_readdir\_plus(true)}、\texttt{allow\_other(true)} 和 \texttt{default\_permissions(true)}。其中，\texttt{force\_readdir\_plus} 使内核优先走 \texttt{readdirplus} 路径，客户端可以在目录遍历返回中附带目录项的基本属性信息；\texttt{default\_permissions} 则把基础权限检查交由 Linux VFS 完成。客户端返回给内核的属性 TTL 默认为 1 秒，\texttt{max\_write} 设置为 128\,KiB。

这些配置并不神秘，但它们共同决定了一个事实：RucksFS 的客户端并不是单纯的“RPC 转发器”。它既要满足 FUSE 挂载的交互约束，又要在多服务之间协调语义，还要控制目录遍历与属性访问的代价。因此，客户端本身就是系统设计的一部分，而不只是一个薄薄的封装层。

\subsection{实现取舍与当前边界}

RucksFS 当前实现有意保持了“单 MetadataServer + 单默认 DataServer”的主路径。虽然 \texttt{DataLocation} 与 \texttt{VfsCore::with\_data\_servers()} 已经为多 DataServer 路由预留接口，但元数据分片、多副本一致性和物理空间回收都尚未进入本文实现范围。与此同时，MetadataServer 已经完整实现了目录事务、增量更新、后台折叠和延迟删除；因此，本文更关注的是“客户端/元数据服务/数据服务分离条件下的元数据关键路径优化”，而不是大规模分片与副本管理。

这也是全文设计边界最需要明确的一点：RucksFS 已经不是单机文件系统原型，而是一个具备真实客户端、元数据服务和数据服务边界的分布式原型；但它也尚未扩展到多元数据分片、多副本容错和完整回收策略。这一定位决定了后续实验的意义：它主要验证的是元数据组织方式与客户端协调逻辑，而不是一个大规模分布式控制面的最终形态。

\subsection{本章小结}

本章给出了 RucksFS 的具体实现方案。系统以 FUSE 客户端、MetadataServer 和 DataServer 组成三段式请求路径，以 \texttt{VfsCore} 协调元数据与数据操作；元数据被拆分为 4 个 RocksDB 列族，并以大端序编码组织 inode、目录项、增量日志和系统状态；目录操作依赖悲观事务与统一加锁顺序保证并发正确性；父目录热更新则通过增量日志与后台折叠削弱写争用。基于这套设计，第四章将进一步验证其正确性与性能表现。
