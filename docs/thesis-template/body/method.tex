\section{系统设计与实现}
\label{sec:design}

RucksFS 是一个采用 Rust 实现的分布式文件系统原型。系统由三部分组成：FUSE 客户端提供 POSIX 入口，MetadataServer 负责管理命名空间与 inode 元数据，DataServer 则处理文件数据的读写操作，客户端与服务端之间通过 gRPC 通信。本设计关注的核心是如何围绕元数据访问路径，高效组织三件关键事务：目录项如何映射到 RocksDB 键空间，目录修改如何在并发下保持正确性，以及客户端如何在元数据服务与数据服务之间完成协调。

\subsection{设计目标}

本文实现 RucksFS 时遵循了四个设计目标。

第一，客户端必须提供标准 POSIX 访问入口。应用程序通过普通的 \texttt{open()}、\texttt{rename()}、\texttt{unlink()} 等系统调用访问挂载点，而不需要为 RucksFS 单独改写接口或调用方式。

第二，元数据与文件数据在服务层显式分离。这不仅因为二者的访问模式迥异，也是因为本文的研究重点本来就放在元数据路径上。元数据更新涉及目录项组织、inode 属性维护、事务提交与并发控制等复杂逻辑，将其集中在 MetadataServer 中并基于 RocksDB 进行专门优化更为合理；文件数据访问则主要表现为按偏移读写与刷盘，更接近顺序 I/O 路径，交由 DataServer 单独处理更容易划清问题边界。这样的拆分可以避免数据读写细节干扰本文对元数据组织方式的分析。

第三，目录操作必须在并发下保持可解释的正确性。名字唯一、目录非空检查、跨目录重命名原子性以及延迟删除等语义，都不能依赖“通常不会冲突”这样的经验假设，而需要在事务边界内被明确保证。

第四，系统应尽量把元数据性能瓶颈集中到少数可分析、可优化的关键路径上，例如目录项查找、事务提交与父目录属性更新，而不是把大量文件系统语义完全外包给通用数据库后，再把性能与正确性问题留给黑盒后端。

\subsection{系统整体架构}
\label{sec:architecture}

RucksFS 的运行时由三个主要角色组成：挂载在客户端机器上的 FUSE 客户端、负责命名空间与 inode 状态的 MetadataServer，以及负责文件内容读写的 DataServer。客户端收到来自内核的 FUSE 请求后，并非将所有操作不加区分地转发到同一后端，而是先经过本地的 \texttt{VfsCore} 路由层，再根据请求类型决定应发送给元数据面还是数据面。图~\ref{fig:overall_arch} 给出了系统的整体请求路径。

从请求走向上看，这种拆分对应三条不同路径。

第一，元数据路径。诸如 \texttt{lookup}、\texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rename} 和 \texttt{readdir} 等请求，经 FUSE 客户端进入 \texttt{VfsCore} 后，被转发给 MetadataServer，再由后者访问 RocksDB。

第二，数据路径。\texttt{read}、\texttt{write}、\texttt{truncate} 与 \texttt{flush} 等请求，由 \texttt{VfsCore} 选择对应的 DataServer，并交由原始磁盘后端执行。

第三，协调路径。某些操作天然跨越元数据面与数据面。例如，客户端写入文件数据后，还需要通知 MetadataServer 更新文件大小和 \texttt{mtime}；又如 \texttt{unlink} 或覆盖式 \texttt{rename} 完成后，MetadataServer 会通过响应结构告知客户端哪些 inode 需要在 DataServer 侧清理数据。

客户端不直接持有 RocksDB，也不直接管理原始磁盘布局。它只负责在 FUSE 请求、MetadataServer 和 DataServer 之间做语义编排。这样做的直接好处是，文件系统的正确性主要集中在元数据事务设计与客户端协调逻辑中，边界较清晰，调试路径也更可控。

上述三段式架构确定了 RucksFS 的全局布局。接下来的几节将依次展开各层的具体设计：元数据如何在 RocksDB 键空间中组织，事务模型如何保证并发正确性，核心 POSIX 操作如何映射到这套键空间和事务上，以及客户端如何协调元数据面与数据面。

\subsection{元数据模型与存储布局}
\label{sec:datamodel}

本节先给出面向实现的元数据全景，再分别展开列族划分、键编码和值编码的细节与设计考量。这里的“全景”只回答三个问题：系统保存哪些对象，这些对象分别落在哪个列族，以及它们的 key/value 编码形式是什么。至于为什么这样划分、这种编码对各类操作意味着什么，则放到后续小节再讨论。

\subsubsection{元数据方案总览}

RucksFS 从一棵目录树出发理解命名空间。用户看到的是按目录组织的层次结构，但在持久化层，真正需要保存的核心对象只有两类：inode 与 dentry（目录项）。其中，inode 表示文件系统对象本体，负责保存类型、权限、大小、链接计数和时间戳等属性；dentry 表示目录树中的一条父子边，记录“父目录中的某个名字指向哪个 inode”。普通文件、目录和符号链接都先由 inode 标识，再通过 dentry 挂接到目录树中。

这样一来，路径本身不是单独保存的对象，而是由一串 dentry 逐级串联出来的结果。以 \texttt{/docs/paper.tex} 为例，系统先在根目录下找到名字 \texttt{docs} 对应的 dentry，得到目录 inode；再在该目录下找到名字 \texttt{paper.tex} 对应的 dentry，最终定位到目标 inode。围绕这两个核心对象，RucksFS 在 RocksDB 中分别组织 inode 相关记录与 dentry 相关记录，并在此基础上补充数据位置、属性增量和系统状态等辅助元数据。图~\ref{fig:cf_kv_layout} 给出了这套方案的整体视图。

\subsubsection{列族划分与设计考量}

在上述全景布局之上，RucksFS 需要回答的下一个问题是：为什么这些逻辑对象要被这样分组，而不是落入一个统一键空间。有鉴于此，RucksFS 没有采用“所有元数据混放”的做法，而是按访问模式把 RocksDB 实例拆成 4 个列族：\texttt{inodes}、\texttt{dir\_entries}、\texttt{delta\_entries} 和 \texttt{system}。它们分别承担 inode 相关点查、目录项存储、属性增量追加以及系统状态保存。

\texttt{inodes} 列族承担“对象本体”的点查负载。inode 基础属性、\texttt{DataLocation} 和符号链接目标虽然在语义上并不完全同类，但其访问模式一致，均表现为按 inode 直接定位的点查，因此被组织在同一列族中。把这些内容放在一起，可以避免为了少量辅助元数据再额外维护新的 MemTable 和压缩状态。

\texttt{dir\_entries} 列族承担命名空间边关系。\texttt{lookup} 需要按“父 inode + 名字”定位单条目录项，\texttt{readdir} 需要按父目录枚举全部子项，这两种访问都依赖前缀局部性。因此该列族启用了 9 字节前缀提取器，使同一父目录下的目录项可以按前缀连续排列。与把目录项和 inode 混放在一起相比，这种做法更有利于目录遍历与前缀 Bloom 过滤器生效；与一些把完整路径作为键的方案相比，它又避免了跨目录 \texttt{rename} 时对子树全量改写。

\texttt{delta\_entries} 列族承担热点目录的增量写入。其设计目标不是长期保存完整状态，而是承接高频的 \texttt{mtime}/\texttt{ctime}/\texttt{nlink} 变化，避免每次目录修改都去重写父目录 inode 基值。由于这一列族中的条目短小、生命周期短，而且会被后台折叠线程周期性消费，因此这里刻意关闭压缩，以减少热路径上的编解码负担。

\texttt{system} 列族只保存 inode 分配器等低频系统元信息。将这部分内容单独隔离，可以避免分配器状态与热点目录项共享同一压缩和缓存行为。虽然它在访问频率上远低于其他列族，但仍需要与普通元数据一起参与原子提交，因此保留在同一 RocksDB 实例中，而不是另起一套外部存储。

整体来看，这种列族划分遵循的是“按访问模式组织布局”的原则，而不是“按语义模块机械拆分”的原则。对本文而言，列族设计本身就是元数据关键路径优化的一部分，因此需要先在结构上讲清“为什么这样分”，再讨论“每类记录如何编码”。

\subsubsection{键编码与目录映射}

这种设计遵循的是”以边为中心”的命名空间建模思路。目录树中的每一条父子边，都对应 \texttt{dir\_entries} 列族中的一条记录。\texttt{lookup(parent, name)} 退化为点查一条目录项键；\texttt{readdir(parent)} 退化为扫描同一前缀下的全部目录项键；\texttt{rename} 则退化为删除旧边、插入新边。与使用全路径作为键相比，这种建模方式避免了跨目录重命名时对子树全量重写。第~\ref{sec:pjdfstest}~节中 pjdfstest 的 \texttt{rename}、\texttt{readdir}、\texttt{lookup} 用例将直接检验这一建模方式的正确性。

\textbf{与 TableFS 键编码方案的对比。} 最早将 KV 存储应用于文件系统元数据的 TableFS（Ren et al., 2013）同样采用“父 inode + 文件名”作为目录项键。但 TableFS 使用原生字节序编码 inode 编号，导致同一父目录下的所有目录项在 LSM-tree 的键空间中不一定物理相邻——因为 LevelDB 按字节序比较键值，inode 编号 256 的字节序表示为 \texttt{\textbackslash x01\textbackslash x00\textbackslash x00\textbackslash x01}，而 inode 编号 257 表示为 \texttt{\textbackslash x01\textbackslash x00\textbackslash x00\textbackslash x02}，两者之间可能隔着大量其他 inode 的键。RucksFS 采用大端序编码，使得 inode 编号的大小顺序与字典序一致，同一父目录下的所有子项天然在键空间中连续排列。这一差异在 \texttt{readdir} 场景下尤为关键：TableFS 需要在扫描时额外过滤父目录编号，而 RucksFS 只需一次前缀迭代即可获得全部目录项。此外，大端序也使得前缀 Bloom 过滤器在同一父目录内的假阳性率更低。

\subsubsection{Value 编码与定长序列化}

在实现层，inode 的基础属性由 \texttt{FileAttr} 抽象表示。它对应上层接口最常使用的一组对象属性，也是 \texttt{InodeValue} 编码时需要覆盖的核心字段，其内容主要包括 inode 编号、文件大小、模式位、链接计数以及访问、修改和状态变更时间戳。

在键编码之外，不同类型 value 的布局也经过了针对性设计。inode value 采用 57 字节的定长二进制布局，依次保存版本号、inode 编号、文件大小、模式位、链接计数以及三类时间戳；目录项 value 固定为 12 字节，即 \texttt{[child\_inode: u64 BE][child\_mode: u32 BE]}；delta value 则采用极短的 \texttt{DeltaOp} 编码。

这三种 value 的设计分别服务于不同目标。首先，把 \texttt{mode} 一并放入目录项 value，可以让 \texttt{readdir} 在不额外回查 inode 的情况下直接得到对象类型。其次，delta value 使用短小的 \texttt{DeltaOp} 编码，是为了让热点目录上的属性更新尽可能退化为小对象追加写。最后，inode value 采用定长布局，则有利于在点查路径上减少解析开销。

MetadataServer 在 RocksDB 中存储的 inode 基础值并不使用通用序列化框架，而是采用固定长度的 \texttt{InodeValue} 手工编码。当前布局总长度为 57 字节，包含版本号、inode 编号、文件大小、模式位、链接计数以及三类时间戳。采用固定长度编码有两个直接好处：其一，读写路径只需要按偏移切片，不必承担变长解码开销；其二，调试时可以直接从 RocksDB 中观察二进制布局，问题定位更直接。

固定长度编码也意味着模式演进需要显式处理。为此，\texttt{InodeValue} 的第一个字节保留为格式版本号。只要未来确实需要扩展字段，就可以通过版本字段引入兼容逻辑，而不必打破既有键空间。

\subsection{事务模型与元数据一致性}
\label{sec:txn_model}

\subsubsection{事务边界与原子提交}

RucksFS 中的大多数元数据操作都不是单键更新。以 \texttt{create} 为例，它至少会同时修改目录项、目标 inode、数据位置以及父目录属性；\texttt{rename} 则往往同时涉及源目录、目标目录、被移动对象和潜在被覆盖对象。因此，事务模型首先要回答的问题不是“如何加锁”，而是“哪些修改必须作为一个不可分割的整体提交”。

为此，RucksFS 没有在业务代码中直接散落地调用 RocksDB 原生事务接口，而是通过 \texttt{StorageBundle} 和 \texttt{AtomicWriteBatch} 封装跨列族写入。调用方先通过 \texttt{StorageBundle::begin\_write()} 建立事务上下文，再向其中追加 \texttt{BatchOp}，最后统一提交。这样做的意义在于：上层逻辑可以围绕“目录项是否存在”“应写入哪些 inode 记录”“父目录需要追加哪些增量”来组织操作，而不必显式处理不同列族的底层细节。提交时，底层最终调用 RocksDB 事务的 \texttt{commit()}，从而把跨 \texttt{inodes}、\texttt{dir\_entries} 和 \texttt{delta\_entries} 的修改作为一个原子单元持久化。

\subsubsection{悲观并发控制与冲突处理}

在原子提交之外，RucksFS 还需要保证并发目录操作的正确性。这里采用的是 RocksDB TransactionDB 的悲观并发控制（PCC）接口。其关键操作是 \texttt{get\_for\_update}：事务在读取目录项键或 inode 键的同时获取排他锁，使“检查条件”和“执行修改”处于同一事务视图内。对文件系统而言，这一点尤为重要，因为 \texttt{create}、\texttt{rmdir}、\texttt{rename} 等操作都包含典型的 TOCTOU 风险：若只在事务外先检查，再在事务内修改，就会留下竞态窗口。

多对象操作还会引入死锁问题。RucksFS 的做法是：凡是可能涉及多个 inode 的事务，都先收集全部相关 inode 编号，再按升序依次加锁。\texttt{rename} 是最典型的例子：源目录、目标目录、被移动对象以及潜在被覆盖对象都可能同时参与同一事务。统一的加锁顺序把“先锁谁、后锁谁”变成了确定规则，从而避免不同线程因加锁顺序不一致而形成循环等待。当前实现还为锁等待配置了 5 秒超时；若事务冲突无法立即消解，MetadataServer 会先在服务端执行有限重试，最多 3 次，并采用指数退避；若重试后仍然失败，再将其作为 \texttt{TransactionConflict} 向上返回，并在 FUSE 层映射为 \texttt{EAGAIN}。

\subsubsection{增量更新与读写一致性}

仅靠事务并不能解决所有热点问题。目录修改还会带来一个更隐蔽的瓶颈：父目录属性更新。按照 POSIX 语义，\texttt{create}、\texttt{unlink}、\texttt{mkdir} 和 \texttt{rmdir} 都会影响父目录的 \texttt{mtime}/\texttt{ctime}，而涉及子目录的操作还会影响 \texttt{nlink}。如果每次都直接读取并重写父目录 inode，那么在热点目录下，冲突会集中到同一条基值记录上，LSM-tree 的追加写优势也会被频繁的“读--改--写”抵消。

RucksFS 的处理方式是把这类更新从基值重写改为增量追加。写路径不直接修改父目录在 \texttt{inodes} 中的基础值，而是把变化编码为 \texttt{DeltaOp}，追加到 \texttt{delta\_entries} 中。当前增量类型包括 \texttt{IncrementNlink}、\texttt{SetMtime}、\texttt{SetCtime} 和 \texttt{SetAtime}。其中，普通文件的创建与删除只影响父目录时间戳；\texttt{mkdir}、\texttt{rmdir} 以及跨目录移动目录时，才会额外引入 \texttt{nlink} 增量。这样一来，热点目录上的高频更新就从覆盖同一条 inode 基值，转变为向独立增量记录追加。

读取路径则负责把“基础值 + 增量”重新合成为最新视图。MetadataServer 在加载 inode 时，优先从 \texttt{InodeFoldedCache} 读取已经折叠好的结果；若缓存未命中，则先读取 \texttt{inodes} 中的基础值，再扫描属于该 inode 的全部 \texttt{DeltaOp}，按顺序完成折叠。为了避免读路径的折叠成本无限增长，后台的 \texttt{DeltaCompactionWorker} 会周期性检查 dirty inode，并在增量条目超过阈值时，将折叠后的新基值写回 \texttt{inodes}，同时删除已消费的增量。当前实现的默认阈值为 32 条，检查周期为 5 秒。其核心思想可以概括为：用受控的读路径折叠成本，换取写路径上的冲突削减。与 Mantle 的 Delta Record 相比，RucksFS 的 DeltaOp 机制沿用了“以增量换写放大”的基本思路，但将追加、折叠和提交都约束在同一 RocksDB 实例内完成，因而实现更轻量。

这一机制带来的收益不容易在低并发下显形：父目录上的写争抢必须达到一定密度，覆盖式更新与增量追加的差距才会拉开。第~\ref{sec:delta-vs-nodelta}~节将用 Delta 与 NoDelta 两种编译变体的对照实验直接量化这一点——在共享父目录的元数据负载下，关闭 DeltaOp 的构建在客户端数从 32 增长到 64 时出现 40\% 的吞吐下滑，而启用 DeltaOp 的构建保持线性增长。

\subsubsection{删除语义与句柄生命周期}

事务原子性之外，元数据一致性还体现在删除语义上。POSIX 允许目录项已经消失，而已打开句柄仍继续访问原文件。要满足这一点，仅靠一次 \texttt{unlink} 事务并不够，因为名字空间可见性与数据回收时机并不总是重合。RucksFS 因此在 MetadataServer 中维护了 \texttt{open\_handles} 与 \texttt{pending\_deletes} 两类状态。\texttt{open()} 成功后，系统递增对应 inode 的打开计数；\texttt{release()} 则递减该计数。

当 \texttt{unlink} 或覆盖式 \texttt{rename} 使某个普通文件的 \texttt{nlink} 降到 0 时，MetadataServer 并不会一律立刻删除其数据。若该 inode 仍有打开句柄，系统只把它加入 \texttt{pending\_deletes}，并在最后一个句柄关闭时再通知客户端执行数据侧清理。这样，目录项删除、链接计数归零和物理数据回收就被明确区分为三个阶段：名字首先从命名空间中消失，可达性随后由 \texttt{nlink} 表示，而物理删除则延迟到没有进程再持有该对象时进行。对外部观察者而言，这正对应了 POSIX 所要求的”名字已删除，但已有句柄仍然有效”的语义。pjdfstest 中的 \texttt{unlink::open\_file\_not\_freed} 用例直接检验了这一行为，测试结果见第~\ref{sec:pjdfstest-semantic}~节。

\subsection{核心操作实现}
\label{sec:posix_impl}

RucksFS 的大多数元数据操作遵循同一条主线：先定位少量相关目录项键和 inode 记录，再在同一事务内完成检查与修改，并按需为父目录追加属性增量。本节逐个考察这些操作的实现流程，展示键编码、PCC 和 DeltaOp 三者如何被组合与复用。

\subsubsection{create 与 mkdir}

\texttt{create} 的难点并不在于“写入一个新 inode”，而在于同时满足三件事：目标名字必须原本不存在，新对象必须以完整状态出现，父目录的属性更新还要与前两者保持原子一致。若把这些动作拆成多个独立步骤，就很容易出现中间态，例如 inode 已经创建但目录项尚未建立，或者同名检查通过后又被其他并发事务抢先插入。算法~\ref{alg:create}给出了当前实现中的 \texttt{create} 流程。

\begin{algorithm}[htbp]
    \caption{\texttt{create(parent, name, mode, uid, gid)} 的核心流程}
    \label{alg:create}
    \begin{algorithmic}[1]
        \State 开启事务 batch
        \State 对目录项键 \texttt{D[parent, name]} 执行 \texttt{get\_for\_update}
        \If{目录项已存在}
            \State 返回 \textsc{EEXIST}
        \EndIf
        \State 分配新 inode 编号，构造普通文件的 \texttt{InodeValue}
        \State 向 \texttt{inodes} 列族写入新 inode 基值
        \State 向 \texttt{dir\_entries} 列族写入父目录到新文件的映射
        \State 向 \texttt{inodes} 列族写入该 inode 的 \texttt{DataLocation}
        \State 向 \texttt{delta\_entries} 追加父目录的 \texttt{SetMtime} 与 \texttt{SetCtime}
        \State 提交事务
        \State 更新本地缓存并返回新文件属性
    \end{algorithmic}
\end{algorithm}

从键空间组织的角度看，\texttt{create} 体现了边中心建模的优势：它不需要改写整条路径，也不需要搬移任何子树，只需在 \texttt{dir\_entries} 中插入一条新边，并在 \texttt{inodes} 中补齐目标对象的基础记录和数据位置记录。新对象的可见性由一条目录项记录与一组 inode 相关记录共同建立，而不是由某个单一结构承担。

另一方面，父目录属性并不通过覆盖写回基础 inode 来更新，而是追加 \texttt{SetMtime} 与 \texttt{SetCtime} 两条 \texttt{DeltaOp}。这一点把 \texttt{create} 从“目录项插入 + 父目录基值重写”的组合，变成了“目录项插入 + 少量新记录写入 + 增量追加”的组合。前者在热点目录下很容易把竞争集中到父目录 inode 上，后者则更符合 LSM-tree 擅长的追加写模式。

\texttt{mkdir} 与 \texttt{create} 共享同一事务骨架，但语义上比普通文件创建多出一层目录约束。新对象的类型位变为目录，初始 \texttt{nlink} 为 2；同时，父目录除了更新时间戳之外，还需要追加 \texttt{IncrementNlink(+1)}，因为新增了一个子目录。普通文件创建不会改变父目录 \texttt{nlink}，而目录创建会，这一差异看似细小，却直接决定了后续 \texttt{rename}、\texttt{rmdir} 和一致性检查的正确性。

\subsubsection{unlink、rmdir 与 release}

如果说 \texttt{create} 是“建立一条新边”，那么 \texttt{unlink} 和 \texttt{rmdir} 则分别对应“删除一条普通文件边”和“删除一条目录边”。二者的共同点是：都需要在事务内同时处理目录项可见性、目标对象状态以及父目录属性变化；它们的差异则来自 POSIX 对普通文件和目录的不同约束。

对 \texttt{unlink} 而言，核心工作流是：锁定目标目录项，确认其并非目录，删除这条目录项记录，递减目标 inode 的 \texttt{nlink}，再根据链接计数和打开句柄状态决定是否进入延迟删除分支。这里的关键收益在于，目录项删除和对象状态更新处于同一事务内，因此不会出现“名字已经消失，但链接计数还未更新”之类的中间态。父目录的 \texttt{mtime}/\texttt{ctime} 更新同样不直接重写基值，而是继续走增量日志路径。

\texttt{rmdir} 的难点在于空目录检查。这里若采用“先扫一遍目录、确认为空、再开启事务删除”的写法，检查结果在进入事务前就可能失效。RucksFS 因而把”是否为空”的判断也放进事务内部，通过一致性视图下的前缀扫描完成。若目录非空，则直接返回 \texttt{ENOTEMPTY}；若目录为空，则删除目标目录项与目标 inode，并对父目录追加 \texttt{IncrementNlink(-1)} 以及时间戳增量。空目录约束由事务本身保证，而非依赖上层约定。

\texttt{release} 则补上了删除语义的最后一环。\texttt{unlink} 和覆盖式 \texttt{rename} 可以让名字从命名空间中消失，但只要仍有进程持有打开句柄，物理数据就不能立刻清理。每次客户端关闭文件句柄时，\texttt{release} 都会递减打开计数；若某个 inode 先前已经进入 \texttt{pending\_deletes} 且此时计数降为 0，MetadataServer 才把该 inode 返回给客户端，通知后者继续在 DataServer 侧执行最终的数据删除。这样一来，名字可见性、元数据可达性和物理数据回收就被清晰地分成了三个阶段。

\subsubsection{rename}

\texttt{rename} 是前述设计选择汇合得最集中的操作。它既依赖“以边为中心”的键建模，也依赖统一的加锁顺序、覆盖删除语义以及父目录属性增量，因此可以视为整个元数据路径中最能体现系统设计取舍的一个操作。它可能同时涉及源目录、目标目录、被移动对象以及潜在被覆盖对象四类状态，且要求整个过程对外原子可见。算法~\ref{alg:rename}概括了这一流程。

\begin{algorithm}[htbp]
    \caption{\texttt{rename(old\_parent, old\_name, new\_parent, new\_name)} 的核心流程}
    \label{alg:rename}
    \begin{algorithmic}[1]
        \State 读取并锁定源目录项
        \State 若目标目录项存在，则读取其 inode 信息
        \State 收集全部相关 inode 编号并排序，按序执行 \texttt{get\_for\_update}
        \State 检查源对象与目标对象的类型兼容性
        \If{目标存在且为非空目录}
            \State 返回 \textsc{ENOTEMPTY}
        \EndIf
        \If{目标存在}
            \State 按覆盖删除逻辑处理目标对象
        \EndIf
        \State 删除源目录项，写入目标目录项
        \State 更新被移动对象的 \texttt{ctime}
        \If{移动的是目录且跨父目录}
            \State 对旧父目录追加 \texttt{IncrementNlink(-1)}
            \State 对新父目录追加 \texttt{IncrementNlink(+1)}
        \EndIf
        \State 对涉及的父目录追加时间戳增量
        \State 提交事务并返回被覆盖对象列表
    \end{algorithmic}
\end{algorithm}

从实现角度看，\texttt{rename} 的主体可以理解为”删除旧边、插入新边，并按需处理被覆盖对象”。这是边中心建模的直接收益：即使发生跨目录移动，系统也只需要改动少量与源路径和目标路径直接相关的记录，而不需要像全路径键方案那样对子树做级联改写。\texttt{rename} 的复杂性主要来自语义约束，而不是来自底层数据搬移。

排序加锁是 \texttt{rename} 能够避免死锁的关键，但它的意义不止于避免“互相等待”。更重要的是，统一的加锁顺序让源目录、目标目录以及被覆盖对象可以被纳入同一个稳定事务中，从而使“检查目标是否存在”“检查目标目录是否为空”“删除旧目录项”“写入新目录项”“处理覆盖删除”等动作保持同一一致性视图。对外部观察者而言，\texttt{rename} 因而只表现为一次原子切换，而不会暴露“旧名字已删、新名字未建”这样的中间态。

另一个容易被忽视的细节是，目录跨父目录移动时才需要调整父目录 \texttt{nlink}。移动普通文件不会影响任何目录的 \texttt{nlink}；同一父目录内对目录重命名，也不会影响父目录 \texttt{nlink}。这些差异直接决定了增量日志中应追加哪些 \texttt{DeltaOp}，也说明 \texttt{rename} 的正确性并不仅仅取决于“目录项有没有改对”，还取决于相关目录属性是否被同步维护。

\subsubsection{readdir、open 与写入协调}

\texttt{readdir} 是最直接体现键空间局部性的操作。给定父目录 inode 后，MetadataServer 构造前缀 \texttt{[D][parent: u64 BE]}，即可利用 RocksDB 的前缀迭代器顺序扫描该目录下的全部 dentry。由于目录项 value 中已经携带子对象的 \texttt{mode}，普通 \texttt{readdir} 不必再为每个条目回查一次 inode，就能得到文件类型信息。这说明前文在目录项 value 中额外保存 \texttt{child\_mode} 并不是冗余字段，而是专门为了降低目录遍历的放大成本。

\texttt{open} 则是元数据面和数据面的衔接点。它本身不承担数据读写，而是负责把“这个 inode 的数据应该去哪台 DataServer 读写”这一信息交给客户端。因此，\texttt{open} 返回的不只是句柄，还包括对应的 \texttt{DataLocation}。这样，后续 \texttt{read}、\texttt{write} 和 \texttt{truncate} 就不必每次都回到 MetadataServer 重新查询路由，而可以直接访问正确的数据后端。

写入完成后的元数据更新同样体现了 RucksFS 的职责划分：DataServer 只负责把字节写到正确位置，并不主动维护文件长度或 \texttt{mtime}；这些语义仍由 MetadataServer 统一掌握。因此，客户端在 \texttt{write\_data} 成功后，会显式调用 \texttt{report\_write(inode, new\_size, mtime)} 回写元数据。这种设计的好处是边界清晰：数据路径保持简单，元数据语义集中维护；代价则是客户端需要承担一次额外的协调动作。更完整的路由与服务编排细节将在下一节展开。
\subsection{数据服务与客户端协调}
\label{sec:data_path}

到上一节为止，本文讨论的重点仍然是”单个元数据操作如何在 MetadataServer 内部完成”。但在完整系统中，用户真正触发的是跨越 FUSE 客户端、MetadataServer 和 DataServer 的端到端路径。文件系统的正确性不仅取决于元数据事务本身，还取决于客户端如何在多个服务之间编排请求、如何维护句柄到数据位置的映射，以及如何把 FUSE 的可见语义落实到分离的元数据面与数据面上。本节因此转向系统级视角，讨论这条跨服务路径是如何组织起来的。

\subsubsection{VfsCore 路由与语义编排}

客户端内部的核心协调层是 \texttt{VfsCore}。它一方面向 FUSE 暴露统一的 \texttt{VfsOps} 接口，另一方面同时持有 \texttt{MetadataOps} 与 \texttt{DataOps}，负责决定一次请求究竟应当落到哪个服务、是否需要跨服务串联，以及返回结果后还需要补哪些后续动作。\texttt{VfsCore} 不是简单的 RPC 转发器，而是整条端到端路径的编排者。

从路由角度看，请求大致可以分为三类。第一类是纯元数据操作，如 \texttt{lookup}、\texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rmdir}、\texttt{rename}、\texttt{readdir}、\texttt{getattr} 和 \texttt{setattr}，它们直接交由 MetadataServer 处理。第二类是纯数据操作，如 \texttt{read}、\texttt{write}、\texttt{truncate} 和 \texttt{fsync}，它们在文件已打开的前提下可以依据句柄中缓存的 \texttt{server\_id} 直接路由到对应的 DataServer。第三类则是跨服务复合操作，例如 \texttt{open} 需要先从 MetadataServer 取回 \texttt{DataLocation}，而 \texttt{release}、覆盖式 \texttt{rename} 或可能触发截断的 \texttt{setattr} 则需要在元数据结果返回后，再由客户端补发数据侧清理或截断请求。

这种编排方式有一个明确的设计意图：把”只涉及名字空间和 inode 状态的语义”约束在 MetadataServer 内，把”真正写字节、截断数据、清理数据”的工作留在 DataServer 内，而把跨两侧的复合语义交由客户端显式协调。这种职责分配使边界清晰：MetadataServer 专注于元数据事务，DataServer 专注于字节读写，\texttt{VfsCore} 则负责把二者缝合成用户所见的 POSIX 行为。当然，客户端会因此比传统本地文件系统中的薄代理更复杂，但这是分离式架构下不可回避的一层。

\subsubsection{DataServer 与原始磁盘后端}

在数据侧，RucksFS 刻意保持了一条非常直接的实现路径。DataServer 只是 \texttt{DataStore} 的服务化封装层，当前唯一的数据后端是 \texttt{RawDiskDataStore}。它把所有 inode 的内容映射到同一个 \texttt{data.raw} 文件中，并按固定公式
\[
\text{disk\_offset} = \text{inode} \times \text{max\_file\_size} + \text{file\_offset}
\]
计算物理偏移；随后通过 \texttt{pread} 与 \texttt{pwrite} 完成读取和写入。DataServer 并不维护复杂的块映射结构，而是把 inode 编号直接解释为数据文件中的一个逻辑槽位。

这种固定地址映射的优点非常直接：数据路径不需要空闲空间位图、extent 树或块分配器，寻址是常数时间，服务端状态也非常简单。对于本文关注的“元数据关键路径优化”而言，这种极简设计还有一个额外收益：它把数据侧复杂度尽量压低，使第~\ref{sec:experiments}~章中的性能结果更容易归因于元数据组织、客户端协调和网络通信，而不是被复杂的数据块管理逻辑所掩盖。

当然，这一设计也有明显局限。每个 inode 都会预留 \texttt{max\_file\_size} 大小的逻辑地址空间，当前默认值为 128\,MiB，因此大量小文件会带来很低的空间利用率。此外，\texttt{delete\_data} 目前仍是空操作，这意味着对象删除后物理空间不会立即回收。本文之所以仍采用这一方案，并不是忽视空间管理，而是有意把工程复杂度留给后续扩展，以换取当前原型在数据路径上的可解释性与实现简洁性。

并发语义方面，由于写入偏移由调用方显式指定，不同客户端若写入同一 inode 的不同偏移，不会因为 DataServer 内部再分配地址而产生额外竞争；\texttt{pread}/\texttt{pwrite} 也保证每次 I/O 以调用为边界完成。若不同写入请求本身覆盖了重叠区域，则那属于上层应用的并发语义问题，而不是 DataServer 需要额外裁决的状态冲突。

\subsubsection{FUSE 可见语义与缓存策略}

在用户态文件系统中，很多”性能表现”实际上首先体现为”内核缓存会如何看待这个文件系统”。因此，本节需要在实验之前把 FUSE 层的几个关键可见语义讲清楚，否则第~\ref{sec:experiments}~章中的结果很难被正确理解。RucksFS 在这里主要控制三件事：目录项缓存、属性缓存以及目录遍历接口的选择。

第一，目录项缓存（entry cache）决定了内核在多长时间内相信“名字到 inode”的映射仍然有效。RucksFS 在 \texttt{lookup} 响应中设置了 1 秒的 entry TTL，使目录变更不会被过久地隐藏在内核缓存之后。第二，属性缓存（attr cache）决定了 \texttt{stat}、\texttt{getattr} 等请求的放大程度。RucksFS 默认同样使用 1 秒的属性 TTL，以在一致性和 RPC 次数之间取得折中。第三，客户端显式启用 \texttt{readdirplus} 路径，使目录遍历能够在返回目录项的同时附带基本属性，从而减少“先 \texttt{readdir}、再对每个条目逐个 \texttt{getattr}”带来的往返开销。

除了缓存之外，FUSE 语义中还有两个与分离式架构密切相关的点。其一，\texttt{open} 在 RucksFS 中不仅产生句柄，还承担把数据路由信息交给客户端的职责；其二，\texttt{fsync} 当前只保证数据侧 \texttt{flush}，元数据仍依赖 RocksDB 的事务提交与 WAL 持久化，而尚未实现严格的双阶段刷写。这意味着 RucksFS 对 FUSE 的实现并不是把内核接口逐字照搬到远端服务，而是根据元数据面/数据面分离后的职责边界，对各个接口给出可以落地的语义解释。

\subsubsection{gRPC 接口与模块边界}

在这条端到端路径上，gRPC 不只是通信细节，它同时决定了“哪些状态应由哪个服务直接返回”“哪些后续动作需要客户端继续协调”。因此，接口设计本身也是系统结构的一部分。RucksFS 在接口层面遵循两个原则：一是元数据操作和数据操作严格分离；二是一次 RPC 尽量返回下一个阶段所需的全部信息，避免把简单操作拆成多轮往返。

据此，MetadataServer 的接口大体分成四组：命名空间操作（如 \texttt{Lookup}、\texttt{Create}、\texttt{Rename}、\texttt{Readdir}），属性操作（\texttt{GetAttr}、\texttt{SetAttr}），句柄相关操作（\texttt{Open}、\texttt{Release}），以及写入后的元数据协调（\texttt{ReportWrite}）。DataServer 的接口则刻意保持收敛，只暴露 \texttt{Read}、\texttt{Write}、\texttt{Truncate} 和 \texttt{DeleteData} 这类与字节内容直接相关的操作。这样的划分，使“谁负责对象语义、谁负责数据内容”在接口层面就已经被固定下来。

消息格式同样服务于这一目标。元数据响应通常直接携带完整的 \texttt{FileAttr}、\texttt{DataLocation}、\texttt{purged\_inodes} 或 \texttt{needs\_truncate} 等信息，避免客户端为了完成一个上层操作再额外发起新的探测 RPC。连接层面，当前原型采用每客户端单 gRPC 通道、共享 HTTP/2 连接的方式，以控制实现复杂度。这种选择并非最终形态，但足以支撑本文所关注的元数据语义验证与原型级性能评估。

\texttt{VfsCore}、RawDiskDataStore、FUSE 缓存设置与 gRPC 接口共同组成了 RucksFS 的跨服务执行路径。上一节讨论的是”单个操作在元数据层如何正确完成”，本节补足的是”这些操作在完整系统中如何被协调并暴露给内核与应用”。

\subsection{错误处理与恢复机制}

一旦系统从单进程存储引擎扩展到”客户端 + MetadataServer + DataServer”的三段式路径，正确性问题就不再只来自事务冲突，还来自 RPC 失败、服务崩溃和跨服务状态不一致。由于第~\ref{sec:experiments}~章主要评估正确性与性能结果，而不再解释系统行为本身，因此这里需要先把故障模型和恢复边界讲清楚。

\subsubsection{事务冲突、RPC 失败与错误码映射}

最常见的一类失败是 MetadataServer 内部的事务冲突。对此，RucksFS 不会立即把 RocksDB 的底层错误暴露给用户，而是先在服务端执行有限重试：最多 3 次，并采用指数退避。只有在重试后仍然失败时，才将其作为 \texttt{TransactionConflict} 向上返回，并在 FUSE 层映射为 \texttt{EAGAIN}。这使偶发冲突不会直接转化为用户可见错误，同时也让错误语义保持明确：\texttt{EAGAIN} 表示的是“事务冲突仍未消解”，而不是任意 I/O 异常。

另一类失败来自普通 RPC 或 I/O 调用。当前原型对此采取了更保守的策略：客户端根据 gRPC 返回的错误状态向上报告相应文件系统错误，而不会在数据路径上额外维护待确认写队列，也未实现基于 \texttt{inode + offset} 的透明幂等重发。简言之，RucksFS 当前具备的是”事务冲突自动重试”，而不是”任意跨服务失败都自动恢复”。这一边界需要明确说明，否则读者会误以为系统已经实现了更强的故障屏蔽能力。

\subsubsection{服务崩溃后的恢复语义}

当 MetadataServer 崩溃时，其持有的 RocksDB TransactionDB 会依赖 WAL 执行恢复：已经提交但尚未刷入 MemTable 的内容会在重启后恢复，未完成提交的事务则不会伪装成已成功操作。这意味着，只要某次元数据操作已经向客户端返回成功，它对应的持久化结果就不会因为 MetadataServer 进程退出而丢失。客户端侧则会把崩溃表现为一次 RPC 失败或重试；必要时，像 \texttt{open} 这样的操作可以重新向 MetadataServer 获取新的 \texttt{DataLocation}。

DataServer 的崩溃语义更简单，也更受当前实现边界约束。若负责某个文件的 DataServer 不可达，则对应的 \texttt{read} 或 \texttt{write} 会直接向上返回 \texttt{EIO}。由于当前原型没有引入数据副本或多 DataServer 故障切换，数据服务故障不会被透明掩蔽，而是直接暴露为该文件暂时不可访问。由于 \texttt{delete\_data} 目前仍是空操作，删除路径也不存在“删除已提交但物理空间未及时回收”的额外恢复问题。

\subsubsection{当前实现边界与论文讨论范围}

在介绍完正常路径和故障路径之后，还需要明确本文的讨论边界。RucksFS 当前有意保持“单 MetadataServer + 单默认 DataServer”的主路径。虽然 \texttt{DataLocation} 与 \texttt{VfsCore::with\_data\_servers()} 已经为多 DataServer 路由预留了接口，但元数据分片、多副本一致性、物理空间回收以及严格双阶段 \texttt{fsync} 都尚未纳入本文实现范围。

这一定义并不是缺省性的”未做完”，而是本文研究目标的一部分：本论文更关注在元数据面与数据面分离的前提下，如何通过键编码、事务控制、客户端协调和增量更新优化元数据关键路径，而不是讨论完整的大规模分布式控制面。也正因为如此，第~\ref{sec:experiments}~章中的实验结果主要验证的是这套设计在当前边界内的正确性与性能特征，而不意图证明系统已经覆盖多副本容错、跨节点故障转移或空间回收等更大范围的工程能力。

\subsection{本章小结}

本章从系统实现的角度完整展开了 RucksFS 的设计。围绕 inode 与 dentry 两个核心对象，本文说明了元数据如何被组织到不同列族中，并通过边中心键建模把目录树映射到 RocksDB 键空间。借助悲观事务、统一加锁顺序和 DeltaOp 增量更新机制，系统把目录操作中的原子性、并发正确性和热点目录写放大问题统一纳入同一套元数据执行路径中。

在此基础上，本章讨论了这些机制如何落到具体操作实现：\texttt{create}、\texttt{mkdir}、\texttt{unlink}、\texttt{rmdir}、\texttt{rename}、\texttt{readdir} 与 \texttt{open} 共同展示了前文的键编码、事务与删除语义并不是彼此孤立的设计点，而是在真实 POSIX 操作中被反复组合和复用。\texttt{VfsCore} 的路由编排、RawDiskDataStore 的简化数据路径、FUSE 缓存策略以及 gRPC 接口边界，又把这些单点机制拼接成了一条端到端的跨服务执行链路。

最后，本章明确了系统在故障处理和实现范围上的边界：RucksFS 已经具备完整的元数据事务路径、客户端协调逻辑和可解释的错误语义，但仍然停留在单 MetadataServer、单默认 DataServer、无副本、无物理空间回收的原型阶段。到这里为止，第~\ref{sec:experiments}~章所需要依赖的设计前提、语义假设和系统边界都已经交代完毕；接下来便可以转入测试与实验，通过 pjdfstest 合规性测试、mdtest 多系统对比和 Delta/NoDelta 消融实验，验证这些设计在分布式部署下的正确性和有效性。
