# 论文写作 Agent 规范

本文件约束所有参与论文写作和修改的 Agent 的行为。

## 1. 总体目标

论文写作的首要目标不是“像不像 AI”，而是让文本满足三项要求：

1. 事实准确，和代码、实验、图表保持一致。
2. 中文自然，读起来像中文论文，而不是把英文技术文档逐句翻译过来。
3. 结构清楚，读者能迅速理解“为什么这样设计、具体怎么做、这样做的代价是什么”。

任何为了“反 AI 检测”而牺牲准确性、可读性或学术规范的做法，都是不允许的。

## 2. 反 AI 检测与语言规范性的平衡（核心原则）

**反 AI 检测和学术写作规范是需要平衡的两个目标，不可为了其中一个牺牲另一个。**

### 优先级排序

1. **语义准确、逻辑清晰**：最高优先级。读者能顺畅理解是底线。
2. **学术中文规范**：句式完整、用词准确、段落结构合理。
3. **反 AI 检测**：在前两条满足的前提下，避免 AI 特征模式。

### 具体原则

- **不为反检测而制造混乱**：如果一个句子用“第一…第二…第三…”结构最清晰，就用这个结构。不要为了“打破对称”而强行打乱逻辑顺序。
- **不用口语替代学术表述**：反 AI 不等于口语化。“这个问题不好办”不如“该问题的工程复杂度较高”。
- **自然参差优于刻意参差**：句式长短交替应该来自内容本身的需要，而非人为制造。
- **作者判断要有依据**：直说“过度设计”可以，但必须跟上理由，不能只有态度没有论据。

### 反 AI 检测规则（在规范性前提下遵守）

| 编号 | 规则 | 说明 |
|------|------|------|
| R1 | **砍模板化开头** | 禁止“随着……的不断发展”“近年来 X 的快速发展带来了 Y 的 Z”式开头。用具体事实、数据或问题直接切入。 |
| R2 | **避免高度对称的句式** | 不宜连续三段以上使用完全相同的句式骨架。但如果并列结构是最清晰的表达方式，允许保留。 |
| R3 | **减少路标式过渡** | 尽量不用“本章首先…随后…最后…”。如逻辑跳跃较大，可用一句短过渡。 |
| R4 | **加入作者判断** | 该说“不足”就直说，但必须附带依据。避免无依据的主观评价和永远中立的罗列式写法。 |
| R5 | **减少泛化断言** | 避免“已被广泛应用”“具有重要意义”等空话。要么删掉，要么换成带数据或引用的具体说法。 |
| R6 | **自然的语气变化** | 在严谨表述中可以穿插简短判断或直接转折，但前提是不破坏语义连贯性。 |

### AI 高频词黑名单（优先替换）

- “进一步” → 删掉，或用“在此基础上”
- “从根源上” → 换为具体描述
- “表明” → 交替使用“说明”“印证”“可以看出”“数据显示”
- “值得注意的是” → 直接陈述
- “不可忽视” → 用数据说话
- “与此同时” → 删掉或换为“此外”“另一方面”
- “彻底” → 通常删掉

## 3. 中文行文规则（核心）

这是所有 Agent 最容易做差的一部分，必须严格遵守。

### 3.1 作者身份设定（写作风格的”度”）

本文作者是**一名大四本科毕业生**，所有行文必须与这一身份相符。具体表现为：

- **行文以书面语为主**，陈述事实用数据说话，解释原因用”一是…二是…”或”其一…其二…”。
- **术语控制在必要范围内**。该用技术名词的地方直接用（如 \texttt{AtomicWriteBatch}、Percolator 两阶段提交、\texttt{i\_rwsem}、LSM-tree），不翻译、不回避、也不加评价性形容词。
- **不使用学究化的抽象名词或组合词**（如”口径””可比区间””受控消融””归因””结构性优势””机理””端到端””本质上””偏置””窄口径””核心假设”）。
- **不使用口语或网络化表达**（如”跑不动了””挺稳定””就完事了””省掉了””冲回去抢””一路跳到””实实在在””反正””算是””顺手””一并””挺…的”）。
- **语气平实克制**：不用”更关键的是””值得注意的是”这类评论语气；需要强调时直接说明原因或给出依据。
- **承接用词克制**：用”另外””此外””相比之下””需要说明的是””不过”即可，避免路标式导游词。

### 3.2 “度”的对照表（关键参考）

同一件事的三种写法，**以最右列为标准**：

| 过于口语（不接受） | 过于学究（不接受） | **目标（采用）** |
|---|---|---|
| 两套对照系统都不太跑得动了 | 两套对照系统均失去可比性 | JuiceFS+TiKV 和 NFS 都无法继续给出稳定的数据 |
| 省掉了 Raft 和两阶段提交开销 | 规避了 Raft 与两阶段提交的结构性开销 | 不需要承担 Raft 和两阶段提交这一层开销 |
| 反映的更接近两种方案本身的区别 | 反映的是两种方案在元数据写入上的结构性差异 | 更能反映两种方案在元数据写入路径上的差别 |
| 这件事在测试的关键路径之外 | 该动作发生在测试的关键路径之外 | 该后台过程不出现在测试的主要路径上 |
| 一路跳到 3.15、5.81 | 比值跃升至 3.15、5.81 | 比值依次上升到 3.15 和 5.81 |
| 父 inode 写锁就变成了 stat 的隐性瓶颈 | 父 inode 写锁构成 stat 的隐性瓶颈 | 父 inode 的写锁在这一情况下也会影响到 stat 的执行 |
| 挺干净的”崩溃 vs 延续”对照 | 一组清晰的对照性消融实验 | 一组对比明显的实验结果 |
| 整体看，用 KV 做元数据…确实更合适 | KV 方案相对 B+ 树存在结构性优势 | 这些数据表明，使用 KV 存储管理元数据在共享父目录场景下相对传统的 B+ 树方案更有优势 |
| 两个版本自己和自己比 | 对两种构建变体进行受控消融 | 让 Delta 和 NoDelta 两个版本做对比 |
| 跑起来咋样 | 该设计在实际运行环境中的行为 | 在真实部署环境下的表现 |

### 3.3 基本要求

- **按中文表达，不按英文语序翻译**：不要写成”路径中的 `/a/b` 不是一个单独存储的整体实体”这种翻译腔。优先改成”文件系统不会把整条路径作为一个对象保存”。
- **一段只讲一个中心意思**：先说结论，再展开原因或例子。不要一个段落里同时讲定义、动机、实现、评价四件事。
- **句子主谓宾明确**：避免”其””这一点””这种方式”连续指代。
- **先解释概念，再解释作用**：不要一上来就讲抽象判断。
- **少写悬空判断句**：像”这种拆分是合理的””该设计更清晰”都需要补一句理由。
- **”本文/本章”替代”我们”**，保持论文体；”笔者”也可以使用，但不过度。

### 3.4 推荐句式

- 陈述事实：`在 X 条件下，Y 的取值为 Z。`
- 解释原因（两条）：`之所以如此，一是……；二是……。`
- 解释原因（更多条）：`原因可以归纳为以下几点：其一，……；其二，……；其三，……。`
- 解释取舍：`这样做的好处是……，代价是……。`
- 解释流程：`操作先……，随后……，最后……。`
- 引出例子：`以 …… 为例，……。`
- 引出图表：`表~\ref{tab:xxx} 给出了 ……。`、`如图~\ref{fig:xxx} 所示，……。`
- 说明局限：`需要说明的是，……。`

### 3.5 禁止句式与词汇

**学究化词汇黑名单**（首选删除或替换）：

| 学究词 | 改法 |
|---|---|
| 口径、对比口径、实验口径 | 说明、条件、前提 |
| 可比区间、可比性 | 具体写 “$N \in \{2, 8, 32\}$ 三档” |
| 受控消融、消融实验 | “Delta 与 NoDelta 的对比”、”两个版本的对比” |
| 归因、可归因 | “原因来自…” |
| 结构性差异/优势 | “两种方案在 X 上的区别”、”在 X 上更有优势” |
| 机理 | “原因””工作方式” |
| 端到端 | 具体改为”从客户端到服务端””从 FUSE 到 RocksDB”等 |
| 偏置 | “影响””系统性误差” |
| 收敛（用作比喻时） | “减少””缩小” |
| 量化（用作动词） | “衡量””度量””给出具体数值” |

**口语化表达黑名单**：

| 口语 | 改法 |
|---|---|
| 跑不动、塌了、垮了 | “吞吐无法继续增长””出现明显下降” |
| 挺、特别、蛮 | 删掉或改成”较””相对” |
| 就完事了、就 OK 了 | 直接陈述结果 |
| 顺手、顺带、一并（口语意义下） | “同时””此外” |
| 反正、其实（口语意义下） | 通常删掉 |
| 冲回去抢 | “重新进入竞争” |
| 咋样、咋办 | “如何””怎样” |
| 就是、也就是说（口语意义下） | “即” 或删掉 |

**其他禁止句式**：

- **翻译腔抽象句**：如”不是一个单独存储的整体实体””并非一个天然的统一对象”。
- **空泛判断句**：如”这一设计更具合理性””该方案更有优势”。
- **技术名词堆叠**：一整句只堆模块名、接口名、缩写，不解释关系。
- **先结论后失语**：如”这是一个关键设计点”后面没有说明”关键在哪里”。

### 3.6 破折号控制

- **全角破折号 `——` 和英文 em-dash `---` 都是 AI 高频标点**，容易被识别为机器痕迹，且过度使用会打断中文句子的自然节奏。
- **默认不使用破折号**，优先使用衔接词或拆分为独立句子：
  - 解释性补充 → 「即」「也就是」「具体来说」，或独立成句；
  - 举例展开 → 「例如」「比如」，或冒号引出；
  - 转折 → 「不过」「但是」「另一方面」；
  - 强调/插入语 → 括号「（……）」或独立短句；
  - 排比列举 → 分号「；」或顿号「、」。
- **仅在以下场景可以保留破折号**：
  - 数值区间（如 `2--3 倍`、`60\%--80\%`，LaTeX 中用短横 `--`）；
  - 确有必要的强烈停顿或插入，整段最多出现一次。
- **英文摘要同步去除 `---`**，改成逗号 / 分号 / 冒号 / 独立句，或用 `, giving ...`、`, a ...` 之类的同位语结构。

### 3.7 图表引导语要求

- 图表前必须有一到两句引导，说明”为什么这里需要这张图/表”。
- 图表后必须有一段解释，说明”图里最重要的信息是什么”。
- 不允许出现”见下图即可””如图所示，不再赘述”。
- 图和表不能重复表达同一层信息：
  - 图优先表达结构、层级、流程。
  - 表优先表达字段、分类、对比、编码项。

## 4. 专业性与章节组织

- **学术中文**：全文使用规范学术中文，避免口语化、营销化表述。
- **术语一致**：同一概念全文使用同一译名和缩写，首次出现给全称。
- **符号一致**：同一变量全文使用同一符号。
- **段落控制**：单段通常不超过 220 字；单句通常不超过 50 字，过长则主动断句。
- **过渡自然**：章节之间用内容逻辑衔接，不用“下面将介绍……”式导游词。
- **标点规范**：正文中文标点，公式和代码内英文标点。数字与单位间加 `\,`（如 `64\,MB`）。

### 4.1 设计类章节的推荐结构

对“系统设计与实现”这类章节，优先采用以下顺序：

1. **先讲设计对象是什么**：例如系统角色、元数据对象、命名空间模型。
2. **再讲为什么这样设计**：例如访问模式、事务需求、性能瓶颈。
3. **再讲具体如何落到数据结构和接口**：例如列族、键编码、事务流程。
4. **最后讲代价、边界和未完成部分**。

不要一开始就进入代码细节，也不要在尚未定义对象前先讨论优化。

### 4.2 实现事实落点要求

- 凡是和代码强相关的表述，都要能在代码中找到直接依据。
- 但论文不能写成代码注释汇编。出现接口名、结构体名、crate 名时，必须说明其论文语义。
- **接口名只在必要时出现**：如果一句话去掉 `OpenResponse`、`VfsCore` 之后仍能说明问题，就优先用论文语言表达。

## 5. 引用与数据真实性（红线）

- **所有引用的论文必须真实存在**：不允许编造论文标题、作者、会议名或年份。如果不确定某篇论文是否存在，宁可不引用。
- **所有实验数据必须来自实际测试结果**：不允许编造性能数据。摘要和正文中的 ops/s、倍数、百分比必须与实验章节的表格一致。
- **引用数量要求**：全文引用论文不少于 25 篇（当前 `ref.bib` 有 39 条，足够）。
- **数据交叉验证**：同一数据在摘要、正文、表格中出现时必须一致。TiKV 对比倍数为 1.7（9917/5908），不是 1.5。

## 6. LaTeX 规范

- 表格使用 `booktabs` 三线表，标题在表上方。
- 图片标题在图下方。
- 引用格式：`图~\ref{fig:xxx}`、`表~\ref{tab:xxx}`、`第~\ref{sec:xxx}~节`。
- 算法使用 `algorithm` + `algorithmic` 环境。
- 代码片段使用 `verbatim` 或 `minted`（如已配置）。
- 图表过宽时优先重构内容，不要先用强行缩放糊过去。

## 7. 文件操作规范

- 修改 `.tex` 文件前必须先读取当前内容。
- 优先做定向修改；如果用户明确要求重写章节，可以整体重写，但仍需保持实现事实不变。
- 修改后确认前后文衔接、图表编号和引用关系正确。
- 中英文摘要保持同步。
