PART 1

研究背景与动机

为什么要用 KV 存储来管理文件元数据？
背景

什么是文件元数据？为什么它如此重要？

文件系统的"两类数据"

文件系统管理的信息天然分为两部分：

数据 (Data)

文件的实际内容——文档文字、图片像素、代码文本等。操作：read / write。

元数据 (Metadata)

描述文件本身的信息——文件名、大小、权限、时间戳、目录结构等。操作：create / stat / mkdir / readdir / rename / unlink / chmod ...

关键事实：每一次文件访问都必须先经过元数据操作（路径解析 → lookup → 权限检查 → 打开文件）， 元数据是文件系统所有操作的入口。

海量小文件场景加剧问题

现代应用产生海量小文件：AI 训练数据（数十亿小图片）、容器镜像层（万级文件）、微服务日志、IoT 传感器数据。 文件越小、越多，元数据操作占比越大。

性能瓶颈的位置

HDFS NameNode 单机管理极限约 10 亿文件。 阿里巴巴 Pangu 2.0 文件系统论文报告元数据节点承受 数亿 QPS 压力。 元数据管理是文件系统扩展性的核心瓶颈。

问题

传统文件系统为什么管不好海量元数据？

ext4 / XFS 的元数据存储方式

传统文件系统在磁盘上使用固定布局的数据结构来管理元数据：

数据结构作用问题
Inode 表存储文件属性格式化时预分配，数量固定，用完无法扩展
B-tree / htree目录索引每次插入可能触发节点分裂，随机写放大 ~2x
块位图空间分配与 inode 表、目录索引分散在不同磁盘区域
日志 (Journal)崩溃一致性每次元数据修改需先写日志再写数据，双重写入

一次 create 操作的实际开销

在 ext4 上创建一个空文件需要：①分配 inode（读位图→改位图→写位图）→ ②初始化 inode（写 inode 表）→ ③添加目录项（读目录块→插入 B-tree→可能分裂→写目录块）→ ④写 journal 日志。涉及至少 4-6 次磁盘写入，且位置分散。

四个结构性瓶颈

① B-tree 随机写放大

每次创建/删除文件需修改 B-tree 目录索引。节点分裂产生随机写 I/O。 百万文件级别下 B-tree 深度增加，查找开销从 O(log n) 实际劣化。

② 大目录性能退化

ext4 htree 在单目录 >100 万文件时 lookup 性能严重退化。 readdir 需遍历整个 htree，无法高效前缀扫描。

③ 元数据布局分散

inode 表、目录块、位图分布在磁盘不同位置。一次 stat 可能需要多次寻道。 SSD 虽缓解了寻道延迟，但随机 I/O 仍消耗大量 IOPS。

④ 固定容量，无法弹性扩展

Inode 数量在 mkfs 时确定，用完后无法创建新文件——即使磁盘空间充足。

方案

解决思路：用 LSM-tree KV 存储替代传统元数据管理

LSM-tree 如何逐一解决传统瓶颈

传统瓶颈KV 存储 (LSM-tree) 的解法
B-tree 随机写放大 所有写入先追加到内存 MemTable，后台异步刷盘为有序 SST 文件。随机写变顺序写，写入带宽提升 3~10x
大目录性能退化 目录项以 parent_inode + name 为 Key，天然有序。readdir = 按前缀扫描，O(n) 顺序读取，不受目录大小影响
元数据布局分散 所有元数据集中存储在 KV 引擎中，空间局部性极佳。inode 和目录项相邻存放，一次 I/O 可获取多条记录
固定容量无法扩展 KV 存储动态增长，无需预分配。理论上仅受磁盘容量限制

元数据操作的天然适配性

为什么元数据特别适合 KV

  • 写密集 — 创建/修改操作占比 >60%，LSM-tree 写入优势最大化
  • 记录小 — inode ~100-200B，远小于 KV 引擎的 block size，命中率高
  • 前缀扫描 — readdir 映射为按 parent_inode 前缀扫描，利用 SST 有序性
  • 原子批量写 — WriteBatch 天然支持多键原子提交，一次 create 的多个写入可以聚合

第一阶段：学术验证 (2013-2017)

TableFS 首次证明 LSM-tree 管理元数据可行——100 万文件创建比 ext4 快 5 倍。 IndexFS 扩展到 128 MDS 节点，吞吐提升 50-100x。 LocoFS 通过目录/文件元数据解耦，将 KV 利用率从 18% 提升到 93%。

第二阶段：工业落地 (2018-2020)

JuiceFS 用 Redis/TiKV 管理元数据，在生产环境广泛验证。 Hadoop Ozone 用三级 RocksDB 突破 HDFS 10 亿文件限制。 BFO 深入分析批量操作中的元数据开销，提出预取优化。

第三阶段：极致优化 (2023-2025)

SingularFS 证明精心优化的单机 MDS 性能可超过多节点分布式方案。 Mantle 提出 Delta Record + IndexNode 缓存，避免高频读-改-写，吞吐提升 115 倍。

设计要点

单二进制、零网络开销

元数据引擎、数据引擎、FUSE 客户端运行在同一进程内。 类似 JuiceFS 的单 binary 架构，所有操作均为本地函数调用。

Trait-Based 分层解耦

参考 JuiceFS 的 Meta Engine 可替换设计。 通过 MetadataOps / DataOps / VfsOps 三层 trait 定义接口，存储后端可独立替换。

Rust 实现

10,000+ 行代码。利用 Rust 所有权模型保证并发安全，无 GC 停顿。

FUSE 权限委托

使用 default_permissions + allow_other 挂载选项，由内核 VFS 层处理 POSIX 权限检查。

核心设计

文件命名空间到 KV 存储的映射

举例：一个目录树如何存入 KV？

考虑以下目录结构：

/  (inode 1, 目录)
 ├── home/  (inode 2, 目录)
 │   └── alice/  (inode 3, 目录)
 │       ├── notes.txt  (inode 10, 文件, 1024B)
 │       └── photo.jpg  (inode 11, 文件, 2MB)
 └── etc/  (inode 4, 目录)

映射为 KV 后的存储内容

CF: inodes — 每个文件/目录一条记录

Key: [1] → { mode: dir, nlink: 4, mtime: ... }
Key: [2] → { mode: dir, nlink: 3, mtime: ... }
Key: [3] → { mode: dir, nlink: 4, mtime: ... }
Key: [10] → { mode: file, size: 1024, uid: alice }
Key: [11] → { mode: file, size: 2MB, uid: alice }

CF: dir_entries — 每个"父→子"关系一条记录

Key: [1|"home"] → { child: 2, type: dir }
Key: [1|"etc"] → { child: 4, type: dir }
Key: [2|"alice"] → { child: 3, type: dir }
Key: [3|"notes.txt"] → { child: 10, type: file }
Key: [3|"photo.jpg"] → { child: 11, type: file }

这种映射为什么高效？

lookup /home/alice/notes.txt

逐级查询：[1|"home"]→2 → [2|"alice"]→3 → [3|"notes.txt"]→10 → 读 inode [10]。 每一级只需一次精确 KV get，无需遍历。

readdir /home/alice/

前缀扫描 [3|*]，返回所有以 inode 3 为前缀的 Key。 RocksDB SST 文件有序，前缀扫描 = 顺序读取，极其高效。

create /home/alice/new.txt

原子写入：① put inode [12] ② put dir_entry [3|"new.txt"]→12。 WriteBatch 一次提交，保证原子性。

核心设计

Delta 增量更新与操作映射

问题：高频属性修改的写放大

在 /home/alice/ 下连续创建 1000 个文件，每次都需要更新父目录 (inode 3) 的 mtime 和 nlink。 传统做法：每次 读取 57B → 修改 → 写回 57B，1000 次 = 1000 次读-改-写。

方案：Delta Record 追加更新

受 Mantle (SOSP'25) 的 Delta Record 机制启发

create("f1") → append Δ(mtime=T1, nlink+1)
create("f2") → append Δ(mtime=T2, nlink+1)
create("f3") → append Δ(mtime=T3, nlink+1)
getattr(parent) → fold: base + Δ₁ + Δ₂ + Δ₃ → 最终属性

三层架构

  • InodeFoldedCache (LRU) — O(1) 命中返回折叠后属性
  • Fold-on-Read — 缓存未命中时读 base + delta 折叠
  • 后台 Compaction — delta 超过 32 条时异步合并为新 base

POSIX 操作 → KV 操作映射

文件操作KV 操作序列
create ① get_for_update 检查重名 ② put inode ③ put dir_entry ④ commit ⑤ append delta(mtime)
lookup ① get dir_entry(parent+name) ② get inode(child)
readdir ① prefix_scan dir_entries(parent)
unlink ① delete dir_entry ② update inode(nlink-1) ③ nlink=0 则 delete inode
rename ① delete src_entry + put dst_entry ② append delta(both parents)
mkdir ① put inode(dir) ② put dir_entry ③ append delta(parent nlink+1)
rmdir ① check dir empty (prefix_scan) ② delete entry + inode ③ delta(nlink-1)

关键设计原则

  • 关键路径最小化：delta 追加在事务外，丢失仅影响 mtime
  • 双写消除：shares_batch_storage 检测避免 CF 重复写入
  • 定长编码：InodeValue 57B 定长，借鉴 LocoFS 零序列化
并发控制

事务化元数据管理与并发安全

为什么需要事务？

举例：并发 rename 的竞态

线程 A：rename("a", "c") — 删除源目录项 + 写入目标目录项
线程 B：rename("b", "c") — 删除源目录项 + 写入目标目录项
如果不加事务保护，两个线程可能同时写入目标 "c"，导致一个文件丢失。

PCC 悲观并发控制

// create("new.txt") 事务流程
1. begin_transaction()
2. get_for_update(dir_entry[3|"new.txt"])
   → 加行锁，若已存在返回 EEXIST
3. batch_put(inode[12], value)
4. batch_put(dir_entry[3|"new.txt"], 12)
5. commit() → 原子写入 WAL
6. append_delta(parent_mtime) // 事务外

TOCTOU 防护

rmdir 在事务内执行 prefix_scan 检查目录是否为空。 get_for_update 对目录项加行锁，防止检查后被其他线程插入新文件。

冲突重试机制

execute_with_retry

当两个事务同时锁定同一行时，后提交者收到 TransactionConflict。 系统自动重试（最多 3 次），对上层返回 EAGAIN。 参考 RocksDB TransactionDB 的悲观锁语义。

AtomicWriteBatch 跨 CF 原子性

一次 create 需要同时写入 inodes CF（新 inode）和 dir_entries CF（目录项）。 AtomicWriteBatch 保证跨 CF 写入的原子性——要么全部成功写入 WAL，要么全部回滚。

二、元数据性能评测

mdtest — HPC 业界标准元数据压测工具

IO500 (HPC 存储排行榜) 官方指定的元数据性能测试工具。TableFS、IndexFS、LocoFS 等论文均使用 mdtest 作为评测基准。 C 原生 syscall 直接操作挂载点，无 shell 开销。

  • mdtest-easy：每进程独立目录，测试分散创建
  • mdtest-hard：所有进程共享单目录，测试锁竞争
  • 支持 create / stat / read / remove 全流程

  • 对照组：相同硬件上的 ext4，同一 mdtest 参数
  • 规模：10K / 100K / 1M 文件梯度测试

已知不足与优化方向

并发安全问题

部分操作（如 deferred unlink 的 open handle 计数）使用 Mutex 保护， 在高并发下存在锁竞争。多线程同目录写入时 PCC 事务冲突率较高，需要优化锁粒度。

并发性能待优化

每次事务 commit 触发 WAL fsync，严重限制写入吞吐。 优化方向：WAL Group Commit（聚合多次 fsync 为一次）、 WriteBatch 批量聚合、LRU 缓存预热。

FUSE 固有开销

每次文件操作经历 用户态→内核→用户态 上下文切换（~3-5μs）。 这是 FUSE 架构的固有开销，可通过 FUSE 多线程 / splice 等方式部分缓解。

数据存储简陋

RawDiskDataStore 使用 inode × max_file_size + offset 稀疏寻址， 无空间回收、无 extent 管理。后续可引入 Chunk/Slice 模型优化。
