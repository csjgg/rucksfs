PART 1

研究背景与动机

为什么要用 KV 存储来管理文件元数据？
背景

元数据已成为文件系统的核心性能瓶颈

在生产环境中，元数据操作已占文件系统总 I/O 的一半以上。文件越小、数量越多，元数据在总开销中的占比越大，成为制约系统吞吐与扩展性的首要瓶颈。

生产环境数据佐证

指标 数值 来源
元数据占总 I/O >50% Meta 2020 生产集群 [OSDI'20]
元数据占小文件延迟 40% Yang et al. [MSST'19]
元数据节点 QPS 数亿 阿里 Pangu 2.0 [FAST'23]

结论：元数据操作占比超过一半，已成为限制系统吞吐与扩展性的主要矛盾——优化元数据管理刻不容缓。

海量小文件：瓶颈的放大器

大量小文件场景无处不在

AI 训练（千万级小图片）、容器镜像（数万碎片文件）、日志采集（高频 create）、IoT 数据流等场景，均会产生海量小文件，对元数据管理造成巨大压力。

共同特征

文件数量极大、单文件极小 → 每字节有效数据需要的元数据操作反而更多。元数据成为真正的性能主导因素。

数据来源: Meta OSDI'20 Trace · Yang et al. MSST'19 · Alibaba FAST'23

问题 & 方案

为什么用 KV 存储替代传统文件系统管理元数据？

关键洞察：元数据操作以写为主（create/mkdir/unlink 占比 >60%），而传统文件系统的写路径代价极高
读操作（lookup/stat）方面，ext4 的 HTree 和 RocksDB 的 LSM-tree 都是随机点查，性能差距不大。真正的性能鸿沟在写路径上——这恰好是元数据最频繁的操作。

问题 1 — 写放大：每次 create 修改 4 个分散磁盘区域，全部是随机写。HTree 分裂还可能额外增加写入。
问题 2 — 容量固定：inode 数量在 mkfs 时确定，用完即无法创建文件。

LSM-tree KV: create("new.txt") — 1 次顺序追加写

优势 1 — 顺序写：inode + 目录项写入 MemTable + WAL 顺序追加，不产生随机 I/O。
优势 2 — 动态扩容：无需预分配 inode，KV 存储动态增长，仅受磁盘容量限制。
优势 3 — 原子批量：WriteBatch 一次提交，inode + 目录项原子写入。

为什么元数据特别适合 KV 存储

  • 写密集 — create/mkdir/unlink 占比 >60%，LSM-tree 写优势最大化
  • 记录小 — inode 仅 ~100B，KV block 命中率高
  • 前缀扫描 — readdir = 按 parent_inode 前缀扫描，利用 SST 有序性

第一阶段：学术验证 (2013-2017)

TableFS 首次证明 LSM-tree 管理元数据可行——100 万文件创建比 ext4 快 5 倍。 IndexFS 扩展到 128 MDS 节点，吞吐提升 50-100x。 LocoFS 通过目录/文件元数据解耦，将 KV 利用率从 18% 提升到 93%。

第二阶段：工业落地 (2018-2020)

JuiceFS 用 Redis/TiKV 管理元数据，在生产环境广泛验证。 Hadoop Ozone 用三级 RocksDB 突破 HDFS 10 亿文件限制。 BFO 深入分析批量操作中的元数据开销，提出预取优化。

第三阶段：极致优化 (2023-2025)

SingularFS 证明精心优化的单机 MDS 性能可超过多节点分布式方案。 Mantle 提出 Delta Record + IndexNode 缓存，避免高频读-改-写，吞吐提升 115 倍。

核心设计

文件命名空间到 KV 存储的映射

以目录树 /home/alice/{notes.txt, photo.jpg} 为例，对比两种 KV 映射方案：

方案 A：扁平 KV（完整路径做 Key）

S3、COS 等对象存储的常见做法

"/home" → { mode: dir, ... }
"/home/alice" → { mode: dir, ... }
"/home/alice/notes.txt" → { size: 1024, ... }
"/home/alice/photo.jpg" → { size: 2MB, ... }
"/home/alice_backup/a.txt" → { ... }

方案 B：层级 KV（parent_inode + name 做 Key）

本系统采用，两个 Column Family 分别存储

CF: inodes

[2] → { dir, nlink:3 }
[3] → { dir, nlink:4 }
[10] → { file, 1024B }
[11] → { file, 2MB }

CF: dir_entries

[1|"home"] → child:2
[2|"alice"] → child:3
[3|"notes.txt"] → child:10
[3|"photo.jpg"] → child:11
操作 扁平 KV（完整路径） 层级 KV（本系统）
rename 目录
mv /home/alice → /home/bob		 O(N) — 必须改写所有子孙 Key
路径前缀变了，每条记录都要删旧写新		 O(1) — 只改一条 dir_entry
[1|"alice"]→3 改为 [1|"bob"]→3，子孙不动
readdir
ls /home/alice/		 前缀扫描 "/home/alice/"
会误扫到 "/home/alice_backup/..." 需过滤		 前缀扫描 [3|*]（8字节数字前缀）
精确匹配，无误扫；RocksDB 有序 SST 顺序读取
lookup
stat /home/alice/notes.txt		 单次 get "/home/alice/notes.txt"
Key 平均 30+ 字节，比较开销大		 逐级 get：[1|"home"]→[2|"alice"]→[3|"notes.txt"]
3 次 get，但每次 Key 短，且可缓存中间结果
Key 大小 路径越深越长 — 平均 30~100 B
深层嵌套场景下 Key 占比超过 Value		 固定 8 B + name — 平均 ~20 B
Key 紧凑，RocksDB Bloom Filter / 压缩更高效

选择层级 KV 的核心原因：rename 是 POSIX 语义的基础操作，扁平 KV 下 rename 目录代价为 O(N) 是不可接受的。 层级 KV 以 lookup 多一两次 get 为代价（可通过 VFS 缓存消除），换取 rename O(1)、readdir 精确扫描、Key 存储紧凑三大优势。

核心设计

Delta 增量更新机制

问题场景：父目录 inode 热点写入

每次 create / mkdir / unlink / rmdir / rename 都必须更新父目录 inode 的三个属性：

nlink（子项计数） mtime（修改时间） ctime（变更时间）

传统方式需要在事务中 get_for_update(parent) 对父 inode 加行锁 → 读 57B → 反序列化 → 修改 → 序列化 → 写回。 同一目录下所有并发操作被串行化在这把锁上——这是热目录的首要瓶颈。

解法：Delta 追加写（来自 Mantle SOSP'25）

不修改父 inode 本体，而是向独立的 delta_entries 列族追加一条增量记录（5~9 字节），描述"需要对父 inode 做什么变化"：

// mkdir("/data/hot_dir/subA") → 追加 delta 到父目录 inode
delta_entries CF: put( key=[X|parent_ino|seq], val=IncrNlink(+1) ) // 5B
delta_entries CF: put( key=[X|parent_ino|seq], val=SetMtime(now) ) // 9B
delta_entries CF: put( key=[X|parent_ino|seq], val=SetCtime(now) ) // 9B

追加操作无需读取父 inode、无需事务行锁——N 个并发线程可以同时写入，吞吐从 1/N 变为 N。

❌ 传统：读-改-写（串行瓶颈）

Thread A: lock(parent) → read 57B → nlink++ → write 57B → unlock
Thread B: blocked, waiting for lock...
Thread C: blocked, waiting for lock...
N 并发串行化 → 吞吐 = 1/N

✅ Delta：追加写（无锁并行）

Thread A: append Δ(nlink+1, mtime=T1) → 5~9B
Thread B: append Δ(nlink+1, mtime=T2) → 5~9B
Thread C: append Δ(nlink+1, mtime=T3) → 5~9B
无需读父 inode、无需加锁 → 吞吐 = N

读取路径：Fold-on-Read

① 查 LRU 缓存 → 命中 O(1) 返回
② miss → read base (inodes CF) + scan deltas
③ fold: base + Δ₁ + ... + Δₙ → 最终属性值
④ 写入缓存，后续直接命中

Fold 语义：可交换 & 可结合

IncrementNlink(n) — 整数加法，可结合
SetMtime(t) / SetCtime(t) — 取 max，可交换

并发 delta 无论到达顺序如何，fold 结果都正确

后台 Compaction

每 5s 扫描，delta 累积 > 32 条时触发：fold 进 base → 原子写回 inodes CF → 清除旧 delta → 缓存失效。读放大始终有界。

核心价值：将父 inode 更新从事务关键路径中移除。传统方案所有同目录操作串行等待父 inode 行锁；Delta 方案在事务外追加 5~9B 记录， 父 inode 完全不参与锁争抢。Mantle 论文在百度 EB 级存储验证：同目录高并发 create 吞吐提升两个数量级。

并发控制

PCC 事务 + Delta —— 高效的并发元数据管理

RocksDB TransactionDB (Pessimistic) 提供 ACID 事务语义；我们在此基础上用 Delta 机制进一步缩小事务临界区。

1 原子性 (Atomicity)
所有写操作先缓存在内存 WriteBatch 中，commit 时一次性写入 WAL + MemTable——要么全部成功，要么全部回滚。 跨 Column Family 同样保证原子性。
2 隔离级别 & 行级锁
PCC 模式下每次写 / GetForUpdate 都先获取 行级排他锁。 默认 Read Committed；通过 SetSnapshot 可提升到 Repeatable Read。
3 冲突检测 & 重试
加锁失败立即返回 Busy，无需等到 commit 再做冲突检测。 我们封装 execute_with_retry，冲突时自动重试最多 3 次。

核心 POSIX 操作示例

事务只锁 dir_entry，不锁父 inode——同目录不同文件名的操作可完全并行；父目录属性更新通过 Delta 事务外追加。
create("new.txt")
1. begin_txn()
2. get_for_update(dir_entry)
→ 已存在则 EEXIST
3. put(inode) + put(dir_entry)
4. commit()
5. append_delta(parent)
rename — 跨目录
1. begin_txn()
2. get_for_update(sorted keys)
→ 按 ID 排序加锁，防死锁
3. delete(src) + put(dst)
4. commit()
5. delta(old_parent + new_parent)
rmdir
1. begin_txn()
2. get_for_update(entry + inode)
3. prefix_scan → 非空 ENOTEMPTY
→ TOCTOU 安全
4. delete(entry) + delete(inode)
5. commit()
6. delta(parent, nlink-1)

设计关键：事务只保护结构变更（inode + dir_entry 的原子写入）， 父目录属性更新（nlink/mtime）通过 Delta 在事务外追加完成。即使 Delta 丢失，仅影响 mtime 准确性，不影响文件系统结构完整性。

一、POSIX 正确性验证

pjdfstest — POSIX 合规测试套件

FreeBSD/Linux 文件系统社区标准工具，覆盖 chmod、chown、link、mkdir、open、rename、rmdir、 symlink、truncate、unlink 等操作的边界情况。 将 RucksFS 挂载为 FUSE 后，在挂载点上运行完整套件。

Rust 单元 / 集成测试

基于真实 RocksDB 后端（非 mock），覆盖正常和异常路径。 补充 pjdfstest 未覆盖的场景：write+read 一致性、deferred unlink、fsck 一致性检查。 当前已有 228 个自动化测试。

二、元数据性能评测

自研 Rust 单机压测工具（主要方案）

使用 Rust 编写专用的元数据压测工具，直接调用 POSIX syscall 操作挂载点， 精确控制并发模型和测试参数，无外部依赖。

  • 支持 create / stat / read / remove 全流程
  • 单目录高竞争 + 多目录分散两种模式
  • 可定制并发线程数、文件规模、目录深度
评测对比设计
  • 对照组：相同硬件上的 ext4，相同测试参数
  • 规模：10K / 100K / 1M 文件梯度测试
  • 并发：1 / 2 / 4 / 8 线程扩展性测试
  • 指标：ops/sec（与 TableFS 论文数据可比）
  • 后续考虑引入 mdtest（IO500 标准工具）作为补充评测参考
